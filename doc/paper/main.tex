\begin{abstract}
    知识图谱作为一种语义网络，能够以图结构形式描述实体及其之间的关系，在知识管理、智能问答、推荐系统等领域有着广泛的应用。然而，传统知识图谱查询需要用户掌握Cypher等查询语言，对非技术用户存在一定门槛。

    针对上述问题，本文设计并实现了一个基于自然语言的知识图谱查询与可视化系统（neo4jSlave），采用常见的浏览器/服务器结构：用户只需在网页中用中文提问，无需在本地安装图数据库客户端。服务端使用 FastAPI 提供应用接口，在收到问句后先从 Neo4j 中读取当前图数据的结构信息（如实体类型、关系类型），与问句一起交给大语言模型，引导其生成与当前库结构相符、且带参数的只读图查询语句；随后由专门模块对查询语句做规则校验（如禁止写入类操作、绑定参数），再调用 Neo4j 执行。查询结果在浏览器中除表格外，还通过 ECharts 以点边关系图等形式呈现，使“谁与谁如何相连”更直观。图数据库地址、大模型服务地址等可通过环境变量统一配置，便于换环境与复现。整体上，系统把“问一句话”到“看到图上的关系”串成连续流程，弱化了形式化查询语言对终端用户的显式要求。

    在验证方面，对关键模块与对外接口进行了自动化测试，用例均通过，保证了实现与需求约定的一致性。另以自建自然语言问句做人工走查，绝大多数常见问句能端到端得到可执行查询与可视化结果，在表述复杂或易歧义的情形下仍可能出现答非所问或取数不完整，有进一步改进空间。本工作为在垂直领域图数据上搭建“低门槛、可复现”的自然语言查询与可视化模式提供了一套完整、可沿用的工程化路径。
\end{abstract}

\addcontentsline{toc}{section}{Abstract}
\begin{thesisabstracten}
Knowledge graphs, as semantic networks, can describe entities and their relationships in graph structure, and have wide applications in knowledge management, intelligent question answering, and recommendation systems. However, traditional knowledge graph queries require users to master query languages such as Cypher, which presents certain barriers for non-technical users.

To address this gap, this paper designs and implements a natural language--based knowledge graph query and visualization system (neo4jSlave) using a browser/server architecture. Users ask questions in Chinese on a web page, without installing a graph client locally. A FastAPI backend fetches the current graph schema (entity and relationship types) from Neo4j, combines it with the question, and prompts a large language model to produce a parameterized, read-only query aligned with the schema. A dedicated layer validates the query (e.g., enforcing read-only and parameter binding) before execution against Neo4j. Results are shown as tables and as node--link diagrams via ECharts, making relationships easy to read. Service endpoints, the graph database, and the LLM API are configurable through environment variables to support deployment and replication. The design connects ``natural language in'' to ``graphical insight out'' while keeping formal query language usage implicit for end users.

On the evaluation side, automated tests cover the main modules and public interfaces and all pass, supporting consistency with the specified behavior. A manual walk-through with custom natural-language questions shows that most routine questions yield executable queries and visualizations end-to-end; on highly complex or ambiguous phrasings, incorrect or partial answers can still occur. The work offers a full, replicable engineering path for low-threshold, natural-language query and visualization over domain-specific graph data.
\end{thesisabstracten}

\section{系统概述}

\subsection{研究背景}

本课题要解决的问题可概括为三点：让不掌握图查询语言的人也能用自然语言在浏览器里查询某个领域知识库中的关联关系，并把结果用关系图等形式直观展示出来，同时在对外提供服务时把误写、误改数据等风险压到可接受水平。

知识图谱以有向图的方式表示实体为顶点、关系为边，可附带顶点和边上的属性，用来回答“与某实体相连的另一类实体是什么、沿怎样的路径连起来”等问题；这类表达在信息检索、推荐、金融风控、生物信息等领域已被广泛接受。工程上，领域数据常被整理成图数据库中的可查询图；其中较常见的一种抽象称为属性图（Property Graph），即把实体与关系都看成可带键值对属性的“对象”，用声明式图查询语言描述模式匹配与路径。与之并列的另一类路线是资源描述框架（RDF, Resource Description Framework）及其配套查询语言 SPARQL：RDF 把知识写成大量“主语—谓语—宾语”的三元组，擅长开放数据互联与规范化的语义网场景；属性图则更贴近面向工程实现的存储与产品迭代。二者并无绝对优劣，取舍取决于数据形态、团队熟悉度与是否已有现成图实例。

在此基础上，Neo4j 是常用的开源属性图数据库之一，其官方查询语言为 Cypher。Cypher 属于形式化查询语言：使用者需要了解标签、关系类型、（可选的）多跳路径等概念。因此，对许多希望“用日常语言提一句、就看到关系图”的终端用户而言，直接手工编写 Cypher 的学习成本仍然偏高。近年在大语言模型（LLM, Large Language Model）推动下，以自然语言生成可执行 SQL、Cypher 等结构化查询已成为可落地的技术路线。本文正是要在上述公认背景下，为某一类以属性图组织的中文领域数据，实现一套可说明、可复现的浏览器/服务器模式（B/S, Browser/Server）系统：读者只需掌握本论文与所附环境说明，便能在本地用同类数据与接口重复本文流程。

为回答“与既有研究是什么关系、现有工具为何仍不足以自然覆盖本课题需求”，下面对知识图谱的建模与查询、自然语言到图查询的既有范式，以及大语言模型与知识图谱协同的常见做法做归纳；本文拟采取的具体技术栈与复现方式则在第2章与后续章节中展开\cite{ref6,ref7,ref8,ref12,ref13,ref14,ref15}。

\subsection{国内外研究现状}

\subsubsection{知识图谱的数据模型、存储与查询基础}

从数据如何被写成“图”看，研究界与产业界已形成了若干可对照的范式。一类是以 RDF 与 SPARQL 为代表的三元组建模：把事实拆成大量三元组，便于在开放万维网上发布、链接和复用，并与本体、推理规则等机制配合；其代价往往是查询与维护需要熟悉较完整的语义网技术栈\cite{ref17}。另一类是属性图：在实体为节点、关系为边的图之上直接挂载属性，查询语言通常围绕模式匹配与路径展开。王鑫等从数据管理角度总结了知识图谱在存储、查询与知识融合等任务上的共性问题，例如图规模大时的索引、查询优化、事务与一致性\cite{ref1}。上述综述所讨论的并非“某一家国内团队的一家之言”，而是当前数据库与信息检索领域对“图这种抽象如何落地”的较普遍共识。Robinson 等以图思维对比关系表建模，说明了为何复杂关联在图上更直观\cite{ref3}；Miller 等从应用侧回顾了以 Neo4j 等为代表的图数据库在分析场景中的典型用法\cite{ref2}。

就与本文选型的关系而言：本课题的实例数据以可导入、可复现为导向，已具备属性图形态，且需要在应用层用一种主流程序语言与 HTTP 服务对接；Cypher 与 Neo4j 驱动的文档与社区资源成熟，Cypher 语义以官方手册为准\cite{ref12}。RDF/SPARQL 体系在需要强语义互联或已有 RDF 存量时更具吸引力，而在已有属性图实例、且团队目标是以最少组件完成“可跑的 Web 查询系统”的前提下，复用属性图 + Neo4j 是务实路线。但仅有图存储仍不足以满足第1.1 节中的易用目标，因为图查询仍依赖形式化语言；这正是下一小节自然语言接口要承接的问题。

\subsubsection{自然语言到结构化查询：从 Text-to-SQL 到 Text-to-Cypher}

上一小节说明了“把领域数据存成可查询的图”的主流做法；但终端用户更常见的需求是用自然语言（NL, Natural Language）提问，而不愿学习形式化语言。为把讨论落到可对照的科研脉络上，先看关系型数据库中研究时间更长、文献更集中的文本到 SQL（Text-to-SQL）问题：其目标是把问句转写为可执行的 SQL（Structured Query Language）。Qin 等的综述将主流技术路线归纳到编码器—解码器、中间表示与预训练语言模型等框架下，并指出模式（Schema）感知（让生成语句能“看得懂表结构、列名”）与可执行性约束（让生成结果真实可跑）是决定效果的关键因素\cite{ref5}。对读者而言，可以把 Text-to-SQL 理解为“自然语言到结构化取数”的一条成熟参照。

再看图数据库场景。当底层存储是属性图、查询语言是 Cypher 时，与 Text-to-SQL 可类比的任务常被称作将自然语言映射为 Cypher 的 Text2Cypher：Ozsoy 等指出，即便借助大语言模型做零样本生成，在复杂语义与多跳约束上仍易产出不完整或错误语句；在公开高质量语料相对有限的前提下，用清洗、整合多源数据做微调，往往能在 双语互译评估（BLEU, Bilingual Evaluation Understudy）等指标上获得可观收益\cite{ref16}。这提示在工程上不能仅靠“换大模型”本身，还需要与当前库结构对齐的提示与约束，才能把生成结果稳住。

在问答（QA, Question Answering）等相近任务中，从自然语言到 SPARQL 或到 SQL/其它逻辑形式，也长期存在解析或检索—排序等流水线\cite{ref5}。它们与图库自然语言查询的共同点在于：都必须在自然语言与符号查询之间的词汇鸿沟、组合式泛化、以及执行安全（例如只读、不做危险写操作）之间做权衡。本文的实现路径不是训练一个专用 Text2Cypher 神经网络（受周期与算力条件约束，也与当前常见的“通用大语言模型＋提示学习、即基于提示词的上下文控制”的落地方式更一致\cite{ref4,ref9}），而是在大语言模型生成后增加应用层只读校验等可解释环节，以控制误执行风险。

\subsubsection{大语言模型、提示学习与知识图谱的融合}

承接上一小节，大语言模型除了用于“写查询”，还被广泛用于与知识图谱在知识增强、对齐、推理与可解释性等方向协同。Pan 等的路线图式综述指出，二者既可互相补充（图可为模型提供事实依托，模型可为图提供自然语言入口），也带来幻觉与结果不一致等风险，通常需借助检索、约束或外部验证抑制\cite{ref11}。在图数据库自然语言查询这一子场景中，可概括为：大语言模型负责从问句到符号查询的软映射；Cypher与 Neo4j 作为硬执行与可审计的载体；二者之间的中间层则常见为只读、参数绑定、超时、结果量限制等策略，否则难以从实验原型推进为可部署服务\cite{ref4,ref9,ref11}。现有 Text2Cypher 的多数论文仍以数据集、评测指标为中心\cite{ref16}，对“与 HTTP 接口、Web 前端、关系图展示联动的、面向某一领域的多标签 Schema 的端到端系统”讨论相对较少；这正是本文要在工程与可复现层面补足的部分。

综上，已有研究在图建模与图查询、自然语言到 SQL/Cypher、大语言模型与知识图谱协同等方向形成了较完整的知识积累，但这些成果并不自动等同于一个能交给他人按文档复现的“中文自然语言＋本领域属性图＋浏览器里看图谱”的成品系统。例如，单独一篇 Text2Cypher 论文或单独一份 Neo4j 手册，都不能直接回答本课题在引言中提出的三点目标。为此，本文不主张训练新模型或新查询语言，而是针对此类已整理为属性图、以中文描述为主的垂直领域数据，设计并实现一套可说明依赖与运行方式的 B/S 系统：在后台用通用人语言模型与提示语将问句转写为参数化 Cypher，在服务器侧做只读等安全校验，再把结果以 Apache ECharts 等组件支持的关系图与表格联合呈现，使自然语言输入、受约束执行、可视化输出成为一体。后续第3 章起将依次说明需求、架构与复现性细节。

\subsection{问题描述与挑战}

在明确了既有研究能回答什么、仍不能代劳什么之后，可以更清晰地体现出本课题要克服的矛盾。若只提供“在数据库客户端里手写 Cypher”的用法，不掌握 Cypher 的用户将难以完成查询；若完全依赖现场技术人员代写，则沟通成本与试错的节奏又不利于自学与复现。另一方面，图查询的中间结果若只以二维表显示，多步相连的关系不总是直观；若能在浏览器中把点与边画出来，更符合多数读者对“关系图”的心理预期。上述两点分别对应易用与可感知，也是第1.1 节中目标的两项。

在引入大语言模型之后，还会叠加新的工程问题：模型可能生成非只读语句、遗漏与 Schema 绑定的参数，或在多标签的库结构上选到不理想的路径。因此，本系统不能只把“大语言模型接到字符串出口”，而必须在前后端中明确划分只读校验、参数与占位符、超时与结果量限制，并协调查询结果的形态，使其同时便于表格展示与关系图绘制。以上矛盾将在后文落实在自然语言模块、数据访问与执行模块、结果转换与可视化模块、以及安全与约束模块中分别展开。

\subsection{课题意义}

从实践意义看，本系统可当作某一领域知识库上的“轻量查询入口”：数据维护、内容编辑、教学或演示等场景中，用户通过浏览器即可访问，无需在本地安装数据库图形客户端，也无需记忆固定 Cypher 模板。从教学与复现意义看，论文后续章节将按“需求—设计—实现—测试”展开，并给出环境配置、应用程序接口（API, Application Programming Interface）约定与关键算法说明，使具备 Python 与 Web 基础的读者能够在本地复现同类流程（将图数据与 LLM 端点替换为自有资源即可平移）。

从技术意义看，本文将“自然语言到 Cypher 的受约束生成＋只读沙箱＋图可视化”串成闭环，讨论了多标签图 Schema（在本文中指当前库中实体类型、关系类型及字段的集合，见第2、4 章）对提示词与取数效果的影响，对与本文类似的垂直领域属性图在低门槛人机交互方面具有一定可移植参考价值。

\subsection{本文主要工作}

全文围绕四个彼此衔接的核心模块展开（测试与评估贯穿第6章，作为对前四项工作的验证）。各模块的设立原因、承担职责与主要优势如下。

（1）自然语言查询模块（NL→Cypher）

原因：降低 Cypher 门槛是本课题的直接目标；领域问句多为中文短句，适合由 LLM 结合 Schema 描述与示例（Few-shot）生成查询。

作用：接收用户自然语言，调用大语言模型 API，输出参数化的 Cypher 与参数字典，供后端执行。

优势：用户无需记忆语法；可通过调整系统提示词与示例快速适配 Schema 变更。局限在于生成质量依赖模型与 Prompt，故必须与校验模块配合（后文详述）。

（2）图数据访问与 Cypher 执行模块

原因：所有查询最终必须在 Neo4j 上执行；需统一会话、只读访问、超时与结果键序，供转换模块消费。

作用：封装 Neo4j 驱动；为自然语言链路与“高级用户直接写 Cypher”模式提供同一执行入口（如 \texttt{/run-cql}）；暴露 Schema 查询（如 \texttt{/schema}）供 LLM 或前端使用。

优势：逻辑集中、便于替换连接参数以指向另一套图实例；与业务层解耦，利于测试与维护。

（3）结果转换与可视化模块

原因：Neo4j 返回的记录中可能包含节点、关系、路径等类型，而 ECharts 关系图需要统一的节点/边 JSON；若仅展示表格，无法体现图结构优势。

作用：将查询结果转换为 ECharts Graph 所需的 \texttt{nodes}/\texttt{links}（及类别、表格列行等）；前端负责渲染与交互（拖拽、缩放等）。

优势：查询层可专注 Cypher；展示层可独立迭代样式与交互，符合前后端分离思路。

（4）安全与约束模块（只读与参数校验）

原因：对外暴露“执行任意 Cypher”能力时，若完全信任 LLM 输出，存在误删、误改或调用危险过程的风险；即便本系统面向只读库，也应在应用层显式拒绝写操作关键字并校验参数完整性。

作用：黑名单正则、可选 EXPLAIN 预检、\texttt{\$param} 占位符与请求体参数对齐、查询超时与硬上限等。

优势：在“智能”与“可控”之间折中：保留 LLM 灵活性，同时把不可接受的操作挡在数据库之外，便于向评审说明安全设计意图。

除上述四部分外，本文还将在第6章对功能与自动化测试进行汇总，用数据与用例说明系统行为是否符合预期。

\subsection{论文组织结构与复现线索}

第3章从可行性、需求与性能角度分析系统为何如此划定功能边界；第4章给出总体架构、模块关系与数据库设计（含多标签演进说明）；第5章按四个核心模块分别给出详细设计与实现（接口、算法与关键代码）；第6章说明测试方法与结果。读者若希望复现，可优先阅读第2章环境与选型、第5章各模块实现与配置说明，并结合仓库中的 \texttt{README.md}、\texttt{.env.example} 与测试脚本逐步搭建环境。

\section{系统开发环境与技术选型}

\subsection{开发环境}

本系统在 GNU/Linux 环境下开发与调试（例如 Ubuntu 22.04 LTS、Debian 12）。宿主机需安装可运行的 Neo4j 实例，既可使用本机服务（如 \texttt{bolt://127.0.0.1:7687}），也可连接局域网或云主机上的远程实例（如 \texttt{bolt://10.x.x.x:7687}）。浏览器建议使用近两年的主流版本（如 Chrome 120+、Edge 120+、Firefox 120+），以保证前端图形交互稳定。后端使用 Python 3.10 及以上版本，并通过虚拟环境管理第三方依赖（如使用 \texttt{python -m venv .venv} 创建环境，配合 \texttt{pip install -r requirements.txt} 安装 \texttt{fastapi}、\texttt{neo4j}、\texttt{httpx}、\texttt{pytest} 等包），以避免与系统 Python 冲突。前端不依赖 Node 构建链，采用原生 HTML/CSS/JavaScript，便于课程与答辩现场快速演示。版本控制使用 Git；日常编辑可使用任意编辑器（如 Vim、VS Code 等）。上述选择的目标是：环境可描述、依赖可列举、在常见 Linux 发行版上可重复搭建，从而降低他人复现时的不确定因素。

\subsection{技术路线与第1章问题的对应关系}

第1章在问题描述中把矛盾归纳为三方面：形式化查询门槛、查询结果如何更易感知、在开放网络环境下执行生成语句需安全约束。技术选型据此形成一条可对照的链路：

\begin{itemize}
    \item 大语言模型 + HTTP 客户端：把自然语言到 Cypher 的映射交给大语言模型完成；通过 超文本传输安全协议（HTTPS, Hypertext Transfer Protocol Secure）等通道调用与 OpenAI 习惯相近的 应用程序接口，便于在不大改业务骨架的前提下更换云厂商或模型端点。
    \item FastAPI：统一以表述性状态转移（REST, Representational State Transfer）风格的 Web API 对外暴露\cite{ref6,ref10}，同时承载“自然语言入口”与“直接提交 Cypher”两条路径，并托管静态前端，部署上保持单一可执行服务。
    \item Neo4j 官方 Python 驱动：保证与数据库协议一致，支持只读会话与查询超时。
    \item ECharts：将图查询结果映射为前端可渲染的数据结构，降低自研渲染引擎的成本。
\end{itemize}

\subsection{技术栈}

在给出按组件独立比较的总览后，本节按“后端语言与框架—数据库—模型服务—可视化—工具链”的顺序说明各技术项的职责与取舍理由，避免读者只见名词而不见关系。

\subsubsection{Python 3.10+}

Python 语法清晰，在数据与 Web 方向库齐全。本系统后端完全采用 Python，便于与 Neo4j 驱动、HTTP 客户端、pytest 测试框架在同一生态内集成。选择 3.10+ 是为了使用较新的类型标注与标准库特性，并与主流 Linux 发行版默认或软件源中的 Python 版本相匹配。

\subsubsection{FastAPI}

FastAPI 基于 Starlette 与 Pydantic，支持异步路由、自动 OpenAPI 文档与请求体验证。相较 Flask，其异步与类型提示更适合同时处理 LLM 网络请求与数据库 IO 的混合场景；相较 Django，其体量更轻，契合本课题“单一服务 + 静态前端”的部署形态。在本系统中的职责是：注册 \texttt{/nlq}、\texttt{/run-cql}、\texttt{/schema} 等路由，挂载静态资源，并在中间件层配置 CORS（演示环境宽松、生产环境应收紧，见第3章非功能需求）。

\subsubsection{Neo4j 图数据库}

Neo4j 使用 Cypher 与属性图模型，社区资料与示例丰富，且支持多标签节点——这与本文领域数据中“道具 / 方块 / 生物”的继承式建模一致（详见第4章）；其应用与概念讨论可见早期文献\cite{ref2}，产品与协议细节以官方资料为准\cite{ref7}。未首选 JanusGraph、ArangoDB 等的原因主要是：本课题已有 Neo4j 导入管线与实测数据规模，且 Cypher 与 LLM 训练语料中的“图查询”示例重合度较高，有利于 Prompt 设计。

\subsubsection{大语言模型（LLM）}

系统通过 OpenAI 兼容的 HTTPS API 调用大模型（如 GPT 系列、Kimi、豆包等），由 \texttt{LLM\_API\_BASE}、\texttt{LLM\_MODEL} 等环境变量配置；大模型提示学习与指令遵循能力的研究脉络见文献\cite{ref4}，商用 GPT-4 系列的技术报告见文献\cite{ref9}。作用是将用户问句与 Schema 摘要拼装为提示词，解析模型返回的 JSON 得到 Cypher 与参数。优势是无需自训专用模型即可迭代；代价是需网络与 API 费用，且生成结果必须经后文安全模块过滤。

\subsubsection{ECharts}

ECharts 为 Apache 开源项目，关系图（Graph）组件成熟，支持力导向布局与交互。在本系统中负责将后端返回的 \texttt{nodes}/\texttt{links} 绘制为可拖拽、可缩放的图谱；与 D3.js 相比上手更快，与 Cytoscape 相比更贴近 Web 单页应用嵌入需求。

\subsubsection{其他工具}

\begin{itemize}
    \item Git：版本管理与协作。
    \item pytest：后端单元测试与 API 契约测试（LLM 可 Mock）。
    \item httpx：异步/同步 HTTP 客户端，用于调用 LLM API。
\end{itemize}

\subsection{系统架构与数据流}

本系统采用前后端分离的 B/S 架构，如图~\ref{fig:architecture}所示：

\begin{figure}[H]
    \centering
    \includegraphics[width=0.95\textwidth]{img/architecture.png}
    \caption{系统总体架构图}
    \label{fig:architecture}
\end{figure}

各层职责及相互关系可概括为一条主数据流：

\begin{enumerate}[(1)]
    \item 浏览器（前端层）：用户输入自然语言或 Cypher；将请求发往 FastAPI；接收 JSON 后由 ECharts 绘圈、由表格区展示属性。
    \item FastAPI（后端层）：对 \texttt{/nlq} 请求先拉取 Schema、再调 LLM 生成 Cypher；对两类执行请求均先做只读校验与参数检查，再调用 Neo4j。
    \item Neo4j（数据层）：在只读会话中执行 Cypher，返回记录流。
    \item 转换逻辑（后端层内）：将记录转为图与表结构，随响应返回前端。
\end{enumerate}

由此可见，自然语言模块、执行模块、安全模块、可视化模块并非孤立罗列：前两者串成“问句→语句→结果”，安全模块横切在执行之前，转换模块衔接数据库与前端展示。第4章将从设计角度进一步展开模块接口与数据库模型。

\section{系统分析}

为避免第2章“技术能选”与第4章“如何设计”之间出现断层，本章先回答“为什么这样划分模块与边界”，再把结论传递给后续设计与实现章节。

\subsection{可行性分析}

\subsubsection{技术可行性分析}

判断依据：本系统依赖的四类技术（Web 框架、图数据库、HTTP LLM、浏览器可视化）均为工业界常见组合，不存在“实验室独有、无法获得”的组件。

为何认为可行：FastAPI 与 Neo4j 官方 Python 驱动文档齐全，可在单机完成“启动服务—连库—返回 JSON”闭环；LLM 侧采用 HTTPS 兼容接口，与具体厂商解耦；ECharts 以 JSON 驱动画布，与后端数据结构天然匹配。

如何在本项目中达成：仓库已提供最小可运行后端与静态前端、\texttt{pytest} 用例及 \texttt{.env.example}，证明四类组件可组装为稳定流水线；自然语言准确率受 Prompt 与 Schema 影响，属效果调优范畴，不改变“能跑通”的技术可行性结论。

\subsubsection{经济可行性分析}

判断依据：开发期成本主要来自人力与机器；运行期成本来自图库存储与 LLM 调用次数。

为何认为可行：Neo4j 社区版与 Python 生态可免费使用；LLM 可按调用量计费，且本系统可通过缓存 Schema、限制单次返回行数等方式控制费用。

如何在本项目中达成：配置项 \texttt{QUERY\_HARD\_LIMIT}、\texttt{QUERY\_TIMEOUT\_MS} 等在实现层限制单次查询规模，降低恶意或失误大查询带来的算力与费用尖峰（详见第5章）。

\subsubsection{操作可行性分析}

判断依据：最终用户是否愿意使用，取决于学习成本与访问路径。

为何认为可行：B/S 架构下用户仅需浏览器；自然语言模式降低 Cypher 学习成本；高级用户仍可切换直接 Cypher，兼顾两类人群。

如何在本项目中达成：单页内提供模式切换与生成语句回显，减少“黑盒”感；Schema 接口便于用户理解当前图里有哪些标签与关系，与第4章“可视化 + Schema”设计一致。

\subsection{需求分析}

\subsubsection{参与者与系统边界}

从用例视角，与本系统直接交互的主要参与者包括：（1）终端用户——通过浏览器使用自然语言查询、查看图谱与表格，通常不具备 Cypher 基础；（2）高级用户/调试者——需要直接提交只读 Cypher、对照 Schema 调试生成结果或做教学演示。二者均通过同一 Web 前端访问后端 API。

外部参与者包括：大语言模型服务（HTTPS 兼容 API，负责 NL→结构化输出）、Neo4j 图数据库（存储领域知识图谱并执行 Cypher）。本论文实现的 neo4jSlave 服务构成系统边界：边界内完成编排、校验、转换与静态页面托管；边界外依赖的 LLM 与图库由部署环境配置（见第2章与第5章）。

\subsubsection{功能需求}

（1）自然语言查询。

原因：降低 Cypher 门槛是课题核心目标。

行为：用户输入中文问句，经 LLM 生成参数化 Cypher 后执行。

关联：依赖 Schema 查询（5）提供标签/关系摘要；依赖安全限制（6）过滤写操作；执行结果交由（3）（4）展示。

（2）直接Cypher执行。

原因：调试、对比与教学演示需要“可重复、可精确控制”的语句通道，不能完全依赖模型生成。

行为：用户在“Cypher”模式下提交语句（API 路径 \texttt{/run-cql}）。

关联：与（1）共用同一套只读校验、参数校验与 Neo4j 会话，避免两套安全逻辑。

（3）图谱可视化。

原因：图数据的价值在于拓扑；表格式难以传达“谁连谁”。

行为：将查询结果映射为 ECharts Graph 的节点与边。

关联：依赖自然语言或 Cypher 模块返回含路径/关系的记录；与（4）互补——图看结构，表看属性。

（4）数据表格展示。

原因：数值属性、长文本描述在图中不便阅读。

行为：按列键展平记录为表格。

关联：与（3）同源 JSON，同一响应体中并行返回，前端分区渲染。

（5）Schema查询。

原因：用户与 LLM 均需知道“图里有什么标签/关系”，否则问句与生成语句易偏离领域。

行为：提供 \texttt{/schema} 等接口。

关联：NL 链路在生成前拉取 Schema；前端也可展示供高级用户参考。

（6）安全限制。

原因：开放执行通道必须防止误删与危险调用。

行为：黑名单、参数校验、超时等。

关联：横切（1）（2），在访问数据库前统一生效。

\subsubsection{非功能需求}

（1）安全性。即便数据库账号为只读，应用层仍应拒绝明显写语句，防止配置失误或 LLM 异常输出。

如何达成：正则黑名单 + 可选 EXPLAIN + 参数白名单式校验（见第4章安全模块与第5章实现）。

（2）性能。交互式查询对尾延迟敏感；LLM 已引入秒级延迟，图查询与渲染不应再放大瓶颈。

如何达成：Neo4j 会话超时、结果行硬上限、前端力导布局在合理节点规模内运行。

（3）可扩展性。课题结束后可能更换模型、迁移图实例或替换可视化方案。

如何达成：环境变量配置 URI 与 API；转换器与路由分层，避免把业务写死在单一文件中。

（4）易用性。非专业用户是主要受众。

如何达成：单页、少步骤、生成语句可回看；错误信息返回可读提示（如缺参列表）。

（5）部署与网络策略。演示环境与生产环境风险不同。

如何达成：开发期宽松 CORS；上线前收紧为白名单并置于反向代理之后（与第1章非功能叙述一致）。

\subsection{性能分析}

（1）LLM API 响应时间。自然语言转 Cypher 通常占端到端延迟的主要部分（常见为 1–3 秒量级，视模型与网络而定）。

应对：不重复调用模型（单次请求携带 Schema）；失败时快速返回错误，避免阻塞线程池。

（2）Neo4j 查询性能。与模式复杂度、索引及数据量相关；本课题数据规模为万级节点、万级有向边，常见路径查询在毫秒至数百毫秒量级。

应对：限制 \texttt{LIMIT}、避免无界全图扫描类语句进入生产（由 Prompt 引导与硬上限双保险）。

（3）前端渲染性能。ECharts 在万级节点仍较流畅，但力导布局成本随边数上升。

应对：与（2）同一上限逻辑耦合——控制返回节点/边数量即控制渲染成本。

综上，性能目标通过“限制规模 + 超时 + 合理 Prompt”组合达成，而非依赖未经验证的无限扩容假设。

\subsection{本章小结}

本章从可行性、需求与性能三方面论证了系统的建设边界：技术上四类组件可组装为稳定流水线；经济上以开源栈与可配置的 LLM 调用为主；操作上以浏览器单页降低使用门槛。需求分析在参与者与用例图（图~\ref{fig:use-case-placeholder}）框架下，将六类功能与五项非功能写成“动因—行为—关联”，并辅以用户故事（表~\ref{tab:user-stories}）明确可验收行为，避免条目孤立。性能分析则把端到端延迟拆解为 LLM、Neo4j 与前端三环节，并给出与第5章实现参数相对应的应对思路。以上结论为第4章的系统结构与第5章的详细实现提供依据。

\section{系统设计}

\subsection{设计原则与模块关联总览}

系统设计遵循三条原则：

（1）职责分离——自然语言生成、语句校验、数据库访问、结果转换、前端渲染各有明确边界；

（2）横切安全——凡进入 Neo4j 的语句必经同一安全与参数策略，避免 NL 与直接 Cypher 两条路径不一致；

（3）可观测——生成语句与错误原因可返回给调用方，便于调试与论文复现。

从宏观关联看，五大块（NL、执行、安全、转换、可视化）并非并列孤立：NL 与执行共享“入口路由 + Neo4j 客户端”；安全嵌入在“执行之前”；转换紧接“记录出库之后”；可视化消费“转换后的 JSON”。图~\ref{fig:modules}给出模块划分，下文各小节在描述单模块时均会回扣上述数据与控制流顺序。

\subsection{系统总体架构}

系统采用分层架构设计，分为表示层、业务逻辑层、数据访问层和外部服务层。为何采用分层：便于替换实现（例如换 LLM 厂商不换库）、便于单测（对 Neo4j 与 LLM 做 Mock）、便于向评审说明“请求从哪进、从哪出”。

\begin{enumerate}[(1)]
    \item 表示层：前端 HTML 页面，使用 ECharts 进行图谱可视化，通过 JavaScript 与后端 API 交互。与下层接口：仅依赖 JSON 合同，不感知 Neo4j 协议细节。
    \item 业务逻辑层：FastAPI 应用，包含 API 路由、自然语言编排、数据转换等。与下层接口：调用 Neo4j 客户端与 httpx LLM 客户端；对上统一返回 HTTP + JSON。
    \item 数据访问层：Neo4j 客户端，负责会话、只读模式与超时。为何独立一层：避免路由处理函数内散落驱动代码，便于集中调优连接参数。
    \item 外部服务层：大语言模型 API。为何单列：其可用性与配额不由本机完全控制，在架构上视为外部依赖，便于标注风险与降级策略（如返回明确错误码）。
\end{enumerate}

\subsubsection{自然语言查询模块}

设计动因：若缺少该模块，非专业用户无法把“中文问句”稳定映射为“合法且高效的 Cypher”；若完全信任模型输出而不接安全与执行层，则无法落地为可靠系统。

与相邻模块的关系：Schema 经数据访问层获取；提示词拼装后调用外部 LLM；生成文本再经安全校验模块与Cypher 执行模块；出库记录由可视化/转换模块统一格式化，形成闭环。

该模块是系统的核心功能，主要流程如图~\ref{fig:nlq-flow}所示：

（1）用户输入自然语言查询；

（2）系统获取当前知识图谱的Schema信息（节点标签、关系类型）；

（3）构建Prompt，将Schema信息和用户查询发送给LLM；

（4）LLM生成Cypher查询语句和参数；

（5）系统进行安全校验（只读关键字黑名单；可选的EXPLAIN预检，默认关闭，见\texttt{ENABLE\_EXPLAIN\_VALIDATE}）；

（6）执行查询，获取结果；

（7）将结果转换为ECharts图数据或表格数据返回。

如何达成较好效果：Prompt 采用系统提示词 + Few-shot，强调多标签 Schema、路径型 \texttt{RETURN path} 与参数化占位符，引导 LLM 生成可执行且利于前端画边的只读查询；自然语言到结构化查询（如 SQL）的深度学习综述可与“NL→Cypher”任务对照阅读\cite{ref5}。最终仍须经黑名单校验，避免“能生成”与“能执行且安全”混为一谈。

\subsubsection{Cypher执行模块}

设计动因：调试人员、课程演示或“已知正确 Cypher”场景需要绕过 LLM，直接验证图数据；若与 NL 模块分裂为两套执行路径，则安全策略易不一致。

与相邻模块的关系：与 NL 模块共用 Neo4j 客户端与同一套只读/参数校验逻辑；查询成功后同样进入转换模块输出图与表，保证前端无分支差异。

（4）执行查询并返回结果。

\subsection{数据库设计}

本系统使用 Neo4j 图数据库存储《迷你世界》知识图谱，采用多标签属性图模型。为避免术语歧义，这里先说明本文中的两个领域词：方块指游戏世界中可采集或可放置的地形/材料单元，生物指可交互实体（如动物、怪物或 NPC）。数据库设计充分利用 Neo4j 的多标签特性，将方块和生物作为“道具”这一上位概念的子类，通过复合标签表达继承关系。

与上层模块的衔接：Schema（标签、关系类型）直接喂给 NL 模块的 Prompt；节点上的 \texttt{Name}、\texttt{MineTool} 等属性既影响 LLM 生成的 \texttt{WHERE} 子句，也影响转换模块如何归类节点颜色与表格列；因此数据建模是与查询体验、可视化效果联动的整体设计。

\subsubsection{架构演进：从 GDB 单标签桥接到 Neo4j 多标签}

早期曾考虑兼容阿里云 GDB（Graph Database）等不支持同一节点多标签的图引擎。在该约束下，无法直接使用 \texttt{:item:block}、\texttt{:item:monster} 这类复合标签表达“方块/生物是一种道具”的继承语义，只能将方块、生物与通用道具拆成不同节点，再通过业务 ID 用桥接关系（历史上曾建模为 PLACE、SUMMON 等）与道具节点相连。

该单标签桥接方案带来两类问题：（1）查询路径变长，自然语言转 Cypher 时模型需记住“先找道具再经 PLACE/SUMMON 跳到方块/生物”的固定套路，生成错误率显著上升；（2）图谱语义与玩家直觉不一致，人工理解与维护成本高。

当前实现已取消 GDB 兼容路径，数据仅面向 Neo4j 导入与查询：方块与生物直接采用多标签节点建模，不再使用 PLACE、SUMMON 等桥接关系。后文 Prompt 与测试用例均以此为准；论文中若仍偶见 PLACE/SUMMON 字样，均指上述已废弃的旧版建模，与现库无关。

\subsubsection{多标签架构设计}

Neo4j支持为一个节点设置多个标签（Multiple Labels），本系统利用这一特性设计继承关系：

\begin{itemize}
    \item 基础标签 :item - 所有游戏道具的父类，包含通用属性（ID, Name, Type, Disc, GetWay）
    \item 复合标签 :item:block - 方块类，继承:item所有属性，同时具有方块专属属性（MineTool, ToolLevel）
    \item 复合标签 :item:monster - 生物类，继承:item所有属性，同时具有生物专属属性（Life, Attack）
\end{itemize}

这种设计的优势：
\begin{enumerate}[(1)]
    \item 查询简化：相较旧版单标签桥接方案，无需经 PLACE/SUMMON 等关系跳转即可获取方块/生物完整属性，如`MATCH (n:item:block) RETURN n.Name, n.MineTool`
    \item 语义清晰：标签本身表达了类继承关系，符合领域模型
    \item 性能优化：避免了额外的JOIN操作，查询效率更高
\end{enumerate}

\subsection{自然语言查询模块的设计与实现}

设计要点。本模块负责将中文问句映射为参数化、只读、利于路径可视化的 Cypher。设计约束包括：（1）在调用 LLM 之前由后端拉取 Schema 摘要并注入提示词，使生成语句与当前库标签/关系类型对齐；（2）强制模型以 JSON 返回 \texttt{cql} 与 \texttt{params}，便于后续校验与 Neo4j 驱动绑定；（3）Prompt 中通过角色设定、多标签说明、Few-shot 与路径优先策略，降低“孤立节点返回”与“有向边漏匹配”概率，与第4章 NL 模块设计动机一致。

实现要点。对外入口为表~\ref{tab:api}中的 \texttt{POST /nlq}；实现上由 FastAPI 路由编排“取 Schema → 调 \texttt{llm\_client.generate\_cypher} → 走安全模块 → 调 Neo4j → 调转换器”的流水线。系统提示词可由仓库内 \texttt{system\_prompt.txt} 外置维护。以下给出请求/响应 JSON 形状及 Prompt 关键结构，便于与第4章时序图对照。

请求体格式：
\begin{lstlisting}[language=json, caption={自然语言查询请求体}]
{
    "query": "查找石剑",
    "options": {
        "limit": 100,
        "debug_raw": false
    }
}
\end{lstlisting}

响应体格式（字段名 \texttt{cql} 为历史命名，内容为Cypher字符串）：
\begin{lstlisting}[language=json, caption={自然语言查询响应体}]
{
    "cql": "MATCH (n:item) WHERE n.Name CONTAINS $name RETURN n",
    "params": {"name": "石剑"},
    "graph": {
        "nodes": [...],
        "links": [...],
        "categories": [...]
    },
    "table": {...}
}
\end{lstlisting}

Prompt 与生成策略（设计展开）。系统通过精心设计的Prompt引导大语言模型生成可执行且满足只读约束的Cypher查询。Prompt采用结构化设计，充分考虑了Neo4j多标签架构的特点。

（1）角色定义

首先明确告知LLM扮演Neo4j Cypher专家的角色，并限定其只生成只读查询，禁止任何写操作。

（2）Schema描述

详细描述知识图谱的多标签结构：
\begin{itemize}
    \item 基础标签：:item（所有道具的父类，包含通用属性ID, Name, Type, Disc, GetWay）
    \item 复合标签：:item:block（方块，继承:item并添加MineTool, ToolLevel属性）、:item:monster（生物，继承:item并添加Life, Attack属性）
    \item 其他标签：:group（分组）、:recipe（配方）、:smelt（熔炼）、:device（设备）
    \item 关系类型：IN\_GROUP、CONSUMES、PRODUCES、DROPS、TOOLMINEDROPS等12种关系
\end{itemize}

（3）多标签查询语法

Prompt中特别强调多标签查询语法：
\begin{itemize}
    \item 查询方块：\texttt{MATCH (n:item:block) WHERE n.Name CONTAINS '关键词'}
    \item 查询生物：\texttt{MATCH (n:item:monster) WHERE n.Name CONTAINS '关键词'}
    \item 查询纯道具：\texttt{MATCH (n:item) WHERE NOT n:block AND NOT n:monster}
\end{itemize}

（4）路径查询设计（核心）

引导LLM生成返回Path对象的查询，使用无方向关系最大化匹配结果。

错误示例（返回孤立节点）：
\begin{lstlisting}[language=cypher]
MATCH (r:recipe)-[:PRODUCES]->(p:item)
MATCH (r)-[:CONSUMES]->(c:item)
RETURN r, p, c  -- 只返回节点，前端无法显示关系
\end{lstlisting}

不推荐（有方向，可能遗漏）：
\begin{lstlisting}[language=cypher]
MATCH path = (material:item)<-[:CONSUMES]-(r:recipe)-[:PRODUCES]->(product:item)
RETURN path  -- 有方向限制，可能漏匹配
\end{lstlisting}

正确示例（无方向路径查询）：
\begin{lstlisting}[language=cypher]
MATCH path = (material:item)-[:CONSUMES]-(r:recipe)-[:PRODUCES]-(product:item)
WHERE material.Name CONTAINS '木料'
RETURN path  -- 无方向，最大化匹配，返回完整路径
\end{lstlisting}

路径查询的优势：
\begin{itemize}
    \item 返回的Path对象包含完整的节点和关系信息
    \item 使用无方向关系（\texttt{-[]-}）最大化匹配所有可能结果
    \item 前端ECharts可以直接解析Path中的关系进行渲染
    \item 用户可以看到完整的知识网络，而非孤立的点
\end{itemize}

（5）Few-shot示例（无方向路径查询）

提供7个典型查询示例，展示无方向路径查询语法。需说明的是，下列“方块挖掘”“生物掉落”等标题来自本文案例数据中的领域称谓，仅用于演示查询模式；迁移到其他领域时，可保持相同路径模板，仅替换节点标签、关系类型与关键词。

\begin{lstlisting}[language=cypher, caption={方块挖掘路径查询（无方向）}]
-- 深积岩用什么工具挖
MATCH path = (n:item:block)-[:TOOLMINEDROPS]-(tool:item)
WHERE n.Name CONTAINS '深积岩'
RETURN path LIMIT 10
\end{lstlisting}

\begin{lstlisting}[language=cypher, caption={生物掉落路径查询（无方向）}]
-- 击败野人会掉落什么
MATCH path = (n:item:monster)-[:DROPS]-(drop:item)
WHERE n.Name CONTAINS '野人'
RETURN path LIMIT 20
\end{lstlisting}

\begin{lstlisting}[language=cypher, caption={合成配方路径查询（无方向）}]
-- 木料能合成什么
MATCH path = (material:item)-[:CONSUMES]-(r:recipe)-[:PRODUCES]-(product:item)
WHERE material.Name CONTAINS '木料'
RETURN path LIMIT 20
\end{lstlisting}

\begin{lstlisting}[language=cypher, caption={配方原料路径查询（无方向）}]
-- 石剑的合成配方
MATCH path = (product:item)-[:PRODUCES]-(r:recipe)-[:CONSUMES]-(material:item)
WHERE product.Name CONTAINS '石剑'
RETURN path LIMIT 20
\end{lstlisting}

（6）输出格式要求

要求LLM返回JSON格式，包含cql（查询语句）和params（参数）两个字段。

上述六点给出的是提示词模板结构。为避免“模板写了但效果不稳”，下面补充部署与调试过程中可复用的经验规则。

Prompt 工程的细化与经验总结。

（1）路径查询优先

强制使用 \texttt{MATCH path = ()-[]-() RETURN path} 语法，确保返回的查询结果可以被前端正确渲染为知识图谱。使用无方向关系（\texttt{-[]-}）可以最大化匹配所有可能的结果，避免有方向查询导致的遗漏。

测试表明：
\begin{itemize}
    \item 路径查询相比孤立节点查询，可视化效果提升显著
    \item 无方向关系查询相比有方向查询，匹配结果更完整
    \item 复合路径查询可以展示完整的知识网络（如原料-配方-产物链）
\end{itemize}

（2）多标签架构优势

利用Neo4j多标签特性，方块和生物节点同时具有:item标签和专属标签（:block或:monster）。这使得：
\begin{itemize}
    \item 可以直接通过:item:block查询方块，同时获取Name和MineTool属性
    \item 无需使用已废弃的 PLACE/SUMMON 桥接关系，查询逻辑与现库一致
    \item 查询语义更清晰，符合领域模型（方块是一种道具）
\end{itemize}

（3）无方向关系匹配

在探索性查询中，使用无方向关系语法：
\begin{itemize}
    \item 无方向：\texttt{(a)-[:TYPE]-(b)} 可以匹配任意方向的关系
    \item 有方向：\texttt{(a)-[:TYPE]->(b)} 只能匹配从a到b的关系
    \item 在迷你世界知识图谱中，无方向查询可以捕获更多关联
\end{itemize}

（4）中文模糊匹配

由于迷你世界道具名称包含融合道具（如"黄铜斧+石剑"），Prompt中强调使用\texttt{Name CONTAINS}进行模糊匹配。

（5）参数化查询

强制使用参数占位符（\texttt{\$param}）代替直接值，降低Cypher注入风险。

LLM 调用与 JSON 解析（实现摘录）。自然语言模块将系统提示词（可由 \texttt{system\_prompt.txt} 外置）与用户问句拼装为 OpenAI 兼容的 \texttt{chat/completions} 请求，并强制 \texttt{response\_format} 为 JSON，便于提取 \texttt{cql} 与 \texttt{params}，与上文“输出格式要求”一致。

\begin{lstlisting}[language=Python, caption={LLM 生成 Cypher（llm\_client.py 摘录）}]
async def generate_cypher(self, nlq, schema_hint, limit):
    payload = {
        "model": self.model,
        "messages": [
            {"role": "system", "content": load_system_prompt()},
            {"role": "user", "content": f"只读 Cypher，LIMIT 默认 {limit}：\n{nlq}"},
        ],
        "temperature": 0.1,
        "response_format": {"type": "json_object"},
    }
    resp = await client.post(f"{self.api_base}/chat/completions", json=payload, headers=headers)
    obj = json.loads(resp.json()["choices"][0]["message"]["content"])
    return obj.get("cql", "").strip(), obj.get("params", {})
\end{lstlisting}

\subsection{图数据访问与 Cypher 执行模块的设计与实现}

设计要点。本模块将所有最终落库的 Cypher 执行收敛到单一 Neo4j 客户端封装：会话使用只读访问模式，查询带超时与数据库名配置，返回记录字典与列键序，供转换模块消费。\texttt{GET /schema} 为 NL 链路与人工浏览提供标签/关系摘要；\texttt{POST /run-cql} 为高级用户提供与 NL 链路相同的执行与安全语义；\texttt{GET /health} 便于部署探活。设计目标是在不复制驱动代码的前提下，使 NL 与直接 Cypher 两条入口共用第4章所述数据访问层契约。

实现要点。实现位于 \texttt{neo4j\_client.py} 等模块；路由层在通过 \texttt{cql\_validator} 校验后才调用 \texttt{run\_read}。下列代码摘录展示只读会话与超时换算，与第3章性能与非功能叙述一致。

\begin{lstlisting}[language=Python, caption={Neo4j 只读查询（neo4j\_client.py 摘录）}]
def run_read(self, cql: str, params: Dict[str, Any] | None = None):
    params = params or {}
    with self._driver.session(
        database=settings.NEO4J_DATABASE,
        default_access_mode="READ",
    ) as session:
        timeout_seconds = settings.QUERY_TIMEOUT_MS / 1000.0
        result = session.run(cql, parameters=params, timeout=timeout_seconds)
        records = [r.data() for r in result]
        return records, list(result.keys())
\end{lstlisting}

\subsection{结果转换与可视化模块的设计与实现}

设计要点。Neo4j 驱动返回的记录中可能同时包含节点、关系与路径对象；而 ECharts Graph 需要统一的 \texttt{nodes}/\texttt{links} 结构。设计上前端采用单页四区布局（输入、图谱、表格、生成语句回显），后端在 JSON 响应中并行返回 \texttt{graph} 与 \texttt{table}，使“看拓扑”与“读属性”互不替代。路径型 \texttt{RETURN path} 与无方向边模式由 NL 模块在 Prompt 侧引导，本模块负责忠实拆解驱动返回的结构，避免丢失边。

转换算法步骤。将 Neo4j 返回的记录转换为 ECharts Graph 格式的步骤为：（1）遍历查询结果中的每个记录；（2）识别节点（Node）和关系（Relationship）对象；（3）为每个节点生成唯一 ID，提取标签作为类别，属性作为节点数据；（4）为每条关系提取起始节点、目标节点和关系类型；（5）进行节点去重，处理孤儿节点。

实现要点。核心逻辑在 \texttt{echarts\_converter.py}；下列摘录展示关系边入库时如何保证端点节点先行入表。

\begin{lstlisting}[language=Python, caption={图数据转换（echarts\_converter.py 摘录）}]
def add_rel(rel: graph.Relationship) -> None:
    src, tgt = _node_id(rel.start_node), _node_id(rel.end_node)
    add_node(rel.start_node)
    add_node(rel.end_node)
    links.append({
        "source": src, "target": tgt,
        "category": rel.type, "label": rel.type,
        "value": dict(rel),
    })
\end{lstlisting}

\subsection{安全与约束模块的设计与实现}

设计要点。安全模块以横切方式作用于 \texttt{/nlq} 与 \texttt{/run-cql}：在调用 Neo4j 之前对语句做只读关键字黑名单匹配，可选 \texttt{EXPLAIN} 预检（由配置项控制，默认关闭以降低图库负载），并对 \texttt{\$param} 占位符与请求体参数做对齐检查。设计动机是：即便数据库账号配置为只读，仍应在应用层拒绝明显写操作与危险 \texttt{CALL dbms.*}，并向调用方返回可读错误，满足第3章非功能需求中的安全性条目。

实现要点。校验逻辑集中在 \texttt{cql\_validator.py}，由路由在执行业务前统一调用。下列黑名单与第6章表~\ref{tab:readonly-test} 中测试用例一一对应。

使用正则表达式黑名单匹配禁止的操作：

\begin{lstlisting}[language=Python, caption={只读校验黑名单（与 cql\_validator.py 一致）}]
WRITE_BLACKLIST = [
    r"\bCREATE\b",
    r"\bMERGE\b",
    r"\bDELETE\b",
    r"\bDETACH\s+DELETE\b",
    r"\bSET\b",
    r"\bREMOVE\b",
    r"\bDROP\s+(?:INDEX|CONSTRAINT|NODE\s+LABEL|REL\s+TYPE|PROPERTY)\b",
    r"\bLOAD\s+CSV\b",
    r"\bCALL\s+DBMS\.",
]
\end{lstlisting}

除上述关键字拦截外，系统未对 \texttt{CALL} 自定义过程（如APOC插件）做白名单约束；在不可信网络环境下部署时，应优先使用Neo4j只读账号或只读副本，并将服务置于内网或网关之后。

参数校验（实现说明）。使用正则表达式提取 Cypher 中的参数占位符 \texttt{\$param}，与请求体提供的参数对比，确保必需参数齐全（\texttt{/run-cql} 与 \texttt{/nlq} 共用逻辑，实现见仓库路由层）。

\begin{thesisconclusion}

本文设计并实现了一个基于自然语言的知识图谱查询与可视化系统（neo4jSlave），针对《迷你世界》游戏知识图谱进行了实际应用验证。

\subsection*{主要工作}

（1）系统架构设计

系统采用前后端分离的B/S架构，后端使用FastAPI框架提供RESTful API服务，前端使用纯HTML/CSS/JavaScript结合ECharts实现知识图谱可视化。这种架构具有轻量级、易部署、跨平台的特点。

（2）自然语言查询模块

设计并实现了基于大语言模型的自然语言到Cypher查询转换模块。通过精心设计的Prompt模板，包含角色定义、Schema描述、Few-shot示例和输出格式要求，引导LLM生成可执行的只读Cypher。人工评测（21条，2026年4月6日）总体通过率约81.0\%；道具、合成、掉落、生物类问句表现较好，分组键、StationLabel 与多实体对比类仍为短板。

（3）安全校验机制

实现了多层安全保护机制：
\begin{itemize}
    \item 正则表达式黑名单：禁止CREATE、MERGE、DELETE等写操作关键字
    \item 可选的EXPLAIN预检（默认关闭）：需要时在部署环境开启后，可借助执行计划做额外检查
    \item 参数校验：确保所有必需参数都已提供，防止注入攻击
\end{itemize}

（4）知识图谱数据

成功导入《迷你世界》游戏知识图谱数据：道具侧可拆分为纯物品、方块、生物等互斥类别（见表~\ref{tab:data-scale}，节点总计7,766），并包含配方、分组、熔炼、设备等实体；有向关系合计12,240条（统计口径见该节说明）。图谱涵盖合成、掉落、挖掘等多类游戏机制。

\subsection*{创新点}

\begin{enumerate}[(1)]
    \item 游戏领域应用：将自然语言查询技术应用于游戏知识图谱领域，降低了玩家查询游戏资料的门槛
    \item 中文优化：针对中文游戏数据特点，优化Prompt设计，支持中文模糊匹配
    \item 轻量级架构：不依赖重量级前端框架，使用原生技术栈实现，部署简单
    \item 模块化设计：各功能模块解耦，便于更换LLM后端或适配其他知识图谱
\end{enumerate}

\subsection*{不足与展望}

当前系统存在以下不足：
\begin{enumerate}[(1)]
    \item 复杂查询与分组、多实体对比在人工评测中仍有失误（表~\ref{tab:nlq-category}中“复杂查询”“关系查询”“边缘情况”等类为50.0\%；整体NLQ约81.0\%，距离稳健生产应用仍有差距）
    \item 缺乏同义词处理能力，用户对同一概念的不同表述无法正确识别
    \item 未实现用户认证和查询历史记录功能
\end{enumerate}

未来工作方向：
\begin{enumerate}[(1)]
    \item 引入检索增强生成（RAG）技术，结合向量数据库提升查询准确性
    \item 构建游戏领域同义词词典，支持语义等价表述
    \item 添加用户反馈机制，根据用户点击行为优化查询结果排序
    \item 支持多数据源融合，实现跨知识图谱的联合查询
\end{enumerate}

\end{thesisconclusion}
