\begin{abstract}

知识图谱以图结构表示实体及实体间的关系，可展示数据中隐含的知识联系，在知识管理、智能问答与推荐等领域应用较广。在属性图数据库中，查询实体之间的关系通常需要使用专门的图查询语言，使用者需掌握相应的语法规则与数据库结构，且查询结果以表格方式呈现，难以直观反映图结构信息。这为非数据库专业的普通用户设置了两类门槛：一是需专门学习查询语言，二是难以从表格结果中直观理解实体间的关联。

本文设计并实现了基于生成式AI的知识子图建构与可视化系统（系统代号 neo4jSlave），采用浏览器/服务器（B/S）架构。用户在浏览器中以中文提出问题，后端先获取当前图数据库的节点标签与关系类型等结构信息，连同问句一并送入大语言模型，得到只读的参数化 Cypher 查询语句；语句经应用层安全校验后在 Neo4j 图数据库中执行。系统将查询记录在后端转换为节点—边结构，并在前端同步展示关系图与属性表格。

验证分为自动化测试与自然语言实例测试两部分。自动化测试覆盖接口行为、安全约束以及查询结果到图结构的转换逻辑，38 条用例均已通过。自然语言实例测试以《迷你世界》游戏知识图谱为测试对象，对 21 条中文问句逐条评判，17 条得到可接受的查询结果，通过率约为 81\%。

本系统为缺少数据库专业背景的用户提供了从中文问句到知识子图建构与图形化浏览的完整通路，适用于领域知识库的轻量查询入口与教学演示等场景。

\end{abstract}

\addcontentsline{toc}{section}{Abstract}
\begin{thesisabstracten}

Knowledge graphs represent entities and their relationships in a graph structure, enabling intuitive access to the knowledge embedded in a database. They are widely applied in knowledge management, question answering, and recommendation systems. Querying a property graph database typically requires a dedicated graph query language: users must understand node labels, relationship types, and query syntax, while results are returned as flat tables that do not convey the topology of the graph. These two barriers---learning the query language and interpreting tabular results---make knowledge graphs inaccessible to non-specialist users.

This thesis presents neo4jSlave, a browser/server system for generative-AI-assisted subgraph construction and visualisation over a knowledge graph. When a user submits a Chinese question, the backend retrieves a summary of the current graph schema (node labels and relationship types) and passes it together with the question to a large language model, which returns a read-only, parameterised Cypher statement. The statement is validated by an application-layer safety module before execution in Neo4j. Returned records are transformed into a node--link structure and displayed as both an interactive graph and a table in the browser.

Evaluation consists of automated tests and a manual walkthrough. The automated suite (38 test cases) covers API contracts, the read-only safety validator, and subgraph conversion logic; all cases passed. The manual walkthrough tested 21 Chinese questions against the Mini World game knowledge graph under a fixed model and acceptance protocol; 17 questions (81\%) received acceptable results. The system provides a complete read-only pipeline from natural-language input to subgraph construction and graph visualisation, lowering the access barrier for domain knowledge bases and supporting educational demonstrations.
\end{thesisabstracten}

\section{系统概述}

\subsection{研究背景与意义}

知识图谱以有向图描述实体及实体间的关系，节点代表实体，边代表关系，形成能够揭示数据内在联系的知识表示形式。这种组织方式在知识管理、智能问答与个性化推荐等任务中应用较广，已成为信息系统领域的重要基础设施\cite{ref17}。在工程实现层面，知识图谱的落地路线主要分为两类：以三元组为基础的 RDF 模型，以及以节点关系为基础的属性图模型。RDF 模型适合开放数据互联与本体推理；属性图在节点和边上直接附加键值属性，贴近业务建模，便于迭代维护\cite{ref1,ref3}。本文采用属性图路线，以 Neo4j 图数据库存储和查询知识图谱，对应的图查询语言为 Cypher。

在属性图数据库中，查询实体及其关系通常需要书写 Cypher 语句。这要求使用者预先掌握节点标签、关系类型与方向、模式匹配语法以及属性条件的写法，还需要能够理解以表格形式返回的查询记录。对于偶尔需要查询的非专业用户，上述学习成本较高；而仅有表格结果也难以直观呈现“谁与谁相连”这类拓扑信息。

近年来，大语言模型（Large Language Model，LLM）在自然语言到结构化查询任务上取得了显著进展\cite{ref4,ref11}，使“用户以中文提问、系统自动生成 Cypher”这一工程路线具备实现条件。将自然语言接口与图形化结果展示相结合，可以有效降低非专业用户的使用门槛，使其无需专门学习 Cypher 即可完成知识图谱的基本查询与浏览。

本文以《迷你世界》游戏知识图谱为测试场景，设计并实现了一套基于生成式AI的知识子图建构与可视化系统。从实践角度看，该系统为领域知识库提供了轻量的浏览入口，适用于教学演示、跨角色协作与业务查询等场景。从方法角度看，本文将自然语言生成、应用层安全约束、查询结果子图建构与图形化展示组织为一条完整的可运行链路，提供可复现的工程实现路径。

\subsection{国内外研究现状}

\subsubsection{知识图谱的存储与查询}

知识图谱的数据组织与查询主要有两条技术路线。第一条是 RDF 与 SPARQL：该路线将知识表示为主谓宾三元组，便于开放发布与跨数据集互联，并可结合本体进行逻辑推理\cite{ref17}。第二条是属性图：以节点和有向关系表示实体结构，在节点或关系上直接附加属性，查询语言围绕模式匹配与路径展开。已有研究从图数据管理、存储引擎与查询优化角度，探讨了图数据库在索引、事务与可扩展性上的关键问题\cite{ref1,ref2,ref3}。

在工程选型上，当数据已以属性图形式组织、目标是以较少组件交付可运行 Web 查询服务时，Neo4j 在文档、驱动生态与社区资源上均较为成熟\cite{ref12}。然而，仅完成“数据可读与语句可执行”并不足以解决终端用户的使用问题——查询入口仍是手动编写 Cypher 时，用户仍需掌握当前图数据库的结构与语法规则。这一缺口是引入自然语言接口的直接动因。

\subsubsection{自然语言到结构化查询}

将自然语言转换为可执行的结构化查询语句是降低数据库使用门槛的核心方法。文本到 SQL（Text-to-SQL）是该方向中研究积累最为丰富的任务；综述工作表明，影响生成质量的两类关键因素是对当前数据库结构的感知能力与生成语句的可执行性\cite{ref5}。这一结论对自然语言到 Cypher 任务同样具有参考价值。

在属性图场景下，自然语言到 Cypher（Text2Cypher）任务已有专项研究。结果表明，在不针对该任务专门训练模型的情况下，零样本生成对复杂语义、多跳约束与细粒度条件的处理仍存在不稳定性；通过有监督微调可在评测指标上取得提升\cite{ref16}。因此，工程系统除依赖模型本身外，还需结合 Schema 感知的提示设计与执行后处理来提升生成质量。

\subsubsection{大语言模型与知识图谱的协同}

大语言模型与知识图谱的协同研究涵盖多个方向，包括以图结构为模型补充外部知识、统一符号与向量表示，以及辅助推理与增强结果可解释性\cite{ref11}。在自然语言查询子场景中，两者分工相对清晰：大语言模型负责将问句转化为候选查询语句，图数据库负责确定性执行并返回结构化结果，应用层负责执行前后的安全约束与结果整理\cite{ref4,ref9,ref11}。

这种结构兼顾了灵活性与可控性，但也面临若干风险：模型生成的语句可能与实际图结构不符，同一问句在不同调用下可能输出不一致的结果，以及执行权限被误用等。针对这些风险，需要通过规则约束与执行前后的验证机制加以抑制。本文采用“受约束生成结合应用层校验”的工程路径，在通用大语言模型与已有属性图数据的条件下，构建可复现的中文问句查询服务。

\subsection{主要工作与论文组织}

本文的主要工作围绕四个方面展开。第一，设计并实现了以 B/S 架构为基础的知识子图建构与可视化系统，后端基于 FastAPI 提供自然语言查询、直接 Cypher 查询与图数据库结构摘要三类接口，前端使用原生 HTML/JavaScript 实现单页面交互，以 ECharts 绘制可交互的关系图。第二，在自然语言查询路径上，系统先获取当前图数据库的结构信息，连同用户问句注入提示词后送入大语言模型，得到只读的参数化 Cypher 语句。第三，在执行链路上，所有进入数据库的语句须经应用层统一校验，包括写操作关键字过滤、参数占位符一致性检查、查询超时与返回行数上限控制；执行返回的记录统一转换为前端可直接渲染的节点—边结构。第四，以《迷你世界》知识图谱为领域数据，完成自动化测试与自然语言实例测试，对系统功能正确性、子图建构效果与可视化可用性进行验证。

本文其余章节组织如下：第 2 章介绍系统的开发环境与关键技术选型；第 3 章对系统进行可行性与需求分析；第 4 章给出系统的整体架构与各功能模块的设计；第 5 章描述各模块的实现细节；第 6 章给出系统测试结果与分析。

\section{系统开发环境与技术选型}

\subsection{开发与运行环境}
\label{subsec:repo-and-paths}

本系统在 GNU/Linux 环境下开发与调试，典型发行版为 Ubuntu 22.04 LTS 和 Debian 12。宿主机需部署可访问的 Neo4j 实例，既可连接本机服务（如 \texttt{bolt://127.0.0.1:7687}），也可连接局域网或云主机上的远程实例。后端使用 Python 3.10 及以上版本，通过虚拟环境管理第三方依赖，主要依赖包括 FastAPI、neo4j 驱动、httpx 与 pytest。前端采用原生 HTML/CSS/JavaScript，浏览器直接加载已编译的 ECharts 库文件，无需 Node.js 构建步骤。版本控制使用 Git，源代码托管于 GitHub：\url{https://github.com/OWALabuy/neo4jSlave}。

系统对运行环境的主要约束来自后端和图数据库两端。后端要求 Python 3.10 以上版本，部分类型注解写法依赖该版本引入的联合类型语法；建议通过 \texttt{python -m venv} 创建独立虚拟环境，避免依赖版本与系统已有软件冲突。Neo4j 官方 Python 驱动通过 Bolt 协议与数据库服务通信，驱动版本应与数据库服务器的主版本号对应；本系统以 Neo4j 5.x 为基准开发与测试。

系统运行时所需的外部配置通过环境变量注入，不写入源代码。仓库提供 \texttt{.env.example} 文件，列出所有变量的键名与说明，主要包含数据库连接信息与大语言模型接口参数，见表~\ref{tab:env-repro}。更换数据库实例或切换大语言模型服务商时，只需修改对应的环境变量值，无需改动任何业务代码。

\subsection{关键技术选型}

本系统的技术路线围绕三项目标构建：降低查询入口门槛、增强结果可理解性、控制生成语句的执行风险。

FastAPI 基于 Starlette 与 Pydantic，支持异步路由与请求体自动校验。与 Flask 相比，其异步特性更适合同时处理大语言模型网络调用与数据库 IO 的混合场景；与 Django 相比，其体量更轻，契合本系统“单一服务托管静态前端”的部署形态\cite{ref6,ref10}。

本系统每次自然语言查询请求包含两次网络等待：调用大语言模型通常需要 1 至 3 秒，随后的图数据库查询需要数十至数百毫秒。FastAPI 基于 Python 的协程机制，在等待网络响应期间可以切换处理其他请求，而不会因为某一次模型调用延迟阻塞其他用户的请求。Pydantic 的请求体自动校验则将接口层的类型错误在进入业务逻辑之前拦截，减少后续处理环节的防御性判断代码量。

Neo4j 使用属性图模型与 Cypher 查询语言，社区文档与驱动生态均较为成熟，且支持多标签节点。多标签特性与本文领域数据中道具、方块、生物之间的继承式建模一致\cite{ref7,ref12}。未选用 JanusGraph 或 ArangoDB 的主要原因是本课题已有 Neo4j 导入管线与实测数据，同时 Cypher 在大语言模型训练语料中覆盖较好，有利于提示词设计。

选择 Neo4j 还有一个来自提示词设计角度的考量。Cypher 以类图形化符号表达节点与关系，例如 \texttt{(a)-[:KNOWS]->(b)} 直接对应“a 认识 b”的语义，语法结构与自然语言的对应较为直观。与 SPARQL 等其他图查询语言相比，Cypher 在近年大语言模型训练语料中的覆盖程度更好，通用大语言模型在将自然语言转换为 Cypher 时具有更高的成功率，有利于降低提示词调优的工作量\cite{ref16}。

大语言模型通过 OpenAI 兼容的 HTTPS 接口接入，端点与模型名由环境变量配置，便于在不修改业务代码的情况下切换不同服务商\cite{ref4,ref9}。实例测试时使用阿里云通义百炼平台的 qwen-turbo 模型。

采用 API 而非本地部署模型，主要出于对推理资源与维护成本的权衡。本地部署具备足够推理能力的模型通常需要独立的 GPU 资源，在课程演示与小规模验证场景下，部署与维护成本偏高。商业 API 服务以按量计费的方式提供推理能力，调用延迟稳定，无需在本地维护模型运行环境。OpenAI 兼容接口目前已成为多家大语言模型服务商的通行接口格式，采用该格式后，切换服务商只需修改环境变量，代码层面无需改动。

ECharts 是 Apache 基金会的开源可视化库，关系图组件支持力导向布局与拖拽缩放交互，适合在 Web 单页面中嵌入使用\cite{ref8}。各组件与候选方案的对比概括见表~\ref{tab:tech-compare}。

在知识图谱查询场景中，用户最关心的是实体之间“谁与谁相连”的拓扑关系，而非坐标轴上的数值。力导向布局算法根据节点间的连接数量动态调整位置，连接紧密的节点自然聚拢，连接稀疏的节点自动分散，使图谱的结构特征无需人工干预即可直观呈现。D3.js 同样支持力导向图，但使用门槛较高，需要编写较多底层绑定代码；Cytoscape 更侧重生物信息学场景，预置交互模式与本系统需求不完全吻合。ECharts 关系图组件开箱即用，可直接在 HTML 页面中通过 JavaScript 初始化，与本系统“前端无需构建步骤”的部署目标一致。

\subsection{系统总体架构与数据流}

系统采用分层架构：浏览器中的页面（表示层）、FastAPI 应用（业务逻辑层）、Neo4j 官方 Python 驱动（数据访问层），以及外部的大语言模型 HTTPS 服务，如图~\ref{fig:architecture}所示。

\begin{figure}[H]
    \centering
    \includegraphics[width=0.95\textwidth]{img/architecture.png}
    \caption{系统总体架构图}
    \label{fig:architecture}
\end{figure}

自然语言查询的主要数据流如下：用户在浏览器中提交中文问句，请求发至 FastAPI；后端先获取当前图数据库的节点标签与关系类型，将其与问句一同写入提示词，发送至大语言模型；模型返回包含 Cypher 语句与参数的结构化输出；后端对语句进行只读校验与参数一致性检查，通过后在 Neo4j 只读会话中执行；查询结果经转换模块建构为节点—边结构与表格字段，返回给前端；浏览器使用 ECharts 绘制关系图，并以表格展示属性数据。

高级用户也可绕过自然语言层，直接通过 \texttt{/run-cql} 接口提交 Cypher 语句。该路径与自然语言路径使用同一套安全校验、执行封装与结果转换逻辑，保证两条入口的执行语义一致。

\subsection{本章小结}

本章介绍了系统的开发与运行环境、关键技术选型及系统总体架构。开发环境以 GNU/Linux 为基础，后端使用 Python 虚拟环境管理依赖，所有外部配置通过环境变量注入，部署时不需要修改代码。

技术选型围绕“降低查询门槛、增强结果可理解性、控制执行风险”三项目标展开。FastAPI 的异步机制适配系统的 IO 密集型请求特征；Neo4j 的多标签属性图模型与领域数据的继承式结构吻合，且 Cypher 语法在大语言模型生成任务中具有较好的适配性；大语言模型以 API 方式接入，兼顾推理性能与维护成本；ECharts 的力导向关系图组件直接满足图谱拓扑可视化的核心需求。

总体架构采用“浏览器—FastAPI—Neo4j”三层分层结构，外挂大语言模型 HTTPS 服务。数据流以“问句输入—Cypher 生成—安全校验—数据库执行—结果转换—前端渲染”为主线，各层之间通过定义清晰的 HTTP JSON 接口交互，耦合度低，为后续系统设计与实现提供了稳定的结构基础。

\section{系统分析}

\subsection{可行性分析}

从技术可行性看，本系统所采用的各项技术均有成熟的开源或商业实现，接口标准清晰，不存在技术上的壁垒。FastAPI 已在工业界广泛使用，文档完善；Neo4j 拥有完整的 Python 官方驱动与大量使用案例；大语言模型的 OpenAI 兼容接口已成为业界通行标准，多家服务商均支持该格式；ECharts 的 CDN 版本可直接在 HTML 中引入，无需构建环境。各组件之间通过 HTTP JSON 接口交互，耦合度低，任意一个组件出现问题时，排查范围可以快速定位到对应的服务层，不会引发全局性的系统故障。后端可在单机完成“服务启动—图库连接—JSON 返回”的基本链路，部署所需的软件环境可在标准 Linux 主机上通过包管理工具完成安装，无需专项运维。

从经济可行性看，本系统的开发期成本主要来自人力投入，不需要购买专用硬件或商业软件授权。Neo4j 社区版可免费使用，涵盖本系统所需的图数据库全部核心功能；Python 及所有后端依赖均为开源软件，无许可费用；前端不使用商业 UI 组件库，全部基于原生 Web 技术实现。运行期的主要可变成本来自大语言模型 API 的调用费用，以阿里云通义百炼 qwen-turbo 模型为例，每千个输入 token 的费用极低，在日常演示频率下产生的费用可忽略不计。系统通过超时机制与返回行数上限控制单次查询的资源消耗，进一步防止意外的高额调用费用产生。

从运行可行性看，终端用户仅需通用浏览器即可使用本系统，无需安装专用客户端。服务端的典型部署方案为一台可以运行 Python 的 Linux 主机，不需要专用服务器或 GPU，在个人笔记本上即可完成演示。自然语言查询与直接 Cypher 查询在同一页面切换，系统同时展示生成的查询语句，便于用户核对意图与结果的一致性，也适合课程演示与教学对照场景。中文自然语言界面降低了知识图谱的使用门槛，特别适合需要查询图数据库但不具备 Cypher 编写能力的使用者，无需向使用者提供数据库查询语言的专项培训。

\subsection{需求分析}

\subsubsection{系统边界与用例}

系统直接服务两类用户。第一类是终端用户，通过浏览器输入中文问句，查看知识图谱的图形化结果，通常不具备 Cypher 编写能力。第二类是高级用户，可直接提交 Cypher 语句，用于调试、对照自然语言生成结果或进行教学演示。两类用户均通过同一 Web 前端访问后端接口。系统的外部依赖包括大语言模型服务与 Neo4j 图数据库，两者均作为独立服务存在，不包含在本文实现的系统边界之内。本文系统不包含任何图数据库写入能力，所有数据库访问均限定为只读查询。系统用例如图~\ref{fig:use-case-placeholder}所示。

两类用户的使用方式决定了系统需要支持两条执行路径，且两条路径在底层共享核心实现。终端用户路径依赖大语言模型将自然语言转换为 Cypher，中间生成过程对用户透明，用户只感知最终的图谱与表格；高级用户路径绕过语言模型，直接进入校验与执行阶段，主要用于验证和教学演示。两条路径在 Schema 查询、语句安全校验、数据库访问和结果转换四个环节使用同一套实现，既减少了重复代码，也保证了两条路径在安全策略与数据格式上的一致性。

\subsubsection{功能需求}

系统的功能需求围绕“能查询、子图可建构、结果可理解、可安全执行”四类目标展开，共包含六项功能。自然语言查询是面向终端用户的主入口，用户输入中文问句后，系统调用大语言模型生成只读的参数化 Cypher 语句，经安全校验通过后执行，并将结果建构为图谱与表格两类数据返回。直接 Cypher 查询通过 \texttt{/run-cql} 接口提供，主要用于调试与教学对照，与自然语言入口使用同一套安全校验与执行逻辑。

知识图谱可视化将查询结果中的节点、关系与路径映射为可在浏览器中拖拽缩放的力导向关系图，便于观察实体之间的拓扑连接。数据表格与关系图并行展示，用于呈现节点的文本属性与数值字段。图数据库结构查询通过 \texttt{/schema} 接口提供当前图库的节点标签、关系类型与关键属性列表，既作为自然语言生成前的上下文输入，也供用户直接查阅。安全约束作为横切功能作用于两条执行路径，在语句进入数据库前过滤写操作关键字、检查参数占位符完整性，并与查询超时、返回行数上限共同控制执行风险。各功能对应的用户故事见表~\ref{tab:user-stories}。

六项功能之间存在明确的依赖顺序，而非简单的并列关系。Schema 查询是自然语言查询的前提：系统在每次调用大语言模型前，都会先拉取当前图库的标签与关系类型，将其嵌入提示词。安全约束位于所有查询执行之前，自然语言入口与直接 Cypher 入口产生的语句在进入数据库前均须通过同一套校验。可视化以执行结果为输入：只有数据库返回数据，前端才会渲染关系图与表格。这种依赖结构说明，六项功能共同构成一条从意图表达到结果展示的完整处理链路，改进或替换其中某一项均会对链路上的下游环节产生影响。

\subsubsection{非功能需求}

安全性方面，系统需在数据库账号权限控制之外，在应用层构建独立的语句级防护。即便 Neo4j 账号已配置为只读，模型生成或用户提交的语句中仍可能包含写操作或高危管理命令。系统在语句进入数据库前进行关键字黑名单匹配，并可选择在执行前请求数据库生成执行计划以进一步验证，默认关闭以减少额外数据库负载。这一双重防护形成“数据库权限 + 应用层校验”的纵深防御，单一层面的配置失误不会直接导致写操作被执行。

性能方面，自然语言路径包含一次大语言模型 API 调用，通常耗时约 1 至 3 秒，这一延迟主要由网络传输与模型推理时间决定，在当前架构下无法消除。在此基础上，数据库查询应保持在毫秒至数百毫秒量级，前端绘图应在可接受时间内完成，以确保整体响应时间不会因后续环节而进一步拉长。系统对单次查询设置超时时间与返回行数上限，避免因 Cypher 语句性能不佳而长时间阻塞请求处理线程。

可扩展性方面，系统将连接信息与密钥放入环境变量，并在程序结构上区分路由层、数据库客户端层与结果转换层，使替换某一组件时的影响范围局部化。以替换大语言模型为例，只需修改 \texttt{LLM\_API\_BASE} 与 \texttt{LLM\_MODEL} 两个环境变量，无需改动路由层或数据库层代码；若未来需要支持其他图数据库，只需在数据库客户端层添加适配器，上层路由与结果转换逻辑不受影响。在更大范围的场景中，将系统整体切换到另一个 Neo4j 知识图谱只需完成两项工作：按目标数据库的 Schema 重新编写系统提示词，以及修改部署配置中的数据库连接参数；后端代码与前端页面均无需改动，具体步骤见第~\ref{sec:system-design}~章。

易用性方面，用户在同一页面完成“提问—查看生成语句—查看图谱与表格—修改提问”的完整循环，无需在多个工具或页面之间切换。系统同时回显大语言模型生成的 Cypher 语句，使用户可以对照提问意图与实际执行的语句，判断生成结果是否符合预期。接口在请求不合法时返回具体的错误说明（如“包含写操作关键字”或“参数占位符不匹配”），而非笼统的服务器错误码，便于用户调整提问方式或排查配置问题。

\subsection{本章小结}

本章从可行性与需求两个维度对系统进行了分析。可行性分析从技术、经济与运行三个角度论证了系统在当前条件下的实施基础：所用技术栈均有成熟实现，开发与运行成本可控，终端用户无需专项培训即可上手使用。

需求分析明确了系统的服务对象与系统边界：终端用户与高级用户分别对应自然语言查询与直接 Cypher 查询两条入口，大语言模型服务与 Neo4j 图数据库作为外部依赖不包含在实现范围之内。功能需求梳理了六项核心功能，并说明了各功能之间“Schema 查询→语言生成→安全校验→执行→可视化”的依赖顺序，而非孤立的功能点集合。非功能需求从安全性、性能、可扩展性与易用性四个方面设定了约束条件，为后续系统设计提供了可量化的参照依据。

\section{系统设计}
\label{sec:system-design}

\subsection{设计原则与模块关联总览}

系统设计遵循三条原则。其一为职责分离：自然语言生成、语句校验、数据库访问、子图建构与浏览器展示在代码边界上相互独立，避免一处修改引发不可预期的连锁影响。其二为横切安全：所有进入 Neo4j 的语句受同一安全与参数策略约束，自然语言入口与直接 Cypher 入口不产生执行语义上的差异。其三为结果可核对：生成的查询语句与拒绝原因均可回显给调用方，便于用户核对系统理解是否与提问意图一致。

系统核心功能划分为五个模块，如图~\ref{fig:modules}所示。自然语言模块产生查询意图，安全模块位于执行之前，执行模块负责访问 Neo4j，查询结果子图建构模块处理查询记录，可视化模块消费建构后的结构化数据。五个模块不是并列的功能集合，而是形成一条有明确先后次序的处理链路。

系统存在两条独立的入口：自然语言查询入口与直接 Cypher 查询入口。自然语言入口依次经过 Schema 摘要获取、提示词组装与大语言模型调用，产生包含查询语句与参数的结构化输出；直接 Cypher 入口由用户手动提供语句与参数，不经过语言模型。两条入口在安全校验模块汇合：所有语句在进入数据库之前均须通过关键字黑名单匹配与参数一致性校验。校验通过后，执行模块在 Neo4j 只读会话中运行语句，取回原始记录；转换模块将原始记录整理为统一的节点—边结构与表格字段，构成 HTTP 响应体的数据载荷，由前端的可视化模块消费并渲染。这种“双入口—单出口”结构确保了两条路径在安全策略与数据格式上的一致性：用户无论通过哪条入口提交查询，看到的图谱与表格格式完全相同。各模块在主数据流中的具体实现见第~\ref{sec:detailed-implementation}~章。

\subsection{核心功能模块设计}

\subsubsection{自然语言查询模块}

自然语言查询模块负责将中文问句转化为可执行的 Cypher 语句，是面向终端用户的核心入口。该模块先从数据访问层获取当前图数据库的 Schema 摘要，再将其与用户问句按固定提示词格式组装，调用大语言模型获得包含查询语句与参数的结构化输出，最后将结果交由安全模块验证。主要交互顺序见图~\ref{fig:nlq-flow}。

提示词采用系统约束与少量示例相结合的策略，重点说明多标签 Schema 的用法、路径式 \texttt{RETURN path} 返回形式以及参数化占位符的要求，引导模型生成利于前端可视化的只读查询语句。模型生成的语句须经安全校验通过后方可执行；生成成功仅表示语言模型侧完成，不等于语句可直接执行。自然语言模块的输出——包含 Cypher 语句文本与参数字典的结构化对象——直接传入安全校验模块；若安全校验拒绝，则将拒绝原因回传给调用方，该次链路在安全模块处终止。

\subsubsection{Cypher 执行模块}

Cypher 执行模块服务于调试、教学演示和已知正确语句的验证场景，允许高级用户绕过大语言模型、直接提交只读 Cypher 语句。该模块与自然语言入口共用 Neo4j 客户端及同一套只读校验与参数校验逻辑，保证两条执行路径在安全策略上的一致性。用户通过此路径提交的语句同样先进入安全校验模块，通过关键字黑名单与参数一致性校验后，方可进入 Neo4j 会话执行；执行结果经相同的结果转换流程处理，前端以同一套图表字段渲染，不区分两条入口的来源。

\subsubsection{查询结果子图建构与可视化模块}

查询结果子图建构与可视化模块解决数据库返回记录与浏览器展示结构之间的格式差异。Neo4j 返回的记录可能包含节点、关系或路径对象，而 ECharts 关系图需要明确的 \texttt{nodes}/\texttt{links} 结构。系统将子图建构逻辑集中在后端，前端只消费约定好的 \texttt{graph}/\texttt{table} 字段，减少页面中的适配代码，也便于单独测试转换逻辑。图数据转换流程如图~\ref{fig:data-convert}所示。建构完成后，前端根据节点标签设置类别，以关系图展示拓扑结构，以表格展示属性与数值字段。该模块是整条处理链路的末端：其输出直接构成 HTTP 响应体，前端与 Neo4j 数据库之间不存在任何直接通信，所有数据访问均经由后端代理完成。这一设计使前端保持纯展示职责，后端可以在不涉及浏览器的情况下独立测试从数据库记录到节点—边结构的整个过程。

\subsubsection{安全校验模块}

安全校验模块位于两条执行入口之后、Neo4j 执行之前。该模块与数据库驱动的超时设置共同构成“语句级 + 会话级”的双层约束，安全校验流程如图~\ref{fig:security-flow}所示。

安全机制包括三类。第一类是关键字黑名单：对语句文本中的写操作关键字（\texttt{CREATE}、\texttt{MERGE}、\texttt{DELETE}、\texttt{SET} 等）进行正则匹配，命中则拒绝执行。第二类是可选执行计划预检：由环境变量控制，默认关闭；开启后在执行前请求数据库生成 \texttt{EXPLAIN} 计划以发现异常访问形态。第三类是参数一致性校验：语句中的 \texttt{\$param} 占位符须在请求体中有对应的同名参数，防止参数遗漏或不匹配。三类校验均通过后，安全模块将语句与参数原样移交执行层，在 Neo4j 只读会话中运行；任意一项校验失败，模块直接向调用方返回包含具体原因的拒绝说明，执行层不会被触及。

\subsubsection{面向其他知识图谱的迁移设计}
\label{subsec:migration}

本系统在设计上将图数据库的具体内容与核心处理逻辑分开：大语言模型通过提示词了解目标数据库的结构，后端通过环境变量连接目标数据库实例。因此，将系统从《迷你世界》知识图谱迁移到另一个 Neo4j 数据库，只需重写提示词并修改部署配置，不涉及任何代码改动。

\paragraph{系统提示词的改写}

系统提示词的作用是告知大语言模型目标数据库的结构，使模型生成与当前库一致的 Cypher 语句。迁移到新数据库时，需要按新库的 Schema 重新编写系统提示词。现有提示词（见附录~\ref{app:prompt-supplement}）可以作为模板，将迷你世界领域的内容替换为目标领域的内容即可。

新提示词需要描述三类信息。第一类是节点标签列表，并列出各标签的主要属性名称。第二类是关系类型列表，说明每种关系的起点标签与终点标签。第三类是一至两个示范性问句及其对应的 Cypher 语句，用于帮助模型理解目标领域的典型查询形态。示范性问句写得越接近实际使用场景，模型生成有效语句的成功率越高。

\paragraph{部署配置的修改}

连接到目标数据库，只需修改四个环境变量：\texttt{NEO4J\_URI} 指定数据库地址，\texttt{NEO4J\_USERNAME} 与 \texttt{NEO4J\_PASSWORD} 提供认证信息，\texttt{NEO4J\_DATABASE} 指定数据库名称。配置方式与初次部署相同，参照仓库中的 \texttt{.env.example} 文件填写对应值即可。

如果迁移时同步更换大语言模型服务商，还需要修改 \texttt{LLM\_API\_BASE}、\texttt{LLM\_API\_KEY} 与 \texttt{LLM\_MODEL} 三个变量。仅更换数据库而保留原有模型服务时，这三个变量不需要改动。

\subsection{数据库设计}
\label{subsec:db-design}

本系统使用 Neo4j 图数据库存储《迷你世界》知识图谱，采用多标签属性图模型。为避免术语歧义，这里先说明本文中的两个领域词：方块指游戏世界中可采集或可放置的地形/材料单元，生物指可交互实体（如动物、怪物或 NPC）。数据库设计充分利用 Neo4j 的多标签特性，将方块和生物作为“道具”这一上位概念的子类，通过复合标签表达继承关系。

数据库设计直接影响自然语言查询和可视化效果。Schema 中的标签与关系类型会进入自然语言模块的提示词，节点上的 \texttt{Name}、\texttt{MineTool} 等属性会影响模型生成 \texttt{WHERE} 条件，也会影响转换模块如何归类节点和生成表格列。因此，数据建模不是孤立的存储设计，而是与查询体验和展示效果联动的整体设计。

\subsubsection{架构演进}

早期曾考虑兼容阿里云 GDB 等不支持同一节点多标签的图引擎。在该约束下，无法直接使用 \texttt{:item:block}、\texttt{:item:monster} 这类复合标签表达“方块/生物是一种道具”的继承语义，只能将方块、生物与通用道具拆成不同节点，再通过业务 ID 用桥接关系（历史上曾建模为 PLACE、SUMMON 等）与道具节点相连。

该单标签桥接方案会拉长查询路径。自然语言转 Cypher 时，模型需要记住“先找道具，再经 PLACE 或 SUMMON 跳到方块/生物”的固定模式，生成错误的概率随之上升。同时，这种建模与用户对“方块和生物也是道具类别”的直观理解不一致，维护成本较高。

当前实现已取消 GDB 兼容路径，数据仅面向 Neo4j 导入与查询。方块与生物直接采用多标签节点建模，不再使用 PLACE、SUMMON 等桥接关系。提示词设计与测试用例均以当前多标签模型为准。

\subsubsection{多标签架构设计}

Neo4j 允许同一节点携带多个标签（Multiple Labels）。本文以 \texttt{:item} 作为道具基类标签（通用属性含 ID、Name、Type、Disc、GetWay）；方块类节点同时使用 \texttt{:item:block}，在基类属性之外增加 MineTool、ToolLevel 等；生物类节点同时使用 \texttt{:item:monster}，增加 Life、Attack 等。具体列示见表~\ref{tab:node-types}。相较早期单标签桥接方案，多标签使 ``方块/生物属于道具'' 直接体现在标签上，查询时可写 \texttt{MATCH (n:item:block) RETURN n.Name, n.MineTool} 等语句而无需经 PLACE、SUMMON 等关系绕行，语义与领域模型一致，并减少不必要的跳转与联结开销。

\subsubsection{Schema 摘要与自然语言模块的衔接}

数据库的节点标签与关系类型不仅是存储层面的设计，也是自然语言模块构建提示词的直接依据。系统通过 \texttt{GET /schema} 接口在运行时向调用方暴露当前图库的结构摘要：后端向 Neo4j 执行元数据查询，取得全部节点标签与关系类型，整理为列表格式后返回。下面是接口响应的示意结构：

\begin{lstlisting}[language=json, caption={\texttt{GET /schema} 响应示例}]
{
    "labels": ["item", "block", "monster", "group", "recipe", "smelt", "device"],
    "relationship_types": [
        "IN_GROUP", "CONSUMES", "PRODUCES", "DROPS",
        "TOOLMINEDROPS", "HANDMINEDROPS", "COVERS", "RUNS_ON"
    ]
}
\end{lstlisting}

自然语言模块在每次处理问句前拉取这份摘要，将其注入提示词，使大语言模型在生成 Cypher 时可以参考当前库的实际标签与关系名称。动态拉取而非硬编码的好处是：图库数据更新后，摘要内容在下一次请求时自动刷新；迁移到新数据库后，摘要同样自动反映新库的结构，无需另外修改代码。

多标签设计在这一环节体现出具体优势。Schema 摘要中的标签以原始名称出现，提示词说明书只需用一行示范语句解释 \texttt{:item:block} 表示方块，模型就能直接生成 \texttt{MATCH (n:item:block)} 形式的查询。若改用旧版桥接方案，标签只有通用的 \texttt{:item}，模型需要额外理解"方块通过 PLACE 关系挂载在道具下"的间接映射，生成正确语句的难度明显增加。

\subsection{本章小结}

本章确立了系统设计的三条原则：职责分离、横切安全与结果可核对；总体架构以分层方式划分浏览器页面、FastAPI 业务层、Neo4j 访问层与大语言模型外部服务。核心功能模块按请求进入、语句校验、数据库执行与结果展示的顺序衔接；数据库设计说明了多标签模型的演进过程、实体—关系图示、节点与关系类型及数据规模，并阐明 Schema 结构对自然语言查询与可视化效果的影响。迁移设计一节说明系统通过提示词与环境变量配置实现数据库解耦，切换到其他 Neo4j 知识图谱只需修改这两部分，代码层面无需改动。以上各节内容为第~\ref{sec:detailed-implementation}~章的详细实现提供设计依据。

\subsection{自然语言查询模块}
\label{subsec:nlq-impl}

\subsubsection{模块设计}

自然语言查询模块的核心设计目标是：在不要求终端用户了解 Cypher 语法的前提下，将中文问句准确映射为可对当前图库执行的只读参数化查询。

接口设计上，\texttt{POST /nlq} 接收包含问句字符串的 JSON 请求体，返回生成的 Cypher 语句、执行参数、图谱结构与表格数据四部分内容。将这四部分合并在单次响应中，使前端只需发起一次请求即可完成从问句到可视化数据的全部获取，减少网络往返次数。

两项设计决策对生成质量影响较大。其一，Schema 在每次请求时动态拉取而非硬编码：图数据库的标签与关系类型可能随数据导入版本变化，动态拉取确保提示词始终反映当前库状态，避免模型参考过时信息生成无法执行的语句。其二，要求模型输出参数化语句而非字符串拼接式 Cypher：参数化可防止特殊字符破坏语句结构，也使安全模块对参数占位符的一致性校验成为可能。

\subsubsection{代码实现}

自然语言查询模块负责将中文问句映射为参数化、只读、便于路径可视化的 Cypher。为使模型生成结果与当前图库一致，后端在调用大语言模型前先拉取 Schema 摘要，并将其与用户问句一并写入提示词。模型返回结果被约束为 JSON，主要字段为 \texttt{cql} 与 \texttt{params}，便于后续安全校验及在 Neo4j 侧按名称绑定参数取值。

对外入口为表~\ref{tab:api}中的 \texttt{POST /nlq}。实现上，FastAPI 根据 URL 路径将请求分发至对应处理函数（路由），编排``取 Schema、调用 \texttt{llm\_client.generate\_cypher}、进入安全模块、访问 Neo4j、调用转换器''的流水线。以下给出请求与响应的数据形状，与时序图~\ref{fig:nlq-flow}一致。

请求体格式：
\begin{lstlisting}[language=json, caption={自然语言查询请求体}]
{
    "query": "查找石剑",
    "options": {
        "limit": 100,
        "debug_raw": false
    }
}
\end{lstlisting}

响应体格式（字段名 \texttt{cql} 为历史命名，内容为Cypher字符串）：
\begin{lstlisting}[language=json, caption={自然语言查询响应体}]
{
    "cql": "MATCH (n:item) WHERE n.Name CONTAINS $name RETURN n",
    "params": {"name": "石剑"},
    "graph": {
        "nodes": [...],
        "links": [...],
        "categories": [...]
    },
    "table": {...}
}
\end{lstlisting}

提示词与生成策略可概括为四点：限定只读查询，注入当前库 Schema，说明多标签与路径式 \texttt{RETURN path} 的使用方式，并要求输出 JSON。仅返回孤立节点时，前端难以显示边信息，因此提示词倾向于引导模型返回路径。完整标签、关系类型与多组少样本示例见附录\ref{app:prompt-supplement}。正文仅保留一个代表性示例：
\begin{lstlisting}[language=cypher, caption={合成问句与路径式返回}]
-- 木料能合成什么
MATCH path = (material:item)-[:CONSUMES]-(r:recipe)-[:PRODUCES]-(product:item)
WHERE material.Name CONTAINS '木料'
RETURN path LIMIT 20
\end{lstlisting}

调用与解析的实现摘录如下。自然语言模块将系统提示词与用户问句拼装为 OpenAI 兼容的对话补全接口请求，并要求响应体为 JSON，以便提取 \texttt{cql} 与 \texttt{params}。

\begin{lstlisting}[language=Python, caption={LLM 生成 Cypher（llm\_client.py 摘录）}]
async def generate_cypher(self, nlq, schema_hint, limit):
    payload = {
        "model": self.model,
        "messages": [
            {"role": "system", "content": load_system_prompt()},
            {"role": "user", "content": f"只读 Cypher，LIMIT 默认 {limit}：\n{nlq}"},
        ],
        "temperature": 0.1,
        "response_format": {"type": "json_object"},
    }
    resp = await client.post(f"{self.api_base}/chat/completions", json=payload, headers=headers)
    obj = json.loads(resp.json()["choices"][0]["message"]["content"])
    return obj.get("cql", "").strip(), obj.get("params", {})
\end{lstlisting}

在调用 \texttt{generate\_cypher} 之前，路由层先通过 Neo4j 客户端查询当前图库的标签与关系类型，将结果拼成纯文本摘要，作为 \texttt{schema\_hint} 参数传入。大语言模型在接收到问句时，已知当前库中存在哪些标签与关系，可以据此生成与实际图库结构一致的 Cypher 语句，而不是依赖静态提示词中可能已过时的描述。

\subsection{图数据访问与 Cypher 执行模块}

\subsubsection{模块设计}

图数据访问模块的设计目标是为两条执行路径（自然语言入口与直接 Cypher 入口）提供统一的数据库访问接口，使上层路由层不需要直接操作驱动细节。

接口设计上，\texttt{run\_read(cql, params)} 是模块对外暴露的唯一读取方法，接受语句字符串与参数字典，返回记录列表与列名序列。返回列名序列的用意是便于转换模块在构建表格字段时保持与数据库原始列的对应关系，而无需在转换时重新解析记录键名。

会话设计上，模块显式指定 \texttt{default\_access\_mode="READ"} 打开只读会话，在驱动层面阻止写操作，与应用层的关键字黑名单校验形成双重约束。超时时间以毫秒为单位通过配置项传入，模块负责换算为秒后注入驱动，调用方不需要了解驱动的时间单位约定。同时，会话在 \texttt{with} 块结束时由驱动自动归还到连接池，避免长时间持有连接导致池资源耗尽。

\subsubsection{代码实现}

任何进入 Neo4j 的查询都由同一个客户端封装统一处理。每次会话以只读模式打开，确保驱动层面不允许写操作。超时时间由配置项以毫秒给出，换算为秒后传入驱动，防止单次查询长时间占用连接。路由层先经过 \texttt{cql\_validator} 的安全校验，通过后才调用 \texttt{run\_read}，将记录列表与列名序列一并返回给转换模块。下列代码展示只读会话的建立与超时换算方式。

\begin{lstlisting}[language=Python, caption={Neo4j 只读查询（neo4j\_client.py 摘录）}]
def run_read(self, cql: str, params: Dict[str, Any] | None = None):
    params = params or {}
    with self._driver.session(
        database=settings.NEO4J_DATABASE,
        default_access_mode="READ",
    ) as session:
        timeout_seconds = settings.QUERY_TIMEOUT_MS / 1000.0
        result = session.run(cql, parameters=params, timeout=timeout_seconds)
        records = [r.data() for r in result]
        return records, list(result.keys())
\end{lstlisting}

\subsection{查询结果子图建构与可视化模块}

\subsubsection{模块设计}

查询结果子图建构模块的设计目标是将 Neo4j 驱动返回的不同类型记录统一转换为节点—边结构。经过这一处理，前端可以直接使用 ECharts 所需的数据格式，不需要再编写额外的格式适配代码。

输出数据结构设计为三部分：\texttt{nodes}（节点列表，含 ID、标签类别与属性字典）、\texttt{links}（关系列表，含起点 ID、终点 ID、关系类型与属性字典），以及 \texttt{categories}（节点类别列表，由节点标签集合导出，供 ECharts 按标签设置颜色分组）。表格字段 \texttt{table} 与子图结构并列放在响应体顶层，供用户查阅节点的长文本属性与数值字段。

节点去重是子图建构设计中的关键细节。当查询返回路径时，同一节点可能在不同路径段中多次出现；若不去重，前端图谱中会出现相同实体的多个副本，导致关系图边的连接混乱。转换模块按 Neo4j 元素 ID（\texttt{element\_id}）对节点做集合去重，保证每个实体在图谱中只出现一次。子图建构逻辑集中在后端，便于在不依赖浏览器的情况下对转换过程单独进行单元测试。

\subsubsection{代码实现}

Neo4j 驱动返回的记录中可能混有节点、关系和路径对象，而前端关系图需要明确区分节点集合与边集合。为此，后端设置专门的转换模块，将出库记录整理为 \texttt{graph} 与 \texttt{table} 两部分，以便前端分别绘制关系图和展示属性表格。转换的主要工作是遍历记录、提取节点与关系、按元素 ID 去重，并处理仅有节点而没有边的情形。核心逻辑在 \texttt{echarts\_converter.py} 中实现，下列代码展示处理一条关系时如何确保两端节点先被写入。

\begin{lstlisting}[language=Python, caption={图数据转换（echarts\_converter.py 摘录）}]
def add_rel(rel: graph.Relationship) -> None:
    src, tgt = _node_id(rel.start_node), _node_id(rel.end_node)
    add_node(rel.start_node)
    add_node(rel.end_node)
    links.append({
        "source": src, "target": tgt,
        "category": rel.type, "label": rel.type,
        "value": dict(rel),
    })
\end{lstlisting}

\subsection{安全与约束模块}

\subsubsection{模块设计}

安全校验模块的设计目标是在 Neo4j 数据库账号权限控制之外，在应用层建立独立的语句级防线。这一防护层的必要性在于：数据库账号的只读配置可能因运维失误被覆盖，而大语言模型在极端情况下可能生成语义歧义的语句，应用层的主动拦截可以消除对账号配置的单点依赖。

校验策略采用黑名单而非白名单。白名单策略（只允许特定 Cypher 模式通过）虽然更为严格，但对语句形态的约束过于收紧，会妨碍自然语言模块生成多样化只读查询的灵活性。黑名单策略明确列举禁止的写操作关键字（\texttt{CREATE}、\texttt{MERGE}、\texttt{DELETE} 等）与管理命令（\texttt{CALL dbms.*}），在不限制合法只读语句形态的前提下拦截常见威胁。

校验机制共三层。第一层是关键字黑名单，覆盖最常见的写操作与高危管理命令。第二层是参数一致性校验，提取语句中的 \texttt{\$param} 占位符，与请求体参数键逐一比对，防止参数遗漏或注入。第三层是可选的 \texttt{EXPLAIN} 预检，由配置项控制，默认关闭，仅在需要进一步验证语句访问形态时开启，避免对数据库产生额外负载。

\subsubsection{代码实现}

任何进入 \texttt{/nlq} 或 \texttt{/run-cql} 的语句，在交给 Neo4j 执行之前都要先经过安全校验模块。校验器对语句文本做正则匹配，拦截 \texttt{CREATE}、\texttt{MERGE}、\texttt{DELETE}、\texttt{SET} 等写操作关键字，以及形如 \texttt{CALL dbms.*} 的管理命令，命中时直接返回错误，不进入执行环节。即便 Neo4j 账号本身已配置为只读，应用层的这一检查仍有必要，因为提交的语句中可能出现预期之外的高危形态。可选的 \texttt{EXPLAIN} 预检由配置项控制，默认关闭，以避免增加数据库的额外开销。校验逻辑集中在 \texttt{cql\_validator.py}，下列黑名单与表~\ref{tab:readonly-test} 中测试用例对应：

\begin{lstlisting}[language=Python, caption={只读校验黑名单（与 cql\_validator.py 一致）}]
WRITE_BLACKLIST = [
    r"\bCREATE\b",
    r"\bMERGE\b",
    r"\bDELETE\b",
    r"\bDETACH\s+DELETE\b",
    r"\bSET\b",
    r"\bREMOVE\b",
    r"\bDROP\s+(?:INDEX|CONSTRAINT|NODE\s+LABEL|REL\s+TYPE|PROPERTY)\b",
    r"\bLOAD\s+CSV\b",
    r"\bCALL\s+DBMS\.",
]
\end{lstlisting}

系统目前未对 \texttt{CALL} 所调用的自定义存储过程建立允许列表，因此部署时需要满足一定的环境前提：图库以只读账号或只读副本连接，服务置于内网，或经反向代理并配合鉴权后再对外暴露。满足这些前提时，能够执行 Cypher 的请求来源已限定在可信范围内。若需在不可信网络中直接暴露接口，应在网关层添加额外的访问控制，或扩展校验逻辑的限制规则。

参数一致性校验通过正则表达式提取 Cypher 语句中所有的 \texttt{\$param} 占位符，再与请求体中的参数键逐一比对，确保二者完全匹配。\texttt{/run-cql} 与 \texttt{/nlq} 共用这一逻辑，路由层的实现位于 \texttt{backend/app/} 下，具体路径约定见第~\ref{subsec:repo-and-paths}~节。

以下代码展示参数一致性校验的核心实现。函数先提取语句中的全部占位符名称，再与请求体参数键做集合比较；若两个集合不相等，立即抛出异常，错误信息列出缺少和多余的键名，便于调试。

\begin{lstlisting}[language=Python, caption={参数一致性校验（cql\_validator.py 摘录）}]
def validate_params(cql: str, params: dict) -> None:
    placeholders = set(re.findall(r'\$(?P<name>\w+)', cql))
    provided = set(params.keys())
    if placeholders != provided:
        missing = placeholders - provided
        extra = provided - placeholders
        raise CqlValidationError(
            f"参数不匹配：缺少 {missing}，多余 {extra}"
        )
\end{lstlisting}

\begin{thesisconclusion}

本文针对属性图数据库使用门槛高、查询结果难以直接呈现拓扑结构的问题，设计并实现了一套基于生成式AI的知识子图建构与可视化系统。系统允许用户在浏览器中以中文提问，自动生成只读的 Cypher 语句并在 Neo4j 中执行，将返回记录建构为关系图与表格两种前端数据形态同步呈现，无需用户学习图数据库查询语言。系统全流程不包含对 Neo4j 的写入操作。

系统以《迷你世界》游戏知识图谱为应用场景，从两个层面完成了验证。自动化测试共 38 个用例，覆盖接口约定、只读校验与子图建构转换，全部通过。对 21 条真实中文问句的逐条评判显示，17 条（81\%）得到可接受的结果。两类测试共同说明，系统在当前数据与配置下能够可靠运行，并能稳定完成查询结果子图建构与前端可视化回显。

\subsection*{主要工作}

（1）系统架构。系统采用前后端分离的浏览器/服务器结构，后端基于 FastAPI 提供自然语言查询、直接 Cypher 查询与图数据库结构摘要三类接口，前端使用原生 HTML/JavaScript 实现单页面交互，以 ECharts 绘制可交互的关系图，并对查询结果完成子图回显。

（2）自然语言到 Cypher。后端在调用大语言模型时注入当前图数据库的结构摘要，要求模型返回只读的参数化 Cypher 语句，使中文问句能够转化为可在 Neo4j 中直接执行的图查询。

（3）安全与执行。所有进入数据库的语句在执行前须经过写操作关键字过滤、参数占位符完整性检查，以及查询超时与返回行数上限等约束，自然语言入口与直接 Cypher 入口遵循相同的执行规则。

（4）数据与领域模型。系统采用 Neo4j 多标签属性图组织《迷你世界》的道具、配方与生物数据；节点、关系类型与数据规模见第~\ref{sec:system-design}~章第~\ref{subsec:db-design}~节，用于验证自然语言查询、受控执行、查询结果子图建构与可视化展示的完整流程。

\subsection*{不足与展望}

当前系统仍有以下不足：
\begin{enumerate}[(1)]
    \item 复杂查询、分组查询和多实体对比仍有失败情形，说明系统在理解复杂语义与精确匹配属性键方面仍有不足。
    \item 同义词和近义表达处理不足，用户使用不同名称指代同一概念时，系统仍可能无法稳定匹配。
    \item 权限控制仅依赖应用层只读校验与部署环境假设，尚未建立完整的用户身份认证与访问控制机制。
\end{enumerate}

未来工作方向：
\begin{enumerate}[(1)]
    \item 引入 RAG（检索增强生成）或实体链接技术，提高实体、属性与关系类型匹配的准确性。
    \item 构建领域同义词与别名词表，使语义等价的表述能够映射到同一图数据库对象。
    \item 完善身份认证、访问控制和审计日志，使系统更适合在非演示环境中部署。
    \item 研究多源图数据的适配方法，使同一自然语言查询流程能够迁移到结构相近的其他领域知识图谱。
\end{enumerate}

\end{thesisconclusion}
