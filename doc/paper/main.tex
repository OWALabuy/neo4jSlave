\begin{abstract}

知识图谱以图结构表示实体及实体间的关系，可展示数据中隐含的知识联系，在知识管理、智能问答与推荐等领域应用较广。在属性图数据库中，查询实体之间的关系通常需要使用专门的图查询语言，使用者需掌握相应的语法规则与数据库结构，且查询结果以表格方式呈现，难以直观反映图结构信息。这为非数据库专业的普通用户设置了两类门槛：一是需专门学习查询语言，二是难以从表格结果中直观理解实体间的关联。

本文设计并实现了一套知识子图建构与可视化系统，命名为 neo4jSlave，采用浏览器/服务器（B/S）架构。用户在浏览器中以中文提出问题，后端首先读取 Neo4j 图数据库的节点标签与关系类型等结构信息。系统将这些结构信息与问句一并提交给大语言模型，由模型生成相应的 Cypher 查询语句。查询语句经应用层安全校验后在数据库中执行，返回的记录在后端组装为节点—边结构，即本文所称的知识子图。前端使用 ECharts 将子图渲染为可交互的关系图，并配以属性表格同步展示。

验证分为自动化测试与自然语言实例测试两部分。自动化测试覆盖接口行为、安全约束以及查询结果到图结构的转换逻辑，38 条用例均已通过。自然语言实例测试以《迷你世界》游戏知识图谱为测试对象，对 21 条中文问句逐条评判，其中 17 条得到了可接受的查询结果，通过率约为 81\%。

本系统使没有数据库专业背景的用户也能直接以中文查询知识图谱，并在浏览器中直观浏览图形化的结果，可作为领域知识库的轻量级查询入口，也适用于课堂教学演示等场景。

\end{abstract}

\addcontentsline{toc}{section}{Abstract}
\begin{thesisabstracten}

Knowledge graphs represent entities and their relationships in a graph structure, and are widely applied in knowledge management, question answering, and recommendation systems. Querying a property graph database typically requires a dedicated graph query language, and query results are returned as flat tables that obscure the topology of the graph. These two barriers---learning the query language and interpreting tabular results---make knowledge graphs difficult to use for non-specialist users.

This thesis presents neo4jSlave, a browser-based system for knowledge subgraph construction and visualisation using generative AI. When a user submits a question in Chinese, the backend reads the node labels and relationship types from the Neo4j graph database and submits them together with the question to a large language model, which generates a corresponding Cypher query. The query is validated by an application-layer safety check before execution. The returned records are assembled into a node--edge structure on the backend---referred to in this thesis as the knowledge subgraph---and rendered in the browser as an interactive graph using ECharts, alongside a property table.

Evaluation consists of automated tests and a natural-language walkthrough. The automated suite covers API behaviour, safety constraints, and subgraph conversion logic; all 38 test cases passed. The walkthrough tested 21 Chinese questions against a Mini World game knowledge graph; 17 questions (approximately 81\%) received acceptable results. The system enables users without database expertise to query a knowledge graph directly in Chinese and browse the results visually, making it suitable as a lightweight query interface for domain knowledge bases and for educational demonstrations.
\end{thesisabstracten}

\section{系统概述}

\subsection{研究背景与意义}

知识图谱以有向图描述实体及实体间的关系，节点代表实体，边代表关系，形成能够揭示数据内在联系的知识表示形式。这种组织方式在知识管理、智能问答与个性化推荐等任务中应用较广，已成为信息系统领域的重要基础设施\cite{ref17}。在工程实现层面，知识图谱的落地路线主要分为两类：以三元组为基础的 RDF 模型，以及以节点关系为基础的属性图模型。RDF 模型适合开放数据互联与本体推理；属性图在节点和边上直接附加键值属性，贴近业务建模，便于迭代维护\cite{ref1,ref3}。本文采用属性图路线，以 Neo4j 图数据库存储和查询知识图谱，对应的图查询语言为 Cypher。

在属性图数据库中，查询实体及其关系通常需要书写 Cypher 语句。这要求使用者预先掌握节点标签、关系类型与方向、模式匹配语法以及属性条件的写法，还需要能够理解以表格形式返回的查询记录。对于偶尔需要查询的非专业用户，这些学习成本较高；而仅有表格结果也难以直观呈现“谁与谁相连”这类拓扑信息。

近年来，大语言模型（Large Language Model，LLM）在将自然语言转为结构化查询方面取得了显著进展\cite{ref4,ref11}，使通过中文提问自动生成 Cypher 语句成为可行的工程路线。将自然语言输入与图形化结果展示相结合，可以让非专业用户无需专门学习 Cypher 即可完成知识图谱的基本查询与浏览。

本文以《迷你世界》游戏知识图谱为测试场景，设计并实现了一套基于生成式 AI 的知识子图建构与可视化系统。该系统为领域知识库提供了轻量的浏览入口，适用于教学演示与日常业务查询等场景。本文还将自然语言生成、安全约束、子图建构与图形化展示整合为一条完整的处理流程，为同类系统提供可参考的实现方案。

\subsection{国内外研究现状}

\subsubsection{知识图谱的存储与查询}

知识图谱的数据组织与查询主要有两条技术路线。第一条是 RDF 与 SPARQL：该路线将知识表示为主谓宾三元组，便于开放发布与跨数据集互联，并可结合本体进行逻辑推理\cite{ref17}。第二条是属性图：以节点和有向关系表示实体结构，在节点或关系上直接附加属性，查询语言围绕模式匹配与路径展开。已有研究从图数据管理、存储引擎与查询优化角度，探讨了图数据库在索引、事务与可扩展性上的关键问题\cite{ref1,ref2,ref3}。

在工程选型上，当数据已以属性图形式组织、目标是以较少组件交付可运行 Web 查询服务时，Neo4j 在文档、驱动生态与社区资源上均较为成熟\cite{ref12}。然而，仅完成“数据可读与语句可执行”并不足以解决终端用户的使用问题——查询入口仍是手动编写 Cypher 时，用户仍需掌握当前图数据库的结构与语法规则。这一缺口是引入自然语言接口的直接动因。

\subsubsection{自然语言到结构化查询}

将自然语言转换为可执行的结构化查询语句是降低数据库使用门槛的核心方法。文本到 SQL（Text-to-SQL）是该方向中研究积累最为丰富的任务；综述工作表明，影响生成质量的两类关键因素是模型对当前数据库结构的理解程度，以及生成语句能否在数据库中实际执行\cite{ref5}。这一结论对自然语言到 Cypher 任务同样具有参考价值。

在属性图场景下，自然语言到 Cypher（Text2Cypher）任务已有专项研究。结果表明，当不针对该任务专门训练模型时，仅凭通用模型直接生成 Cypher 在处理语义复杂或条件较多的问句时仍不稳定；对模型进行专项训练可以改善评测指标\cite{ref16}。因此，工程系统除依赖模型本身外，还需在提示中向模型提供当前图数据库的结构信息，并对生成结果进行执行前校验，以提升实际可用性。

\subsubsection{大语言模型与知识图谱的协同}

大语言模型与知识图谱的协同研究涵盖多个方向，包括以图结构为模型补充外部知识、辅助推理，以及提升模型返回结果的可解释性\cite{ref11}。在自然语言查询场景中，两者的分工相对清晰：大语言模型负责将问句转化为候选查询语句，图数据库负责确定性执行并返回结构化结果，应用层负责执行前后的安全约束与结果整理\cite{ref4,ref9,ref11}。

这种结构兼顾了灵活性与可控性，但也面临若干风险：模型生成的语句可能与实际图结构不符，同一问句在不同调用下可能输出不一致的结果，以及执行权限被误用等。针对这些风险，需要通过规则约束与执行前后的验证机制加以抑制。本文采用应用层约束结合执行前校验的工程路径，在通用大语言模型与已有属性图数据的条件下，构建可运行的中文问句查询服务。

\subsection{主要工作与论文组织}

本文的主要工作围绕四个方面展开。第一，设计并实现了基于 B/S 架构的知识子图建构与可视化系统。后端基于 FastAPI 提供三类接口：自然语言查询、直接 Cypher 查询与图数据库结构摘要。前端以原生 HTML/JavaScript 实现单页面交互，以 ECharts 绘制可交互的关系图。第二，在自然语言查询路径上，系统先获取当前图数据库的结构信息。随后将结构信息与用户问句一同注入提示词，送入大语言模型，得到只读的参数化 Cypher 语句。第三，所有进入数据库的语句须经应用层统一校验，包括写操作关键字过滤与参数占位符一致性检查。校验通过后，执行结果统一转换为前端可直接渲染的节点—边结构。

第四，以《迷你世界》知识图谱为领域数据，通过自动化测试与自然语言实例测试，验证系统的功能正确性、子图建构效果与可视化可用性。

本文其余章节组织如下：第 2 章介绍系统的开发环境与关键技术选型；第 3 章对系统进行可行性与需求分析；第 4 章给出系统的整体架构与各功能模块的设计；第 5 章描述各模块的实现细节；第 6 章给出系统测试结果与分析。

\section{系统开发环境与技术选型}

\subsection{开发与运行环境}
\label{subsec:repo-and-paths}

本系统在 GNU/Linux 环境下开发与调试，典型发行版为 Ubuntu 22.04 LTS 和 Debian 12。宿主机需部署可访问的 Neo4j 实例，既可连接本机服务（如 \texttt{bolt://127.0.0.1:7687}），也可连接局域网或云主机上的远程实例。后端使用 Python 3.10 及以上版本，通过虚拟环境管理第三方依赖，主要依赖包括 FastAPI、neo4j 驱动、httpx 与 pytest。前端采用原生 HTML/CSS/JavaScript，浏览器直接加载已编译的 ECharts 库文件，无需 Node.js 构建步骤。版本控制使用 Git，源代码托管于 GitHub：\url{https://github.com/OWALabuy/neo4jSlave}。

系统对运行环境的主要约束来自后端和图数据库两端。后端要求 Python 3.10 以上版本，部分类型注解写法依赖该版本引入的联合类型语法；建议通过 \texttt{python -m venv} 创建独立虚拟环境，避免依赖版本与系统已有软件冲突。Neo4j 官方 Python 驱动通过 Bolt 协议与数据库服务通信，驱动版本应与数据库服务器的主版本号对应；本系统以 Neo4j 5.x 为基准开发与测试。

\subsection{关键技术选型}

本系统的技术路线围绕三项目标构建：降低查询入口门槛、增强结果可理解性、控制生成语句的执行风险。

系统后端采用 FastAPI 框架。每次自然语言查询需要依次等待大语言模型返回结果（通常 1 至 3 秒）和图数据库完成查询（数十至数百毫秒），是典型的网络等待密集型场景。FastAPI 基于 Python 协程，在等待网络响应期间可以切换处理其他请求，避免某次模型调用的延迟影响其他用户。框架内置的请求参数校验功能可以在请求进入业务逻辑之前检查数据格式，减少后续处理中的重复判断\cite{ref6,ref10}。

图数据库选用 Neo4j，其属性图模型和 Cypher 查询语言的社区文档与驱动库均较为完善\cite{ref7,ref12}。本课题的《迷你世界》领域数据已有现成的 Neo4j 数据文件，无需额外迁移。此外，Cypher 以图形化符号表达节点与关系，例如 \texttt{(a)-[:KNOWS]->(b)} 直接表示“a 认识 b”，语法结构与自然语言的对应较为直观；与其他图查询语言相比，Cypher 在大语言模型训练数据中的覆盖程度更高，有助于提高自然语言转查询的成功率\cite{ref16}。

大语言模型通过 OpenAI 兼容的 HTTPS 接口接入，端点与模型名由环境变量配置，便于在不修改业务代码的情况下切换不同服务商\cite{ref4,ref9}。实例测试时使用阿里云通义百炼平台的 qwen-turbo 模型。

选择以远程 API 接入模型，是因为在本地部署具备足够推理能力的大语言模型通常需要专用的 GPU，在本课题的小规模验证场景下成本偏高。商业 API 服务按实际使用量计费，响应延迟稳定，也无需在本机维护模型运行环境。

ECharts 是 Apache 基金会的开源可视化库，关系图组件支持力导向布局与拖拽缩放交互，适合在 Web 单页面中嵌入使用\cite{ref8}。各组件与候选方案的对比概括见表~\ref{tab:tech-compare}。

在知识图谱查询场景中，用户最关心的是实体之间“谁与谁相连”的拓扑关系，而非坐标轴上的数值。力导向布局算法根据节点间的连接数量动态调整位置，连接紧密的节点自然聚拢，连接稀疏的节点自动分散，使图谱的结构特征无需人工干预即可直观呈现。D3.js 同样支持力导向图，但使用门槛较高，需要编写较多底层绑定代码；Cytoscape 更侧重生物信息学场景，预置交互模式与本系统需求不完全吻合。ECharts 关系图组件开箱即用，可直接在 HTML 页面中通过 JavaScript 初始化，与本系统“前端无需构建步骤”的部署目标一致。

\subsection{系统总体架构与数据流}

系统采用分层架构：浏览器中的页面（表示层）、FastAPI 应用（业务逻辑层）、Neo4j 官方 Python 驱动（数据访问层），以及外部的大语言模型 HTTPS 服务，如图~\ref{fig:architecture}所示。

\begin{figure}[H]
    \centering
    \includegraphics[width=0.95\textwidth]{img/architecture.png}
    \caption{系统总体架构图}
    \label{fig:architecture}
\end{figure}

自然语言查询的主要数据流如下：用户在浏览器中提交中文问句，请求发至 FastAPI；后端先获取当前图数据库的节点标签与关系类型，将其与问句一同写入提示词，发送至大语言模型；模型返回包含 Cypher 语句与参数的结构化输出；后端对语句进行只读校验与参数一致性检查，通过后在 Neo4j 只读会话中执行；查询结果经转换模块建构为节点—边结构与表格字段，返回给前端；浏览器使用 ECharts 绘制关系图，并以表格展示属性数据。

熟悉 Cypher 的用户也可以跳过自然语言转译步骤，直接向 \texttt{/run-cql} 接口提交查询语句。该接口与自然语言查询接口共用同一套安全校验与结果转换逻辑，两者的执行行为保持一致。

\subsection{本章小结}

本章从开发环境、技术选型与系统架构三个方面完成了系统的总体规划。系统以 GNU/Linux 为运行基础，后端依赖通过 Python 虚拟环境管理，数据库地址、模型密钥等外部配置均通过环境变量注入，部署时无需修改代码。

各项技术的选型均服务于同一目标：让没有数据库背景的用户能够直接以中文查询知识图谱。FastAPI 适合系统中需要等待外部模型响应的异步场景；Neo4j 的属性图结构贴近领域数据的建模方式，且 Cypher 便于大语言模型生成；模型以远程 API 方式接入，无需在本地部署；ECharts 在浏览器中渲染可交互的关系图，满足图谱的图形化展示需求。

系统整体采用浏览器、FastAPI 应用、Neo4j 数据库三层结构，大语言模型作为外部服务由后端按需调用。各层之间通过 HTTP 接口交互，职责边界清晰，为后续各章的详细设计奠定了基础。

\section{系统分析}

\subsection{可行性分析}

从技术可行性看，系统所用的各项技术均有成熟的开源实现和完善的社区文档。FastAPI、Neo4j、ECharts 均已在实际项目中广泛使用，官方驱动与示例也较为完整。大语言模型通过标准 HTTPS 接口调用，不需要定制集成。整套系统可以在一台普通 Linux 主机上部署和运行，不依赖专用硬件。

从经济可行性看，系统的开发成本主要来自人力投入，不需要购买专用硬件或商业软件授权。Neo4j 社区版可免费使用，后端所用的框架和库也均为开源软件。运行期的主要费用来自大语言模型的调用，按实际请求量计费，在日常演示频率下费用极低。

从运行可行性看，用户只需通过浏览器访问系统，无需安装任何客户端软件。服务端可以部署在普通笔记本上，不需要专用服务器或 GPU。中文自然语言查询界面使没有数据库背景的用户也能直接使用，不需要学习 Cypher 语法。

\subsection{需求分析}

\subsubsection{系统边界与用例}

系统直接服务两类用户。第一类是终端用户，通过浏览器输入中文问句，查看知识图谱的图形化结果，通常不具备 Cypher 编写能力。第二类是高级用户，可直接提交 Cypher 语句，用于调试、对照自然语言生成结果或进行教学演示。两类用户均通过同一 Web 前端访问后端接口。系统的外部依赖包括大语言模型服务与 Neo4j 图数据库，两者均作为独立服务存在，不包含在本文实现的系统边界之内。本文系统不包含任何图数据库写入能力，所有数据库访问均限定为只读查询。系统用例如图~\ref{fig:use-case-placeholder}所示。

两类用户的需求决定了系统需要同时支持两条查询路径。终端用户路径通过大语言模型将中文问句转为 Cypher，生成过程在后台完成，用户只看到最终的关系图和表格结果。熟悉 Cypher 的用户则可以跳过这一步骤，直接提交查询语句，主要用于调试或教学演示。两条路径在安全校验、数据库访问和结果转换等关键步骤上共用同一套逻辑，执行行为保持一致。

\subsubsection{功能需求}

系统共包含六项功能需求，分为查询入口、结果展示和安全保障三类。自然语言查询是主要入口，用户提交中文问句后，系统先读取图数据库的结构信息，再调用大语言模型生成 Cypher 语句，经安全校验后执行，结果以知识子图和属性表格的形式返回。直接 Cypher 查询通过 \texttt{/run-cql} 接口提供，主要供调试和教学演示使用，与自然语言入口使用同一套校验和执行逻辑。

结果展示包括关系图和属性表格两部分，前者用于直观呈现实体之间的连接关系，后者用于展示节点的具体属性值。系统还提供 \texttt{/schema} 接口，返回当前图数据库的节点标签和关系类型，既作为生成 Cypher 时的背景输入，也可供用户直接查阅图库的整体结构。安全校验对两条查询路径均生效，在语句进入数据库前检查是否包含写操作，并与查询超时和返回行数上限一起控制执行风险。各功能对应的用户故事见表~\ref{tab:user-stories}。

六项功能之间存在明确的先后依赖关系。读取图数据库结构是自然语言查询的前提，系统在每次调用大语言模型前都会先获取当前的节点标签和关系类型。安全校验位于执行步骤之前，两条查询路径产生的语句都须经过同一套检查。可视化以查询结果为输入，只有数据库返回数据后，前端才会渲染关系图和表格。

\subsubsection{非功能需求}

系统在安全性上设有两道独立的防线。数据库账号在部署时设置为只读权限，但仅凭账号限制还不够——模型生成的语句或用户直接提交的语句中仍可能包含写操作或危险的管理命令。因此，系统在每条语句进入数据库前还会检查关键字，拦截写类操作。两道防线独立生效，即使其中一道出现配置问题，另一道仍能提供保护。

每次自然语言查询需要等待大语言模型返回结果，通常需要 1 至 3 秒，这是网络传输和模型计算所需的时间。系统对其他环节的要求是不造成额外的明显延迟：数据库查询通常在数十至数百毫秒内完成，前端渲染随后进行。为防止效率低下的 Cypher 语句导致长时间等待，系统对每次查询设有超时时间和返回行数上限。

系统在设计上考虑了后续调整的便利性。连接参数和密钥通过环境变量注入，切换大语言模型服务商时只需修改配置，不需要改动代码。后端代码在结构上区分了路由、数据库访问和结果转换三个部分，各部分职责独立，修改其中一部分不会波及其他部分。如果要将系统接入另一个知识图谱，只需更新数据库连接参数和对应的提示词，前端页面和后端逻辑均无需改动，具体步骤见第~\ref{sec:system-design}~章。

易用性方面，系统的主要考虑是让用户能够判断查询是否按预期执行，出现问题时也知道如何调整。前端在返回结果的同时展示大语言模型生成的 Cypher 语句，用户可以对照自己的提问和实际执行的语句，确认系统的理解是否准确。当请求不合法时，系统返回说明具体原因的错误信息，例如语句中包含写操作，而不是笼统地报告请求失败，便于用户排查问题。

\subsection{本章小结}

本章从可行性与需求两个方面为系统的设计提供了依据。可行性分析表明，系统所用的技术均有成熟实现，开发与运行成本可控，普通用户无需专门学习数据库知识即可使用。

需求分析明确了系统服务两类用户：不熟悉 Cypher 的终端用户通过中文问句查询，熟悉 Cypher 的用户可以直接提交语句。功能需求梳理了六项核心功能及其先后依赖关系，非功能需求从安全性、性能、可扩展性与易用性四个方面设定了约束条件，为后续设计提供了依据。

\section{系统设计}
\label{sec:system-design}

\subsection{设计原则与模块关联总览}

系统设计遵循三条原则。第一，各模块的职责相互独立：自然语言生成、语句校验、数据库访问、子图建构与浏览器展示之间有清晰的边界，修改其中一部分不会影响其他部分。第二，安全校验对两条查询路径统一生效：无论语句来自大语言模型还是用户直接输入，进入数据库前都经过同一套检查。第三，生成结果对用户透明：系统将生成的 Cypher 语句和拒绝原因一并返回，便于用户确认查询是否符合自己的预期。

系统核心功能划分为五个模块，如图~\ref{fig:modules}所示。自然语言模块负责将问句转为查询语句，安全模块在执行前完成校验，执行模块访问数据库取回记录，子图建构模块将记录整理为节点—边结构，可视化模块在前端渲染关系图和表格。五个模块按照固定的先后顺序工作，而不是各自独立运作。

系统存在两条独立的入口：自然语言查询入口与直接 Cypher 查询入口。自然语言入口依次经过 Schema 摘要获取、提示词组装与大语言模型调用，产生包含查询语句与参数的结构化输出。直接 Cypher 入口由用户手动提供语句与参数，不经过语言模型。两条入口在安全校验模块汇合，所有语句在进入数据库之前均须通过关键字黑名单匹配与参数一致性校验。校验通过后，执行模块在 Neo4j 只读会话中运行语句，取回原始记录。转换模块将原始记录整理为统一的节点—边结构与表格字段，由前端的可视化模块消费并渲染。

这种“双入口—单出口”结构确保了两条路径在安全策略与数据格式上的一致性。用户无论通过哪条入口提交查询，看到的图谱与表格格式完全相同。各模块在主数据流中的具体实现见第~\ref{sec:detailed-implementation}~章。

\subsection{核心功能模块设计}

\subsubsection{自然语言查询模块}

自然语言查询模块的职责是将用户的中文问句转化为可执行的 Cypher 语句，是面向终端用户的主要入口。模块首先读取当前图数据库的节点标签和关系类型，将这些结构信息与问句一并填入提示词，发送给大语言模型。模型返回包含 Cypher 语句和查询参数的结果，模块随即将其传给安全校验模块处理。主要交互顺序见图~\ref{fig:nlq-flow}。

提示词采用系统约束与少量示例相结合的策略，重点说明多标签 Schema 的用法、路径式 \texttt{RETURN path} 返回形式以及参数化占位符的要求，引导模型生成利于前端可视化的只读查询语句。模型生成的语句须经安全校验通过后方可执行；生成成功仅表示语言模型侧完成，不等于语句可直接执行。自然语言模块的输出——包含 Cypher 语句文本与参数字典的结构化对象——直接传入安全校验模块；若安全校验拒绝，则将拒绝原因回传给调用方，该次链路在安全模块处终止。

\subsubsection{Cypher 执行模块}

Cypher 执行模块允许熟悉 Cypher 的用户跳过大语言模型，直接提交查询语句，主要用于调试和教学演示。用户提交的语句先经过安全校验，通过后才在 Neo4j 中执行；返回的结果与自然语言路径经过同样的转换处理，前端以相同的格式展示，不区分来源。

\subsubsection{查询结果子图建构与可视化模块}

查询结果子图建构与可视化模块解决数据库返回记录与浏览器展示结构之间的格式差异。Neo4j 返回的记录可能包含节点、关系或路径对象，而 ECharts 关系图需要明确的 \texttt{nodes}/\texttt{links} 结构。系统将子图建构逻辑集中在后端，前端只消费约定好的 \texttt{graph}/\texttt{table} 字段，减少页面中的适配代码，也便于单独测试转换逻辑。图数据转换流程如图~\ref{fig:data-convert}所示。建构完成后，前端根据节点标签设置类别，以关系图展示拓扑结构，以表格展示属性与数值字段。该模块是整条处理链路的末端：其输出直接构成 HTTP 响应体，前端与 Neo4j 数据库之间不存在任何直接通信，所有数据访问均经由后端代理完成。这一设计使前端保持纯展示职责，后端可以在不涉及浏览器的情况下独立测试从数据库记录到节点—边结构的整个过程。

\subsubsection{安全校验模块}

安全校验模块位于两条查询入口之后、数据库执行之前，对所有进入数据库的语句统一检查，校验流程如图~\ref{fig:security-flow}所示。

安全校验包括三项检查。第一项是关键字检查：扫描语句中是否包含写操作关键字（\texttt{CREATE}、\texttt{MERGE}、\texttt{DELETE}、\texttt{SET} 等），发现则拒绝执行。第二项是执行计划预检，由环境变量控制、默认关闭：开启后会在执行前向数据库请求生成查询计划，用于发现潜在的高风险查询。第三项是参数完整性检查：语句中的 \texttt{\$param} 占位符须在请求中有对应的同名参数，防止参数缺失或不匹配。三项检查全部通过后，语句才会进入数据库执行；任意一项失败，系统向用户返回说明具体原因的错误信息，不会触发数据库访问。

\subsubsection{面向其他知识图谱的迁移设计}
\label{subsec:migration}

本系统在设计上将图数据库的具体内容与核心处理逻辑分开：大语言模型通过提示词了解目标数据库的结构，后端通过环境变量连接目标数据库实例。因此，将系统从《迷你世界》知识图谱迁移到另一个 Neo4j 数据库，只需重写提示词并修改部署配置，不涉及任何代码改动。

\paragraph{系统提示词的改写}

系统提示词的作用是告知大语言模型目标数据库的结构，使模型生成与当前库一致的 Cypher 语句。迁移到新数据库时，需要按新库的 Schema 重新编写系统提示词。现有提示词（见附录~\ref{app:prompt-supplement}）可以作为模板，将迷你世界领域的内容替换为目标领域的内容即可。

新提示词需要描述三类信息。第一类是节点标签列表，并列出各标签的主要属性名称。第二类是关系类型列表，说明每种关系的起点标签与终点标签。第三类是一至两个示范性问句及其对应的 Cypher 语句，用于帮助模型理解目标领域的典型查询形态。示范性问句写得越接近实际使用场景，模型生成有效语句的成功率越高。

\paragraph{部署配置的修改}

连接到目标数据库，只需修改四个环境变量：\texttt{NEO4J\_URI} 指定数据库地址，\texttt{NEO4J\_USERNAME} 与 \texttt{NEO4J\_PASSWORD} 提供认证信息，\texttt{NEO4J\_DATABASE} 指定数据库名称。配置方式与初次部署相同，参照仓库中的 \texttt{.env.example} 文件填写对应值即可。

如果迁移时同步更换大语言模型服务商，还需要修改 \texttt{LLM\_API\_BASE}、\texttt{LLM\_API\_KEY} 与 \texttt{LLM\_MODEL} 三个变量。仅更换数据库而保留原有模型服务时，这三个变量不需要改动。

\paragraph{迁移后的验证}

迁移完成后，可以分两步验证结果。第一步运行仓库中的自动化测试套件：这些用例不依赖《迷你世界》的具体数据，在新部署环境中可以直接执行，用于确认接口行为和安全校验逻辑没有受到影响。

第二步是针对新领域设计若干中文问句，覆盖目标数据库中的主要节点标签和关系类型，逐条判断系统生成的 Cypher 是否能正确执行。如果某类问句的成功率偏低，通常意味着提示词中对应的结构描述或示例语句需要进一步调整。

\subsection{数据库设计}
\label{subsec:db-design}

本系统使用 Neo4j 图数据库存储《迷你世界》知识图谱，采用多标签属性图模型。为避免术语歧义，这里先说明本文中的两个领域词：方块指游戏世界中可采集或可放置的地形/材料单元，生物指可交互实体（如动物、怪物或 NPC）。数据库设计充分利用 Neo4j 的多标签特性，将方块和生物作为“道具”这一上位概念的子类，通过复合标签表达继承关系。

数据库设计直接影响自然语言查询和可视化效果。Schema 中的标签与关系类型会进入自然语言模块的提示词，节点上的 \texttt{Name}、\texttt{MineTool} 等属性会影响模型生成 \texttt{WHERE} 条件，也会影响转换模块如何归类节点和生成表格列。因此，数据建模不是孤立的存储设计，而是与查询体验和展示效果联动的整体设计。

\subsubsection{架构演进}

早期曾考虑兼容阿里云 GDB 等不支持同一节点多标签的图引擎。在该约束下，无法直接使用 \texttt{:item:block}、\texttt{:item:monster} 这类复合标签表达“方块/生物是一种道具”的继承语义，只能将方块、生物与通用道具拆成不同节点，再通过业务 ID 用桥接关系（历史上曾建模为 PLACE、SUMMON 等）与道具节点相连。

该单标签桥接方案会拉长查询路径。自然语言转 Cypher 时，模型需要记住“先找道具，再经 PLACE 或 SUMMON 跳到方块/生物”的固定模式，生成错误的概率随之上升。同时，这种建模与用户对“方块和生物也是道具类别”的直观理解不一致，维护成本较高。

当前实现已取消 GDB 兼容路径，数据仅面向 Neo4j 导入与查询。方块与生物直接采用多标签节点建模，不再使用 PLACE、SUMMON 等桥接关系。提示词设计与测试用例均以当前多标签模型为准。

\subsubsection{多标签架构设计}

Neo4j 允许同一节点携带多个标签（Multiple Labels）。本文以 \texttt{:item} 作为道具基类标签（通用属性含 ID、Name、Type、Disc、GetWay）；方块类节点同时使用 \texttt{:item:block}，在基类属性之外增加 MineTool、ToolLevel 等；生物类节点同时使用 \texttt{:item:monster}，增加 Life、Attack 等。具体列示见表~\ref{tab:node-types}。相较早期单标签桥接方案，多标签使 ``方块/生物属于道具'' 直接体现在标签上，查询时可写 \texttt{MATCH (n:item:block) RETURN n.Name, n.MineTool} 等语句而无需经 PLACE、SUMMON 等关系绕行，语义与领域模型一致，并减少不必要的跳转与联结开销。

\subsubsection{Schema 摘要与自然语言模块的衔接}

数据库的节点标签与关系类型不仅是存储层面的设计，也是自然语言模块构建提示词的直接依据。系统通过 \texttt{GET /schema} 接口在运行时向调用方暴露当前图库的结构摘要：后端向 Neo4j 执行元数据查询，取得全部节点标签与关系类型，整理为列表格式后返回。下面是接口响应的示意结构：

\begin{lstlisting}[language=json, caption={\texttt{GET /schema} 响应示例}]
{
    "labels": ["item", "block", "monster", "group", "recipe", "smelt", "device"],
    "relationship_types": [
        "IN_GROUP", "CONSUMES", "PRODUCES", "DROPS",
        "TOOLMINEDROPS", "HANDMINEDROPS", "COVERS", "RUNS_ON"
    ]
}
\end{lstlisting}

自然语言模块在每次处理问句前先读取这份结构摘要，将其填入提示词，使大语言模型能参考当前图库的实际标签和关系名称生成 Cypher。这样做还有一个好处：图库数据更新或系统迁移到新数据库后，摘要内容在下一次请求时自动反映最新结构，不需要手动修改代码。

多标签设计在这里体现出明显的优势。提示词只需用一行示例说明 \texttt{:item:block} 表示方块，模型就能直接生成 \texttt{MATCH (n:item:block)} 形式的查询。如果使用旧版单标签方案，标签只有通用的 \texttt{:item}，模型还需要理解"方块通过 PLACE 关系关联到道具"这样的间接结构，生成正确语句的难度明显更高。

\subsection{自然语言查询模块}
\label{subsec:nlq-impl}

\subsubsection{模块设计}

自然语言查询模块的设计目标是让用户以中文提问，系统自动将问句转化为可在当前图库上执行的 Cypher 语句，用户不需要了解 Cypher 的语法。

\texttt{POST /nlq} 接口接收包含问句的 JSON 请求，一次性返回生成的 Cypher 语句、执行参数、图谱结构和表格数据四部分内容，前端发起一次请求即可获取从问句到可视化所需的全部数据。

两项设计决策对生成效果影响较大。第一，图数据库的结构信息在每次请求时动态读取，而不是事先写入提示词：这样图库结构更新后，提示词始终反映最新的标签和关系类型，避免模型参考过时信息。第二，要求模型返回参数化查询语句，而不是直接拼接字符串：参数化可以防止特殊字符干扰查询逻辑，也为后续的参数完整性检查提供了基础。

\subsubsection{代码实现}

自然语言查询模块的主要工作是根据用户的中文问句生成 Cypher 查询语句。后端在调用大语言模型前先读取当前图数据库的结构信息，将其与问句一并写入提示词；模型返回包含 Cypher 语句和查询参数的 JSON 结果，随后传入安全校验模块处理。

对外入口为表~\ref{tab:api}中的 \texttt{POST /nlq}。请求到达后，系统依次完成读取 Schema、调用大语言模型、安全校验、数据库执行和结果转换五个步骤，时序见图~\ref{fig:nlq-flow}。以下给出请求与响应的数据格式。

请求体格式：
\begin{lstlisting}[language=json, caption={自然语言查询请求体}]
{
    "query": "查找石剑",
    "options": {
        "limit": 100,
        "debug_raw": false
    }
}
\end{lstlisting}

响应体格式（字段名 \texttt{cql} 为历史命名，内容为Cypher字符串）：
\begin{lstlisting}[language=json, caption={自然语言查询响应体}]
{
    "cql": "MATCH (n:item) WHERE n.Name CONTAINS $name RETURN n",
    "params": {"name": "石剑"},
    "graph": {
        "nodes": [...],
        "links": [...],
        "categories": [...]
    },
    "table": {...}
}
\end{lstlisting}

提示词与生成策略可概括为四点：限定只读查询，注入当前库 Schema，说明多标签与路径式 \texttt{RETURN path} 的使用方式，并要求输出 JSON。仅返回孤立节点时，前端难以显示边信息，因此提示词倾向于引导模型返回路径。完整标签、关系类型与多组少样本示例见附录\ref{app:prompt-supplement}。正文仅保留一个代表性示例：
\begin{lstlisting}[language=cypher, caption={合成问句与路径式返回}]
-- 木料能合成什么
MATCH path = (material:item)-[:CONSUMES]-(r:recipe)-[:PRODUCES]-(product:item)
WHERE material.Name CONTAINS '木料'
RETURN path LIMIT 20
\end{lstlisting}

调用与解析的实现摘录如下。自然语言模块将系统提示词与用户问句拼装为 OpenAI 兼容的对话补全接口请求，并要求响应体为 JSON，以便提取 \texttt{cql} 与 \texttt{params}。

\begin{lstlisting}[language=Python, caption={LLM 生成 Cypher（llm\_client.py 摘录）}]
async def generate_cypher(self, nlq, schema_hint, limit):
    payload = {
        "model": self.model,
        "messages": [
            {"role": "system", "content": load_system_prompt()},
            {"role": "user", "content": f"只读 Cypher，LIMIT 默认 {limit}：\n{nlq}"},
        ],
        "temperature": 0.1,
        "response_format": {"type": "json_object"},
    }
    resp = await client.post(f"{self.api_base}/chat/completions", json=payload, headers=headers)
    obj = json.loads(resp.json()["choices"][0]["message"]["content"])
    return obj.get("cql", "").strip(), obj.get("params", {})
\end{lstlisting}

在调用 \texttt{generate\_cypher} 之前，路由层先通过 Neo4j 客户端查询当前图库的标签与关系类型，将结果拼成纯文本摘要，作为 \texttt{schema\_hint} 参数传入。大语言模型在接收到问句时，已知当前库中存在哪些标签与关系，从而生成与实际图库结构一致的 Cypher 语句，而不是依赖静态提示词中可能已过时的描述。

\subsection{图数据访问与 Cypher 执行模块}

\subsubsection{模块设计}

图数据访问模块负责与 Neo4j 数据库交互，对上层屏蔽连接和会话的具体操作细节。

模块向外提供统一的查询方法 \texttt{run\_read(cql, params)}，接受 Cypher 语句和查询参数，返回记录列表和列名。两条查询路径均通过这一方法访问数据库，上层不需要各自处理连接细节。列名随记录一并返回，便于转换模块在构建表格时保持与数据库字段的对应关系。

会话设置为只读模式，在数据库驱动层面阻止写操作，与应用层的关键字检查共同构成两道独立防护。每次查询设有超时时间，会话结束后连接自动释放，不会长时间占用数据库资源。

\subsubsection{代码实现}

任何进入 Neo4j 的查询都由同一个客户端封装统一处理。每次会话以只读模式打开，确保驱动层面不允许写操作。超时时间由配置项以毫秒给出，换算为秒后传入驱动，防止单次查询长时间占用连接。路由层先经过 \texttt{cql\_validator} 的安全校验，通过后才调用 \texttt{run\_read}，将记录列表与列名序列一并返回给转换模块。下列代码展示只读会话的建立与超时换算方式。

\begin{lstlisting}[language=Python, caption={Neo4j 只读查询（neo4j\_client.py 摘录）}]
def run_read(self, cql: str, params: Dict[str, Any] | None = None):
    params = params or {}
    with self._driver.session(
        database=settings.NEO4J_DATABASE,
        default_access_mode="READ",
    ) as session:
        timeout_seconds = settings.QUERY_TIMEOUT_MS / 1000.0
        result = session.run(cql, parameters=params, timeout=timeout_seconds)
        records = [r.data() for r in result]
        return records, list(result.keys())
\end{lstlisting}

\subsection{查询结果子图建构与可视化模块}

\subsubsection{模块设计}

查询结果子图建构模块的职责是将数据库返回的记录转换为 ECharts 可以直接使用的节点—边结构，使前端无需再做格式适配。

输出结构分为三部分：节点列表（含 ID、标签和属性）、关系列表（含起点、终点和关系类型），以及节点标签列表（供 ECharts 按标签为节点着色分组）。属性表格与子图结构并列放在响应中，供用户查阅节点的文本属性和数值字段。

节点去重是建构过程中需要特别处理的环节。当查询返回路径时，同一节点可能出现在多段路径中；若不去重，前端关系图中会出现重复节点，导致连线混乱。模块按 Neo4j 为每个节点分配的唯一 ID（\texttt{element\_id}）去重，保证每个实体在图谱中只出现一次。转换逻辑集中在后端，便于单独进行测试。

\subsubsection{代码实现}

Neo4j 驱动返回的记录中可能混有节点、关系和路径对象，而前端关系图需要明确区分节点集合与边集合。为此，后端设置专门的转换模块，将出库记录整理为 \texttt{graph} 与 \texttt{table} 两部分，以便前端分别绘制关系图和展示属性表格。转换的主要工作是遍历记录、提取节点与关系、按元素 ID 去重，并处理仅有节点而没有边的情形。核心逻辑在 \texttt{echarts\_converter.py} 中实现，下列代码展示处理一条关系时如何确保两端节点先被写入。

\begin{lstlisting}[language=Python, caption={图数据转换（echarts\_converter.py 摘录）}]
def add_rel(rel: graph.Relationship) -> None:
    src, tgt = _node_id(rel.start_node), _node_id(rel.end_node)
    add_node(rel.start_node)
    add_node(rel.end_node)
    links.append({
        "source": src, "target": tgt,
        "category": rel.type, "label": rel.type,
        "value": dict(rel),
    })
\end{lstlisting}

\subsection{安全与约束模块}

\subsubsection{模块设计}

安全校验模块在应用层对所有查询语句做独立检查，不依赖数据库账号的权限配置。这样设计是因为账号配置可能因操作失误被更改，或者大语言模型在某些情况下生成意外的语句，应用层的检查可以作为额外的一道保障。

系统选择黑名单而非白名单策略。白名单只允许特定形式的语句通过，限制过于严格，会影响自然语言模块生成多样化查询的能力。黑名单则明确列举禁止的写操作关键字（\texttt{CREATE}、\texttt{MERGE}、\texttt{DELETE} 等）和管理命令（\texttt{CALL dbms.*}），在不限制正常只读语句的前提下拦截常见的危险操作。

校验分三个步骤。第一步是关键字检查，覆盖常见写操作和管理命令。第二步是参数完整性检查，对比语句中的占位符与请求中提供的参数，防止参数缺失或不匹配。第三步是可选的执行计划预检，由配置项控制、默认关闭，仅在需要额外验证时开启，以避免增加数据库的额外开销。

\subsubsection{代码实现}

进入 \texttt{/nlq} 或 \texttt{/run-cql} 的语句，在提交给数据库执行前都要先经过安全校验模块。模块扫描语句中是否包含 \texttt{CREATE}、\texttt{MERGE}、\texttt{DELETE}、\texttt{SET} 等写操作关键字，以及形如 \texttt{CALL dbms.*} 的管理命令，发现则直接返回错误，不进入执行步骤。即便数据库账号已设为只读，这一检查仍有必要，因为语句中可能出现超出预期的危险操作。可选的执行计划预检由配置项控制，默认关闭，以避免增加数据库的额外开销。校验逻辑集中在 \texttt{cql\_validator.py}，下列黑名单与表~\ref{tab:readonly-test} 中测试用例对应：

\begin{lstlisting}[language=Python, caption={只读校验黑名单（与 cql\_validator.py 一致）}]
WRITE_BLACKLIST = [
    r"\bCREATE\b",
    r"\bMERGE\b",
    r"\bDELETE\b",
    r"\bDETACH\s+DELETE\b",
    r"\bSET\b",
    r"\bREMOVE\b",
    r"\bDROP\s+(?:INDEX|CONSTRAINT|NODE\s+LABEL|REL\s+TYPE|PROPERTY)\b",
    r"\bLOAD\s+CSV\b",
    r"\bCALL\s+DBMS\.",
]
\end{lstlisting}

系统目前未对 \texttt{CALL} 调用的自定义存储过程建立白名单，因此部署时有一定的环境要求：图库应以只读账号连接，服务应置于内网，或经过鉴权后再对外开放。满足上述条件时，能够执行查询的请求来源已限定在可信范围内。若需在不可信网络中直接开放接口，应在网关层添加额外的访问控制，或扩展校验规则。

参数完整性校验从 Cypher 语句中提取全部 \texttt{\$param} 占位符，再与请求中的参数键逐一比对，确保两者完全一致。\texttt{/run-cql} 与 \texttt{/nlq} 共用这一逻辑，具体文件路径见第~\ref{subsec:repo-and-paths}~节。

以下代码展示参数完整性校验的核心实现。函数先提取语句中的全部占位符名称，再与请求参数键做集合比较；若两个集合不相等，返回错误信息并列出缺少和多余的键名，便于排查问题。

\begin{lstlisting}[language=Python, caption={参数一致性校验（cql\_validator.py 摘录）}]
def validate_params(cql: str, params: dict) -> None:
    placeholders = set(re.findall(r'\$(?P<name>\w+)', cql))
    provided = set(params.keys())
    if placeholders != provided:
        missing = placeholders - provided
        extra = provided - placeholders
        raise CqlValidationError(
            f"参数不匹配：缺少 {missing}，多余 {extra}"
        )
\end{lstlisting}

\begin{thesisconclusion}

本文针对属性图数据库使用门槛高、查询结果难以直观呈现实体关系的问题，设计并实现了一套基于生成式 AI 的知识子图建构与可视化系统。系统允许用户在浏览器中以中文提问，自动生成 Cypher 语句并在 Neo4j 中执行，将返回记录转换为关系图和属性表格在前端展示，用户不需要学习图数据库查询语言。系统的所有数据库操作均限定为只读，不支持写入。

系统以《迷你世界》游戏知识图谱为应用场景，从两个层面完成了验证。自动化测试共 38 个用例，覆盖接口行为、只读校验和子图建构转换，全部通过。对 21 条真实中文问句的逐条评判显示，17 条（81\%）得到可接受的结果。两类测试表明，系统在当前数据与配置下能够可靠运行，并能完成从查询到图形化展示的完整流程。

\subsection*{主要工作}

（1）系统架构。系统采用前后端分离的浏览器/服务器结构，后端基于 FastAPI 提供自然语言查询、直接 Cypher 查询与图数据库结构摘要三类接口，前端使用原生 HTML/JavaScript 实现单页面交互，以 ECharts 渲染可交互的关系图，并以表格展示属性数据。

（2）自然语言到 Cypher。后端在调用大语言模型时将当前图数据库的结构信息填入提示词，要求模型返回只读的参数化 Cypher 语句，使中文问句能够转化为可在 Neo4j 中执行的图查询。

（3）安全与执行。所有进入数据库的语句在执行前须经过写操作关键字过滤和参数完整性检查，并设有查询超时和返回行数上限，自然语言入口与直接 Cypher 入口遵循相同的校验规则。

（4）数据与领域模型。系统采用 Neo4j 多标签属性图组织《迷你世界》的道具、配方与生物数据，用于验证自然语言查询、安全执行、子图建构与可视化展示的完整流程。

\subsection*{不足与展望}

当前系统仍有以下不足：
\begin{enumerate}[(1)]
    \item 复杂查询、分组查询和多实体对比仍有失败情形，原因主要是系统对复杂语义的理解和对属性键名称的匹配还不够准确。
    \item 同义词和近义表达的处理较弱，用户用不同名称指代同一概念时，系统可能无法正确匹配。
    \item 权限控制目前仅依靠应用层的只读校验，没有完整的用户身份认证机制，不适合在公开环境中直接部署。
\end{enumerate}

未来工作方向：
\begin{enumerate}[(1)]
    \item 引入 RAG（检索增强生成）或实体链接技术，提高系统对实体名称、属性和关系类型的匹配准确率。
    \item 建立领域同义词和别名词表，使不同说法指代同一概念时能够得到一致的查询结果。
    \item 完善身份认证和访问控制，使系统能够在非演示场景中安全部署。
    \item 探索将系统迁移到其他领域知识图谱的方法，使同一套查询流程能够适配结构相近的不同数据库。
\end{enumerate}

\end{thesisconclusion}
