\begin{abstract}
    知识图谱以图结构表示实体与关系，在知识管理、智能问答与推荐等任务中应用较广。在属性图数据库中，要取出``谁与谁以何种边相连''通常须书写 Cypher 等形式化语言：使用者需了解节点标签、关系有向、模式匹配与 \texttt{WHERE} 中属性键的选取，并在客户端理解表格式行集。上述可统称为面向终端用户的\textit{使用门槛}。本研究面向的读者是希望在浏览器中探索已入库的领域知识、但\textit{不准备}为单次取数而先系统掌握 Cypher 与图客户端操作的终端用户；\textit{在工程上}``做好''的验收是：在第3章用户故事、第6章可对照样例与协议下，以中文问句经同一受控链路得到可读的表与点边图，并配套安全与可复现说明（细则见第3、6章，不在摘要展开）。

    本研究在浏览器/服务器结构下设计并实现系统 neo4jSlave。前端以中文提问；FastAPI 自 Neo4j 读取图结构摘要（即当前库中节点标签、关系类型等的简短清单，后文同）并与问句一并送入大语言模型，得到参数化只读 Cypher；经应用层只读与参数规则校验后由 Neo4j 执行。所谓\textit{只读 Cypher}，指不修改图数据、不执行管理类调用的取数语句。结果以表格与 ECharts 点边图呈现，连接与端点可经环境变量配置。

    验证分两类。其一为自动化：测试 38 条，覆盖语句校验、记录到图谱结构转换与对外 API 契约，全部通过，其中对 \texttt{/nlq} 的自动化路径对大语言模型使用 Mock，故不度量问句到语句的语义是否``真的符合用户原意''。其二为自然语言走查：在人工按 \texttt{doc/paper/nlq\_eval\_protocol.md} 做通过/不通过判定的协议下、并在第6章钉死的环境中，以 21 条中文问句做一次性统计。在此\textit{对照框架}下，可接受问句条数与总条数之比约 81.0\%（第6章按类统计表）；与未参加同一套协议的其他模型、数据集\emph{不}作横向排名比较。与公开 Text2Cypher 基准竞赛类工作相比，本研究侧重点是可独立部署的 HTTP 全链路与可复现配置，而非训练新刷榜模型。可复现所涉公开仓库、路径约定与环境变量表见第2章。
\end{abstract}

\addcontentsline{toc}{section}{Abstract}
\begin{thesisabstracten}
Knowledge graphs represent entities and relationships in a graph-structured form and are widely used in information systems. In a property graph store, ad hoc questions such as ``how entities are linked'' are usually answered by Cypher: one must know labels, directed relationship types, pattern match syntax, and how to read tabular result rows. The \emph{usability gap} in this sense is the training and trial cost before a non-developer can write a first correct query. This work targets end users who wish to explore an ingested knowledge graph in a web browser and do not intend to invest in a full Cypher and desktop-client path for occasional queries. As an acceptance criterion, the delivered service should allow Chinese questions, under a documented and testable path, to yield readable tables and a node--link view together with the same application-layer safety story as in Chapters~3 and~6. Details of stories and test tables are in those chapters.

This work implements neo4jSlave, a browser--server system. The backend reads a \emph{schema summary} (a short list of current node labels and relationship types) from Neo4j, sends it with the question to a large language model, and receives parameterized \emph{read-only} Cypher; here \emph{read-only} means statements that do not modify graph data or call administrative procedures. The FastAPI layer validates read safety and parameters before execution. Configurable environment variables set endpoints and connection strings. The 81.0\% accept rate for natural language is not compared against other models or leaderboards: it is reported in a \emph{fixed} protocol, human pass/fail judgment, and the frozen environment tabulated in Chapter~6, alongside 38 \texttt{pytest} cases (with the LLM mocked for \texttt{/nlq} in the automated path). The emphasis is a deployable HTTP stack and replicable configuration rather than training a new model for a public Text2Cypher challenge. A public source repository, relative path convention, and the environment table are in Chapter~2.
\end{thesisabstracten}

\section{系统概述}

\subsection{研究背景}

本课题要解决的问题可概括为三点：让不掌握图查询语言的人也能用自然语言在浏览器里查询某个领域知识库中的关联关系，并把结果用关系图等形式直观展示出来，同时在对外提供服务时把误写、误改数据等风险压到可接受水平。为便于非图数据库方向读者把``门槛''说具体，不妨这样理解使用门槛的\textit{对象}：不借助本系统时，用户要亲自完成``把中文里提到的道具/方块/生物等实体名，映射到 Cypher 中 \texttt{MATCH} 子句的节点标签、\texttt{WHERE} 中的 \texttt{Name CONTAINS} 或等值条件，并选对关系有向与跳数''；这通常比学会用关键词在网页中搜索要慢得多。贯穿全文的直叙例子是：在《迷你世界》类属性图上，用中文问``查找石剑''或``木料能合成什么''，而不是先写出完整 \texttt{MATCH\ldots RETURN} 再在表格行里自己看出合成链。

知识图谱以有向图的方式表示实体为顶点、关系为边，可附带顶点和边上的属性，用来回答“与某实体相连的另一类实体是什么、沿怎样的路径连起来”等问题；这类表达在信息检索、推荐、金融风控、生物信息等领域已被广泛接受。工程上，领域数据常被整理成图数据库中的可查询图；其中较常见的一种抽象称为属性图（Property Graph），即把实体与关系都看成可带键值对属性的“对象”，用声明式图查询语言描述模式匹配与路径。与之并列的另一类路线是资源描述框架（RDF, Resource Description Framework）及其配套查询语言 SPARQL：RDF 把知识写成大量“主语—谓语—宾语”的三元组，擅长开放数据互联与规范化的语义网场景；属性图则更贴近面向工程实现的存储与产品迭代。二者并无绝对优劣，取舍取决于数据形态、团队熟悉度与是否已有现成图实例。

在此基础上，Neo4j 是常用的开源属性图数据库之一，其官方查询语言为 Cypher。Cypher 属于形式化查询语言：使用者需要了解标签、关系类型、（可选的）多跳路径等概念。因此，对许多希望“用日常语言提一句、就看到关系图”的终端用户而言，直接手工编写 Cypher 的学习成本仍然偏高。近年在大语言模型（LLM, Large Language Model）推动下，以自然语言生成可执行 SQL、Cypher 等结构化查询已成为可部署的技术路线之一。在已有工作基础上，本研究为以属性图组织、以中文描述为主的领域数据，实现可说明依赖与可复现运行的 B/S（Browser/Server）服务；可复现依赖、公开仓库地址与路径约定见第2章。

下面对知识图谱的建模与查询、自然语言到图查询的既有工作，以及大语言模型与知识图谱的协同作归纳，以便与后文第2章起的技术选型和实现一一对照\cite{ref6,ref7,ref8,ref12,ref13,ref14,ref15}。现有文献与产品通常分别回答“图如何建”“自然语言到 Cypher 的模型指标如何”，缺口一：在既定属性图与中文问句下，可交予他人依文档在本地起服务的“问句—受约束 Cypher—表格与关系图”闭环往往不成整体；本研究的对象即实现上述闭环，并在第2--6 章自环境、需求、设计、实现到测试与统计表可核对。

\subsection{国内外研究现状}

\subsubsection{知识图谱的数据模型、存储与查询基础}

从数据如何被写成“图”看，研究界与产业界已形成了若干可对照的范式。一类是以 RDF 与 SPARQL 为代表的三元组建模：把事实拆成大量三元组，便于在开放万维网上发布、链接和复用，并与本体、推理规则等机制配合；其代价往往是查询与维护需要熟悉较完整的语义网技术栈\cite{ref17}。另一类是属性图：在实体为节点、关系为边的图之上直接挂载属性，查询语言通常围绕模式匹配与路径展开。王鑫等从数据管理角度概括了图数据在存储、查询与知识融合等任务上的一类共性问题（如规模与索引、查询优化、事务等）\cite{ref1}。Robinson 等以图思维对比关系表建模\cite{ref3}；Miller 等从应用侧回顾了以 Neo4j 等为代表的图数据库用法\cite{ref2}。

在实例数据已具备可导入、可复现属性图形态、且应用层需以主流语言与 HTTP 服务对接时，Cypher 与 Neo4j 的文档与驱动支持成熟，Cypher 语义以官方手册为准\cite{ref12}。RDF/SPARQL 在强互联或 RDF 存量下更合适；在已有属性图、且交付以最少组件提供“可运行的 Web 查询”时，选择属性图与 Neo4j 是上述约束下的一种技术选型。但仅有图库仍不解决第1.1 节中的易用问题。易用问题在此特指：即便库内已有数据，只要入口仍是``手写 Cypher''，用户就仍须越过上一段所述的语法与 Schema 理解门槛；故自然语言接口在下一小节单独讨论。

\subsubsection{自然语言到结构化查询}

上一小节说明了“把领域数据存成可查询的图”的主流做法；但终端用户更常见的需求是用自然语言（NL, Natural Language）提问，而不愿学习形式化语言。为把讨论落到可对照的科研脉络上，先看关系型数据库中研究时间更长、文献更集中的文本到 SQL（Text-to-SQL）问题：其目标是把问句转写为可执行的 SQL（Structured Query Language）。Qin 等的综述将主流技术路线归纳到编码器—解码器、中间表示与预训练语言模型等框架下，并指出模式（Schema）感知与可执行性约束（保证生成可执行）是决定效果的关键因素\cite{ref5}。Text-to-SQL 可作为“自然语言到结构化取数”的一条常见参照，便于与后文至 Cypher 的任务对照。

再看图数据库场景。当底层存储是属性图、查询语言是 Cypher 时，与 Text-to-SQL 可类比的任务常被称作将自然语言映射为 Cypher 的 Text2Cypher：Ozsoy 等指出，即便借助大语言模型做零样本生成，在复杂语义与多跳约束上仍易产出不完整或错误语句；在公开高质量语料相对有限的前提下，用清洗、整合多源数据做微调，往往能在 双语互译评估（BLEU, Bilingual Evaluation Understudy）等指标上获得可观收益\cite{ref16}。这提示在工程上不能仅靠“换大模型”本身，还需要与当前库结构对齐的提示与约束，才能把生成结果稳住。

在问答（QA, Question Answering）等相近任务中，从自然语言到 SPARQL 或到 SQL/其它逻辑形式，也长期存在解析或检索—排序等流水线\cite{ref5}。它们与图库自然语言查询的共同点在于：都必须在自然语言与符号查询之间的词汇鸿沟、组合式泛化、以及执行安全（只读、避免危险子串等）之间做权衡量化以外的工程取舍。本研究未在专用语料上训练新的 Text2Cypher 小模型，而在通用大语言模型与提示词约束的基础上，在生成之后叠加应用层只读校验、参数绑定与超时/行数等规则，以限制误执行风险，与当前常见“通用 LLM + 受控后处理”的落地方式一致\cite{ref4,ref9}。

\subsubsection{大语言模型、提示学习与知识图谱的融合}

承接上一小节，大语言模型除了用于“写查询”，还被广泛用于与知识图谱在知识增强、对齐、推理与可解释性等方向协同。Pan 等的路线图式综述指出，二者既可互相补充（图可为模型提供事实依托，模型可为图提供自然语言入口），也带来幻觉与结果不一致等风险，通常需借助检索、约束或外部验证抑制\cite{ref11}。在图数据库自然语言查询子场景中，常见分工是：大语言模型作问句到 Cypher 的非确定性（软）映射，Neo4j 作确定性的执行与可记录日志；中间层在文献与工程中多为只读、参数绑定、超时、结果量限制等，以控制部署风险\cite{ref4,ref9,ref11}。以数据集与自动评测为主的 Text2Cypher 工作较多\cite{ref16}，与在单一领域、多标签 Schema 下、把同一应用层同时接到 HTTP 接口与可交互关系图展示的论述相对独立。

缺口与本文的对应关系（可核对）：上节缺口一具体为——文献和手册不单独构成“可在他机依文档起服务”的中文问句、受约束 Cypher 与可浏览结果的闭环。本研究不引入新图查询语言、不以训练新模型为课题目标，而是实现上述闭环：大语言模型生成参数化、只读 Cypher，经应用层安全与参数校验，结果以 ECharts 与表格输出；需求与复现性自第3章起、测试与 38 条自动化与 21 条走查表见第6章。

\subsection{问题描述与挑战}

\paragraph{与摘要一致的贯穿用例。} 若将``查找石剑''在脑中翻译成操作序列：不借助本系统时，用户应打开 Cypher 客户端、参照 Schema 知悉道具类标签、手写含 \texttt{Name} 的 \texttt{WHERE} 子句并 \texttt{RETURN}，在纯表结果中自己辨认节点；借助本系统时，同一句话可在浏览器中提交，经第5章所述自然语言链路与第4章多标签图，得到可对照协议判定是否``通过''的表与关系图。后文第4、5、6章在讲模块与测试时，均可回到这一简例理解``本系统在替用户省掉哪一步''，而不预设读者会独立调试 Neo4j。

不掌握 Cypher 的终端用户难直接在手写 Cypher 路径下完成探索；仅依赖现场代写则难以稳定复现与自学。表格式中间结果不突出拓扑；在浏览器中呈现点与边，有利于把握“谁与谁以何种边相连”。上两项对应第1.1 节目标中的易用与可感知。

在引入大语言模型之后，还会叠加新的工程问题：模型可能生成非只读语句、遗漏与 Schema 绑定的参数，或在多标签的库结构上选到不理想的路径。因此，本系统不能只把“大语言模型接到字符串出口”，而必须在前后端中明确划分只读校验、参数与占位符、超时与结果量限制，并协调查询结果的形态，使其同时便于表格展示与关系图绘制。以上矛盾将在后文落实在自然语言模块、数据访问与执行模块、结果转换与可视化模块、以及安全与约束模块中分别展开。

\subsection{课题意义}

从实践意义看，本系统可当作某一领域知识库上的“轻量查询入口”：数据维护、内容编辑、教学或演示等场景中，用户通过浏览器即可访问，无需在本地安装数据库图形客户端，也无需记忆固定 Cypher 模板。从教学与复现意义看，论文后续章节将按“需求—设计—实现—测试”展开，并给出环境配置、应用程序接口（API, Application Programming Interface）约定与关键算法说明，使具备 Python 与 Web 基础的读者能够在本地复现同类流程（将图数据与 LLM 端点替换为自有资源即可平移）。

从技术意义看，本研究将自然语言到 Cypher 的受约束生成、应用层只读与参数等规则、与结果到关系图的转换，并置为一条可核对的实现链；多标签 Schema 与提示、取数形态的关系在数据与第5章说明。与同类垂直领域属性图在``低门槛访问''上是否可迁移，先须说明：此处的``低门槛''不是抽象口号，而是第1.1 节中``不必先学会手写完整 Cypher''在另一数据集上能否仍成立。该可迁移性取决于其 Schema 是否与本文假设一致、部署是否仍满足只读与接口约束，不作未经验证的外推。

\subsection{本文主要工作}

第6章的测试以第\ref{subsec:ch3-functional-req}~节所列功能为对象；本节仅给出四模块的划分及与后章的对应，避免与第3、4章逐条重复“原因—作用—优势”体例。

（1）自然语言查询模块（NL→Cypher）。职责：自 Neo4j 取 Schema 摘要、调用大语言模型、得到参数化只读 Cypher；详细流程与问题见第\ref{subsec:ch3-functional-req}~节、第4.3.1 与第5.1节。

（2）图数据访问与 Cypher 执行模块。职责：在只读、超时、统一参数语义下执行 Cypher，提供 \texttt{/run-cql}、\texttt{/schema} 等；见第\ref{subsec:ch3-functional-req}~节、第4.3.2 与第5.2节。

（3）结果转换与可视化模块。职责：将驱动返回的节点/关系/路径等转为 ECharts 与表结构；见第\ref{subsec:ch3-functional-req}~节、第4.3.4 与第5.3节。

（4）安全与约束模块。职责：在两类入口前对语句与参数作统一处理。设计不变量为：（i）不执行应用层已拦截的写类子串与参数不齐全语句；（ii）\texttt{\$param} 与请求体中参数名一致时方可执行（实现见第5.4节）；（iii）单次查询在超时与行数上限内结束。说明见第4.3.3 节与第5.4节。“在数据库层已只读时仍保留黑名单”不作为价值判断，而作为上述不变量所针对的故障假设与部署说明。

第6章汇总自动化用例、人工问句与模块对应关系。

\subsection{技术路线与第1章问题的对应关系}

第1章在问题描述中把矛盾归纳为三方面：形式化查询门槛、查询结果如何更易感知、在开放网络环境下执行生成语句需安全约束。第2章以下各小节可理解为对上述三点的\textit{工程回答}，而不是与需求无关的栈展览：大语言模型与提示用于减轻``手写 Cypher''；ECharts 与第5章转换器用于把行集变成点边；FastAPI 与只读校验用于在对外接口上先挡一层写类与危险调用。技术选型据此形成一条可对照的链路：

\begin{itemize}
    \item 大语言模型 + HTTP 客户端：把自然语言到 Cypher 的映射交给大语言模型完成；通过 超文本传输安全协议（HTTPS, Hypertext Transfer Protocol Secure）等通道调用与 OpenAI 习惯相近的 应用程序接口，便于在不大改业务骨架的前提下更换云厂商或模型端点。
    \item FastAPI：统一以表述性状态转移（REST, Representational State Transfer）风格的 Web API 对外暴露\cite{ref6,ref10}，同时承载“自然语言入口”与“直接提交 Cypher”两条路径，并托管静态前端，部署上保持单一可执行服务。
    \item Neo4j 官方 Python 驱动：保证与数据库协议一致，支持只读会话与查询超时。
    \item ECharts：将图查询结果映射为前端可渲染的数据结构，降低自研渲染引擎的成本。
\end{itemize}

\subsection{技术栈}

在给出按组件独立比较的总览后，本节按“后端语言与框架—数据库—模型服务—可视化—工具链”的顺序说明各技术项的职责与取舍理由，避免读者只见名词而不见关系。

\subsubsection{Python 3.10+}

Python 语法清晰，在数据与 Web 方向库齐全。本系统后端完全采用 Python，便于与 Neo4j 驱动、HTTP 客户端、pytest 测试框架在同一生态内集成。选择 3.10+ 是为了使用较新的类型标注与标准库特性，并与主流 Linux 发行版默认或软件源中的 Python 版本相匹配。

\subsubsection{FastAPI}

FastAPI 基于 Starlette 与 Pydantic，支持异步路由、自动 OpenAPI 文档与请求体验证。相较 Flask，其异步与类型提示更适合同时处理 LLM 网络请求与数据库 IO 的混合场景；相较 Django，其体量更轻，契合本课题“单一服务 + 静态前端”的部署形态。在本系统中的职责是：注册 \texttt{/nlq}、\texttt{/run-cql}、\texttt{/schema} 等路由，挂载静态资源，并在中间件层配置 CORS（演示环境宽松、生产环境应收紧，见第3章非功能需求）。

\subsubsection{Neo4j 图数据库}

Neo4j 使用 Cypher 与属性图模型，社区资料与示例丰富，且支持多标签节点——这与本文领域数据中“道具 / 方块 / 生物”的继承式建模一致（详见第4章）；其应用与概念讨论可见早期文献\cite{ref2}，产品与协议细节以官方资料为准\cite{ref7}。未首选 JanusGraph、ArangoDB 等的原因主要是：本课题已有 Neo4j 导入管线与实测数据规模，且 Cypher 与 LLM 训练语料中的“图查询”示例重合度较高，有利于 Prompt 设计。

\subsubsection{大语言模型（LLM）}

系统通过 OpenAI 兼容的 HTTPS API 调用大模型（如 GPT 系列、Kimi、豆包等），由 \texttt{LLM\_API\_BASE}、\texttt{LLM\_MODEL} 等环境变量配置；大模型提示学习与指令遵循能力的研究脉络见文献\cite{ref4}，商用 GPT-4 系列的技术报告见文献\cite{ref9}。作用是将用户问句与 Schema 摘要拼装为提示词，解析模型返回的 JSON 得到 Cypher 与参数。优势是无需自训专用模型即可迭代；代价是需网络与 API 费用，且生成结果必须经后文安全模块过滤。

\subsubsection{ECharts}

ECharts 为 Apache 开源项目，关系图（Graph）组件成熟，支持力导向布局与交互。在本系统中负责将后端返回的 \texttt{nodes}/\texttt{links} 绘制为可拖拽、可缩放的图谱；与 D3.js 相比上手更快，与 Cytoscape 相比更贴近 Web 单页应用嵌入需求。

\subsubsection{其他工具}

\begin{itemize}
    \item Git：版本管理与协作。
    \item pytest：后端单元测试与 API 契约测试（LLM 可 Mock）。
    \item httpx：异步/同步 HTTP 客户端，用于调用 LLM API。
\end{itemize}

\subsection{系统架构与数据流}

本系统采用前后端分离的 B/S 架构，如图~\ref{fig:architecture}所示：

\begin{figure}[H]
    \centering
    \includegraphics[width=0.95\textwidth]{img/architecture.png}
    \caption{系统总体架构图}
    \label{fig:architecture}
\end{figure}

各层职责及相互关系可概括为一条主数据流：

\begin{enumerate}[(1)]
    \item 浏览器（前端层）：用户输入自然语言或 Cypher；将请求发往 FastAPI；接收 JSON 后由 ECharts 绘圈、由表格区展示属性。
    \item FastAPI（后端层）：对 \texttt{/nlq} 请求先拉取 Schema、再调 LLM 生成 Cypher；对两类执行请求均先做只读校验与参数检查，再调用 Neo4j。
    \item Neo4j（数据层）：在只读会话中执行 Cypher，返回记录流。
    \item 转换逻辑（后端层内）：将记录转为图与表结构，随响应返回前端。
\end{enumerate}

由此可见，自然语言模块、执行模块、安全模块、可视化模块并非孤立罗列：前两者串成“问句→语句→结果”，安全模块横切在执行之前，转换模块衔接数据库与前端展示。第4章将从设计角度进一步展开模块接口与数据库模型。

\subsection{可行性分析}

\subsubsection{技术可行性分析}

Web 框架、图数据库、经 HTTPS 调用的大语言模型与浏览器端可视化均为可获取的成熟技术。FastAPI 与 Neo4j 官方 Python 驱动支持在单机完成“启动服务—连库—返回 JSON”；LLM 经 HTTPS 兼容接口接入，与具体厂商解耦；ECharts 以 JSON 驱动画布。本仓库已提供可运行后端、静态前端、\texttt{pytest} 用例与 \texttt{.env.example}，可组成本系统流水线。自然语言问句的优劣受提示与 Schema 影响，属效果调优，不改变可搭建、可服务的技术结论。

\subsubsection{经济可行性分析}

开发期以人力与算力为主，运行期以图库存储与 LLM 调用量为主。Neo4j 社区版与相关 Python 依赖可零许可费使用，LLM 多按量计费；通过缓存 Schema 与第5章中 \texttt{QUERY\_HARD\_LIMIT}、\texttt{QUERY\_TIMEOUT\_MS} 等可限制单次查询成本。

\subsubsection{操作可行性分析}

B/S 架构下终端用户仅需浏览器，自然语言与直接 Cypher 可切换，与第4章单页、Schema 可见与语句回显设计一致。

\subsubsection{功能需求}
\label{subsec:ch3-functional-req}

（1）自然语言查询。

原因：课题要解决的``门槛''是：用户不经过培训即难以手写第1.1 节所述的 Cypher 与在结果里看出关联；大语言模型在此承担``把问句非正式表述映射到库上可执行语句''的角色，因而是核心入口。

行为：用户输入中文问句，经 LLM 生成参数化 Cypher 后执行。

关联：依赖 Schema 查询（5）提供标签/关系摘要；依赖安全限制（6）过滤写操作；执行结果交由（3）（4）展示。

（2）直接Cypher执行。

原因：调试、对比与教学演示需要“可重复、可精确控制”的语句通道，不能完全依赖模型生成。

行为：用户在“Cypher”模式下提交语句（API 路径 \texttt{/run-cql}）。

关联：与（1）共用同一套只读校验、参数校验与 Neo4j 会话，避免两套安全逻辑。

（3）图谱可视化。

原因：图数据的价值在于拓扑；表格式难以传达“谁连谁”。

行为：将查询结果映射为 ECharts Graph 的节点与边。

关联：依赖自然语言或 Cypher 模块返回含路径/关系的记录；与（4）互补——图看结构，表看属性。

（4）数据表格展示。

原因：数值属性、长文本描述在图中不便阅读。

行为：按列键展平记录为表格。

关联：与（3）同源 JSON，同一响应体中并行返回，前端分区渲染。

（5）Schema查询。

原因：用户与 LLM 均需知道“图里有什么标签/关系”，否则问句与生成语句易偏离领域。

行为：提供 \texttt{/schema} 等接口。

关联：NL 链路在生成前拉取 Schema；前端也可展示供高级用户参考。

（6）安全限制。

原因：开放执行通道必须防止误删与危险调用。

行为：黑名单、参数校验、超时等。

关联：横切（1）（2），在访问数据库前统一生效。

\subsubsection{非功能需求}

（1）安全性。即便数据库账号为只读，应用层仍应拒绝明显写语句，防止配置失误或 LLM 异常输出。

如何达成：正则黑名单 + 可选 EXPLAIN + 参数白名单式校验（见第4章安全模块与第5章实现）。

（2）性能。交互式查询对尾延迟敏感；LLM 已引入秒级延迟，图查询与渲染不应再放大瓶颈。

如何达成：Neo4j 会话超时、结果行硬上限、前端力导布局在合理节点规模内运行。

（3）可扩展性。课题结束后可能更换模型、迁移图实例或替换可视化方案。

如何达成：环境变量配置 URI 与 API；转换器与路由分层，避免把业务写死在单一文件中。

（4）易用性。主要受众不掌握 Cypher。易用性在此不抽象成``好看''，而落实为：用户不必离开浏览器即可完成``提问—看生成句—看表与图—改问''的闭环；与（1）并列，共同构成第1.1 节中``自然语言在浏览器中查询并感知结果''的验收子集。

如何达成：单页、少步骤、生成语句可回看；错误信息返回可读提示（如缺参列表），减少``接口返回 500 而不知道为什么''的再次调试成本（该成本在评审视角下与``门槛''同构）。

（5）部署与网络策略。演示环境与生产环境风险不同。

如何达成：开发期宽松 CORS；上线前收紧为白名单并置于反向代理之后（与第1章非功能叙述一致）。

\subsection{性能分析}

（1）LLM API 响应时间。自然语言转 Cypher 通常占端到端延迟的主要部分（常见为 1–3 秒量级，视模型与网络而定）。

应对：不重复调用模型（单次请求携带 Schema）；失败时快速返回错误，避免阻塞线程池。

（2）Neo4j 查询性能。与模式复杂度、索引及数据量相关；本课题数据规模为万级节点、万级有向边，常见路径查询在毫秒至数百毫秒量级。

应对：限制 \texttt{LIMIT}、避免无界全图扫描类语句进入生产（由 Prompt 引导与硬上限双保险）。

（3）前端渲染性能。ECharts 在万级节点仍较流畅，但力导布局成本随边数上升。

应对：与（2）同一上限逻辑耦合——控制返回节点/边数量即控制渲染成本。

综上，性能目标通过“限制规模 + 超时 + 合理 Prompt”组合达成，而非依赖未经验证的无限扩容假设。

\subsection{本章小结}

本章从可行性、需求与性能三方面论证了系统的建设边界：技术上四类组件可组装为稳定流水线；经济上以开源栈与可配置的 LLM 调用为主；操作上以浏览器单页降低第1.1 节所述``为单次取数而先学 Cypher''型门槛。需求分析在参与者与用例图（图~\ref{fig:use-case-placeholder}）框架下，将六类功能与五项非功能写成“动因—行为—关联”，并辅以表~\ref{tab:req-map}与用户故事（表~\ref{tab:user-stories}）明确可验收行为，避免条目孤立。性能分析则把端到端延迟拆解为 LLM、Neo4j 与前端三环节，并给出与第5章实现参数相对应的应对思路。以上结论为第4章的系统结构与第5章的详细实现提供依据。

\section{系统设计}

\subsection{设计原则与模块关联总览}

第3章在需求上已说清``要建什么''；第4章回答``在结构上如何摆放''。读者若不熟悉 Neo4j，可先抓住一条主线：自然语言与直接 Cypher 在到达 Neo4j 前必须走同一只读/参数/超时策略，转换层把记录变成前端可画的 JSON。贯穿例子仍是：以中文问``查找石剑''类问句，与第1.1 节、第6章样例在叙事上可逐句对齐。

系统设计遵循三条原则：

（1）职责分离——自然语言生成、语句校验、数据库访问、结果转换、前端渲染各有明确边界；

（2）横切安全——凡进入 Neo4j 的语句必经同一安全与参数策略，避免 NL 与直接 Cypher 两条路径不一致；

（3）可观测——生成语句与错误原因可返回给调用方，便于调试与论文复现。

从宏观关联看，五大块（NL、执行、安全、转换、可视化）并非并列孤立：NL 与执行共享“入口路由 + Neo4j 客户端”；安全嵌入在“执行之前”；转换紧接“记录出库之后”；可视化消费“转换后的 JSON”。图~\ref{fig:modules}给出模块划分，下文各小节在描述单模块时均会回扣上述数据与控制流顺序。

\subsection{系统总体架构}

系统采用分层架构设计，分为表示层、业务逻辑层、数据访问层和外部服务层。采用分层的原因包括：便于替换实现（例如换 LLM 厂商不换库）、便于单测（对 Neo4j 与 LLM 做 Mock）、便于在文档中固定“请求从哪进、从哪出”的说明口径。

\begin{enumerate}[(1)]
    \item 表示层：前端 HTML 页面，使用 ECharts 进行图谱可视化，通过 JavaScript 与后端 API 交互。与下层接口：仅依赖 JSON 合同，不感知 Neo4j 协议细节。
    \item 业务逻辑层：FastAPI 应用，包含 API 路由、自然语言编排、数据转换等。与下层接口：调用 Neo4j 客户端与 httpx LLM 客户端；对上统一返回 HTTP + JSON。
    \item 数据访问层：Neo4j 客户端，负责会话、只读模式与超时。为何独立一层：避免路由处理函数内散落驱动代码，便于集中调优连接参数。
    \item 外部服务层：大语言模型 API。为何单列：其可用性与配额不由本机完全控制，在架构上视为外部依赖，便于标注风险与降级策略（如返回明确错误码）。
\end{enumerate}

\subsubsection{自然语言查询模块}

设计动因：若缺少该模块，非专业用户无法把“中文问句”稳定映射为“合法且高效的 Cypher”；若完全信任模型输出而不接安全与执行层，则无法落地为可靠系统。

与相邻模块的关系：Schema 经数据访问层获取；提示词拼装后调用外部 LLM；生成文本再经安全校验模块与Cypher 执行模块；出库记录由可视化/转换模块统一格式化，形成闭环。

该模块是系统的核心功能，主要流程如图~\ref{fig:nlq-flow}所示：

（1）用户输入自然语言查询；

（2）系统获取当前知识图谱的Schema信息（节点标签、关系类型）；

（3）构建Prompt，将Schema信息和用户查询发送给LLM；

（4）LLM生成Cypher查询语句和参数；

（5）系统进行安全校验（只读关键字黑名单；可选的EXPLAIN预检，默认关闭，见\texttt{ENABLE\_EXPLAIN\_VALIDATE}）；

（6）执行查询，获取结果；

（7）将结果转换为ECharts图数据或表格数据返回。

如何达成较好效果：Prompt 采用系统提示词 + Few-shot，强调多标签 Schema、路径型 \texttt{RETURN path} 与参数化占位符，引导 LLM 生成可执行且利于前端画边的只读查询；自然语言到结构化查询（如 SQL）的深度学习综述可与“NL→Cypher”任务对照阅读\cite{ref5}。最终仍须经黑名单校验，避免“能生成”与“能执行且安全”混为一谈。

\subsubsection{Cypher执行模块}

设计动因：调试人员、课程演示或“已知正确 Cypher”场景需要绕过 LLM，直接验证图数据；若与 NL 模块分裂为两套执行路径，则安全策略易不一致。

与相邻模块的关系：与 NL 模块共用 Neo4j 客户端与同一套只读/参数校验逻辑；查询成功后同样进入转换模块输出图与表，保证前端无分支差异。

支持用户直接输入Cypher查询语句执行：

（1）解析Cypher语句，进行只读安全检查；

（2）提取查询中的参数占位符；

（3）校验必需参数是否提供；

（4）执行查询并返回结果。

与相邻模块的关系：作为横切关注点，嵌入 NL 与直接 Cypher 两条入口之后、Neo4j 执行之前；与 Neo4j 驱动的超时设置共同构成“语句级 + 会话级”双保险。

（1）黑名单检查：使用正则表达式匹配，禁止CREATE、MERGE、DELETE、SET等写操作关键字；

（2）EXPLAIN预检（可选，默认关闭）：环境变量\texttt{ENABLE\_EXPLAIN\_VALIDATE=true}时，对查询执行\texttt{EXPLAIN}预检；生产或开发环境可按需开启。

\subsection{数据库设计}

本系统使用 Neo4j 图数据库存储《迷你世界》知识图谱，采用多标签属性图模型。为避免术语歧义，这里先说明本文中的两个领域词：方块指游戏世界中可采集或可放置的地形/材料单元，生物指可交互实体（如动物、怪物或 NPC）。数据库设计充分利用 Neo4j 的多标签特性，将方块和生物作为“道具”这一上位概念的子类，通过复合标签表达继承关系。

与上层模块的衔接：Schema（标签、关系类型）直接喂给 NL 模块的 Prompt；节点上的 \texttt{Name}、\texttt{MineTool} 等属性既影响 LLM 生成的 \texttt{WHERE} 子句，也影响转换模块如何归类节点颜色与表格列；因此数据建模是与查询体验、可视化效果联动的整体设计。

\subsubsection{架构演进}

早期曾考虑兼容阿里云 GDB（Graph Database）等不支持同一节点多标签的图引擎。在该约束下，无法直接使用 \texttt{:item:block}、\texttt{:item:monster} 这类复合标签表达“方块/生物是一种道具”的继承语义，只能将方块、生物与通用道具拆成不同节点，再通过业务 ID 用桥接关系（历史上曾建模为 PLACE、SUMMON 等）与道具节点相连。

该单标签桥接方案带来两类问题：（1）查询路径变长，自然语言转 Cypher 时模型需记住“先找道具再经 PLACE/SUMMON 跳到方块/生物”的固定套路，生成错误率显著上升；（2）图谱语义与玩家直觉不一致，人工理解与维护成本高。

当前实现已取消 GDB 兼容路径，数据仅面向 Neo4j 导入与查询：方块与生物直接采用多标签节点建模，不再使用 PLACE、SUMMON 等桥接关系。后文 Prompt 与测试用例均以此为准；论文中若仍偶见 PLACE/SUMMON 字样，均指上述已废弃的旧版建模，与现库无关。

\subsubsection{多标签架构设计}

Neo4j支持为一个节点设置多个标签（Multiple Labels），本系统利用这一特性设计继承关系：

\begin{itemize}
    \item 基础标签 :item - 所有游戏道具的父类，包含通用属性（ID, Name, Type, Disc, GetWay）
    \item 复合标签 :item:block - 方块类，继承:item所有属性，同时具有方块专属属性（MineTool, ToolLevel）
    \item 复合标签 :item:monster - 生物类，继承:item所有属性，同时具有生物专属属性（Life, Attack）
\end{itemize}

这种设计的优势：
\begin{enumerate}[(1)]
    \item 查询简化：相较旧版单标签桥接方案，无需经 PLACE/SUMMON 等关系跳转即可获取方块/生物完整属性，如`MATCH (n:item:block) RETURN n.Name, n.MineTool`
    \item 语义清晰：标签本身表达了类继承关系，符合领域模型
    \item 性能优化：避免了额外的JOIN操作，查询效率更高
\end{enumerate}

\subsection{自然语言查询模块的设计与实现}

设计要点。本模块负责将中文问句映射为参数化、只读、利于路径可视化的 Cypher。设计约束包括：（1）在调用 LLM 之前由后端拉取 Schema 摘要并注入提示词，使生成语句与当前库标签/关系类型对齐；（2）强制模型以 JSON 返回 \texttt{cql} 与 \texttt{params}，便于后续校验与 Neo4j 驱动绑定；（3）Prompt 中通过角色设定、多标签说明、Few-shot 与路径优先策略，降低“孤立节点返回”与“有向边漏匹配”概率，与第4章 NL 模块设计动机一致。

实现要点。对外入口为表~\ref{tab:api}中的 \texttt{POST /nlq}；实现上由 FastAPI 路由编排“取 Schema → 调 \texttt{llm\_client.generate\_cypher} → 走安全模块 → 调 Neo4j → 调转换器”的流水线。系统提示词可由仓库内 \texttt{system\_prompt.txt} 外置维护。以下给出请求/响应 JSON 形状及 Prompt 关键结构，便于与第4章时序图对照。

请求体格式：
\begin{lstlisting}[language=json, caption={自然语言查询请求体}]
{
    "query": "查找石剑",
    "options": {
        "limit": 100,
        "debug_raw": false
    }
}
\end{lstlisting}

响应体格式（字段名 \texttt{cql} 为历史命名，内容为Cypher字符串）：
\begin{lstlisting}[language=json, caption={自然语言查询响应体}]
{
    "cql": "MATCH (n:item) WHERE n.Name CONTAINS $name RETURN n",
    "params": {"name": "石剑"},
    "graph": {
        "nodes": [...],
        "links": [...],
        "categories": [...]
    },
    "table": {...}
}
\end{lstlisting}

Prompt 与生成策略。系统提示由 \texttt{system\_prompt.txt} 外置维护；可概括为：只读与角色约束、当前库 Schema 片段、多标签与路径式 \texttt{RETURN} 的说明、若干 Few-shot、以及要求 JSON 仅含 \texttt{cql} 与 \texttt{params}。为压缩正文，标签与关系类型完整清单、多组 Few-shot 与调参中的经验性说明见附录\ref{app:prompt-supplement}，与仓库及第4章时序图一致。

路径与返回形态：仅返回互不相连的节点时，驱动层可能缺少可绘边信息，故在提示中引导优先 \texttt{RETURN path}。无向子图模式 \texttt{-[R]-} 与有向子图在 Cypher 中不等价；在探索性问句中，无向可扩大匹配集合，亦可能增加无关边，属提示侧取舍。下面各举反例、次优与可接受示例各一。

错误示例（仅返回节点、不利恢复边）：
\begin{lstlisting}[language=cypher]
MATCH (r:recipe)-[:PRODUCES]->(p:item)
MATCH (r)-[:CONSUMES]->(c:item)
RETURN r, p, c
\end{lstlisting}

次优示例（有向路径，可能缩窄匹配）：
\begin{lstlisting}[language=cypher]
MATCH path = (material:item)<-[:CONSUMES]-(r:recipe)-[:PRODUCES]->(product:item)
RETURN path
\end{lstlisting}

可接受示例（无方向路径，与第4章设计一致）：
\begin{lstlisting}[language=cypher]
MATCH path = (material:item)-[:CONSUMES]-(r:recipe)-[:PRODUCES]-(product:item)
WHERE material.Name CONTAINS '木料'
RETURN path
\end{lstlisting}

典型 Few-shot（无方向；标题用语见附录\ref{app:prompt-supplement} 说明）：
\begin{lstlisting}[language=cypher, caption={合成问句与路径式返回（与附录中其余 few-shot 同型）}]
-- 木料能合成什么
MATCH path = (material:item)-[:CONSUMES]-(r:recipe)-[:PRODUCES]-(product:item)
WHERE material.Name CONTAINS '木料'
RETURN path LIMIT 20
\end{lstlisting}

LLM 调用与 JSON 解析（实现摘录）。自然语言模块将系统提示词（可由 \texttt{system\_prompt.txt} 外置）与用户问句拼装为 OpenAI 兼容的 \texttt{chat/completions} 请求，并强制 \texttt{response\_format} 为 JSON，便于提取 \texttt{cql} 与 \texttt{params}。

\begin{lstlisting}[language=Python, caption={LLM 生成 Cypher（llm\_client.py 摘录）}]
async def generate_cypher(self, nlq, schema_hint, limit):
    payload = {
        "model": self.model,
        "messages": [
            {"role": "system", "content": load_system_prompt()},
            {"role": "user", "content": f"只读 Cypher，LIMIT 默认 {limit}：\n{nlq}"},
        ],
        "temperature": 0.1,
        "response_format": {"type": "json_object"},
    }
    resp = await client.post(f"{self.api_base}/chat/completions", json=payload, headers=headers)
    obj = json.loads(resp.json()["choices"][0]["message"]["content"])
    return obj.get("cql", "").strip(), obj.get("params", {})
\end{lstlisting}

\subsection{图数据访问与 Cypher 执行模块的设计与实现}

设计要点。本模块将所有最终落库的 Cypher 执行收敛到单一 Neo4j 客户端封装：会话使用只读访问模式，查询带超时与数据库名配置，返回记录字典与列键序，供转换模块消费。\texttt{GET /schema} 为 NL 链路与人工浏览提供标签/关系摘要；\texttt{POST /run-cql} 为高级用户提供与 NL 链路相同的执行与安全语义；\texttt{GET /health} 便于部署探活。设计目标是在不复制驱动代码的前提下，使 NL 与直接 Cypher 两条入口共用第4章所述数据访问层契约。

实现要点。实现位于 \texttt{neo4j\_client.py} 等模块；路由层在通过 \texttt{cql\_validator} 校验后才调用 \texttt{run\_read}。下列代码摘录展示只读会话与超时换算，与第3章性能与非功能叙述一致。

\begin{lstlisting}[language=Python, caption={Neo4j 只读查询（neo4j\_client.py 摘录）}]
def run_read(self, cql: str, params: Dict[str, Any] | None = None):
    params = params or {}
    with self._driver.session(
        database=settings.NEO4J_DATABASE,
        default_access_mode="READ",
    ) as session:
        timeout_seconds = settings.QUERY_TIMEOUT_MS / 1000.0
        result = session.run(cql, parameters=params, timeout=timeout_seconds)
        records = [r.data() for r in result]
        return records, list(result.keys())
\end{lstlisting}

\subsection{结果转换与可视化模块的设计与实现}

设计要点。Neo4j 驱动返回的记录中可能同时包含节点、关系与路径对象；而 ECharts Graph 需要统一的 \texttt{nodes}/\texttt{links} 结构。设计上前端采用单页四区布局（输入、图谱、表格、生成语句回显），后端在 JSON 响应中并行返回 \texttt{graph} 与 \texttt{table}，使“看拓扑”与“读属性”互不替代。路径型 \texttt{RETURN path} 与无方向边模式由 NL 模块在 Prompt 侧引导，本模块负责忠实拆解驱动返回的结构，避免丢失边。

转换算法步骤。将 Neo4j 返回的记录转换为 ECharts Graph 格式的步骤为：（1）遍历查询结果中的每个记录；（2）识别节点（Node）和关系（Relationship）对象；（3）为每个节点生成唯一 ID，提取标签作为类别，属性作为节点数据；（4）为每条关系提取起始节点、目标节点和关系类型；（5）进行节点去重，处理孤儿节点。

实现要点。核心逻辑在 \texttt{echarts\_converter.py}；下列摘录展示关系边入库时如何保证端点节点先行入表。

\begin{lstlisting}[language=Python, caption={图数据转换（echarts\_converter.py 摘录）}]
def add_rel(rel: graph.Relationship) -> None:
    src, tgt = _node_id(rel.start_node), _node_id(rel.end_node)
    add_node(rel.start_node)
    add_node(rel.end_node)
    links.append({
        "source": src, "target": tgt,
        "category": rel.type, "label": rel.type,
        "value": dict(rel),
    })
\end{lstlisting}

\subsection{安全与约束模块的设计与实现}

设计要点。安全模块以横切方式作用于 \texttt{/nlq} 与 \texttt{/run-cql}：在调用 Neo4j 之前对语句做只读关键字黑名单匹配，可选 \texttt{EXPLAIN} 预检（由配置项控制，默认关闭以降低图库负载），并对 \texttt{\$param} 占位符与请求体参数做对齐检查。设计动机是：即便数据库账号配置为只读，仍应在应用层拒绝明显写操作与危险 \texttt{CALL dbms.*}，并向调用方返回可读错误，满足第3章非功能需求中的安全性条目。

实现要点。校验逻辑集中在 \texttt{cql\_validator.py}，由路由在执行业务前统一调用。下列黑名单与第6章表~\ref{tab:readonly-test} 中测试用例一一对应。

使用正则表达式黑名单匹配禁止的操作：

\begin{lstlisting}[language=Python, caption={只读校验黑名单（与 cql\_validator.py 一致）}]
WRITE_BLACKLIST = [
    r"\bCREATE\b",
    r"\bMERGE\b",
    r"\bDELETE\b",
    r"\bDETACH\s+DELETE\b",
    r"\bSET\b",
    r"\bREMOVE\b",
    r"\bDROP\s+(?:INDEX|CONSTRAINT|NODE\s+LABEL|REL\s+TYPE|PROPERTY)\b",
    r"\bLOAD\s+CSV\b",
    r"\bCALL\s+DBMS\.",
]
\end{lstlisting}

除上述关键字拦截外，系统未对 \texttt{CALL} 自定义过程（如 APOC 等插件过程）做白名单约束。部署假设为：图库以只读账号或只读副本访问、服务置于内网或经反向代理与鉴权后暴露，从而将可执行 Cypher 的攻击面与误用面限制在可信主体内；在不可信网络下若须开放，应另加网关层策略或扩展校验。

参数校验（实现说明）。使用正则表达式提取 Cypher 中的参数占位符 \texttt{\$param}，与请求体提供的参数对比，确保必需参数齐全（\texttt{/run-cql} 与 \texttt{/nlq} 共用逻辑，实现见源码树 \texttt{backend/app/} 下路由，路径以第\ref{subsec:repo-and-paths}~节为准）。

\subsubsection{pytest 自动化测试过程与样例断言}

本节说明自动化测试的执行步骤，并摘抄仓库 \texttt{tests/} 中与表~\ref{tab:readonly-test}、表~\ref{tab:api-test} 及图数据转换逻辑一一对应的实际断言代码。

（1）环境与命令。在仓库根目录安装依赖（参见 \texttt{README.md}）后，于终端执行：
\begin{verbatim}
python -m pytest tests/ -v
\end{verbatim}
\texttt{tests/conftest.py} 将 \texttt{backend/} 加入 \texttt{sys.path}，测试用例可直接 \texttt{import app.*}；API 测试通过 \texttt{FastAPI TestClient} 与 \texttt{unittest.mock.patch} 模拟 Neo4j、LLM 等外部依赖，无需启动真实 Neo4j 即可验证路由与安全策略。

（2）样例断言 A：只读校验器（摘录）。下列断言分别对应“合法只读通过”与“写操作被拦截”：

\begin{lstlisting}[language=Python, caption={tests/test\_cql\_validator.py 摘录}]
def test_simple_match_is_safe(self):
    cql = "MATCH (n) RETURN n LIMIT 10"
    ok, reason = is_readonly_cql(cql)
    assert ok is True
    assert reason is None

def test_create_is_blocked(self):
    cql = "CREATE (n:Person {name: 'Alice'})"
    ok, reason = is_readonly_cql(cql)
    assert ok is False
    assert "CREATE" in reason
\end{lstlisting}

（3）样例断言 B：API 契约（摘录）。健康检查与不安全 Cypher 被 \texttt{/run-cql} 拒绝的路径：

\begin{lstlisting}[language=Python, caption={tests/test\_api.py 摘录}]
def test_health_check(self, client):
    response = client.get("/health")
    assert response.status_code == 200
    assert response.json() == {"status": "ok"}

@patch("app.main.is_readonly_cql")
def test_run_unsafe_cql_blocked(self, mock_is_readonly, client):
    mock_is_readonly.return_value = (False, "包含写操作：CREATE")
    response = client.post("/run-cql", json={
        "cql": "CREATE (n:Person {name: 'Alice'})", "params": {}})
    assert response.status_code == 400
\end{lstlisting}

（4）样例断言 C：ECharts 转换（摘录）。验证 Neo4j 风格记录经 \texttt{records\_to\_graph} 后节点数与 \texttt{i:id} 主键格式：

\begin{lstlisting}[language=Python, caption={tests/test\_echarts\_converter.py 摘录}]
def test_basic_node_conversion(self, sample_records):
    nodes, links = records_to_graph(sample_records)
    assert len(nodes) == 3
    node_ids = {n["id"] for n in nodes}
    assert node_ids == {"i:1", "i:2", "i:3"}
\end{lstlisting}

\begin{thesisconclusion}

本研究设计并实现了自然语言知识图谱查询与可视化系统（neo4jSlave），在《迷你世界》属性图数据上作应用性验证。在\textbf{对照关系上}，可核对结果分两类、不可混读：第一，\texttt{pytest} 用例 38 条全通过，对应第3章与表~\ref{tab:req-map} 中可自动化验收的 API、校验与转换；第二，自然语言效果以第6章钉死环境与 \texttt{doc/paper/nlq\_eval\_protocol.md} 的协议、人工对 21 条问句作通过/不通过判定的\textit{一次性}走查为据，按类统计见表~\ref{tab:nlq-category}，\textit{在此同一框架下}可接受比例约 81.0\%。后一类比例不与公开 Text2Cypher 竞赛、亦不与未跑同一 21 条的模型作名次比较。第1.1 节中``不系统掌握 Cypher 也能在浏览器中探索图''的叙述，在工程上由（1）--（6）与上两类证据共同支撑，而不是由单一百分比单独支撑。

\subsection*{主要工作}

（1）系统架构。前后端分离的 B/S 结构；FastAPI 提供 REST 风格 API，静态前端为 HTML/CSS/JavaScript，关系图由 ECharts 渲染（与第2、4章技术选型一致）。

（2）自然语言到 Cypher。以 Schema 注入、JSON 约束输出与第5章所述提示结构，在通用大语言模型上产生只读、参数化 Cypher；与第6章中 Mock 的自动化及人工表相互独立地记录能力边界。

（3）安全与执行。与第5.4节、第6章只读与参数等表一致：关键字黑名单、可选 \texttt{EXPLAIN} 预检、\texttt{\$param} 与请求体对齐、超时与行数上限。

（4）数据与领域模型。多标签属性图与数据规模见表~\ref{tab:data-scale}、关系见第4章；为应用性验证数据，不单独主张“游戏领域”为方法创新。

\subsection*{可核对要点（与第6章对应，非推广性“创新点”）}

\begin{enumerate}[(1)]
    \item 可部署的 HTTP 全链路：自然语言与直接 Cypher 经同一安全与执行层，行为由第6章 API 与只读表可对照。
    \item Schema 约束的 NL2Cypher 与后处理：与仅报告模型指标、未给应用层实现细节的 Text2Cypher 工作相比，本研究将只读、参数、超时等纳入同一可运行服务，便于复现与审计（第5--6章）。
    \item 在固定协议下的人工走查：表~\ref{tab:nlq-category} 的 21 条、81.0\% 为协议内统计；中文 \texttt{Name CONTAINS} 等属数据与提示选择，可迁移性取决于目标图 Schema，不作未设对照基线的外推。
    \item 结果到 ECharts 的转换：\texttt{nodes}/\texttt{links} 形态与第6章转换器用例及界面截图可对照。
\end{enumerate}

\subsection*{不足与展望}

当前系统存在以下不足：
\begin{enumerate}[(1)]
    \item 复杂查询与分组、多实体对比在人工评测中仍有失误（表~\ref{tab:nlq-category}中“复杂查询”“关系查询”“边缘情况”等类为50.0\%；整体NLQ约81.0\%，距离稳健生产应用仍有差距）
    \item 缺乏同义词处理能力，用户对同一概念的不同表述无法正确识别
    \item 未实现用户认证和查询历史记录功能
\end{enumerate}

未来工作方向：
\begin{enumerate}[(1)]
    \item 引入检索增强生成（RAG）技术，结合向量数据库提升查询准确性
    \item 构建游戏领域同义词词典，支持语义等价表述
    \item 添加用户反馈机制，根据用户点击行为优化查询结果排序
    \item 支持多数据源融合，实现跨知识图谱的联合查询
\end{enumerate}

\end{thesisconclusion}
