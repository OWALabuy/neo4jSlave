\begin{abstract}
    知识图谱以图结构表示实体与关系，在知识管理、智能问答与推荐等任务中应用较广。在属性图数据库中，要取出``谁与谁以何种边相连''通常须书写 Cypher 等形式化语言：使用者需了解节点标签、关系有向、模式匹配与 \texttt{WHERE} 中属性键的选取，并在客户端理解表格式行集。上述可统称为面向终端用户的使用门槛。本研究面向的读者是希望在浏览器中探索已入库的领域知识、但不准备为单次查询而先系统掌握 Cypher 与图客户端操作的终端用户；在工程上``做好''的验收是：在第3章用户故事、第6章可对照样例与协议下，以中文问句经同一受控链路得到可读的表与点边图，并配套安全与可复现说明（细则见第3、6章，不在摘要展开）。

    本研究在浏览器/服务器结构下设计并实现系统 neo4jSlave。前端以中文提问；FastAPI 自 Neo4j 读取图结构摘要（即当前库中节点标签、关系类型等的简短清单，后文同）并与问句一并送入大语言模型，得到参数化只读 Cypher；经应用层只读与参数规则校验后由 Neo4j 执行。所谓只读 Cypher，指不修改图数据、不执行管理类调用的取数语句。结果以表格与 ECharts 点边图呈现，连接与端点可经环境变量配置。

    系统验证从两方面展开：自动化测试用于检查接口契约、只读校验和图数据转换等稳定功能；人工走查用于观察真实自然语言问句经模型生成查询后的可接受程度。结果显示，自动化用例均通过；在第6章固定的运行环境与判定协议下，21 条中文问句中约 81.0\% 可得到可接受结果。该结果用于说明本文系统在既定领域图数据上的端到端可用性，不作为公开 Text2Cypher 排行或多模型横向比较。可复现所需的公开仓库、路径约定与环境变量表见第2章，测试口径与分项结果见第6章。
\end{abstract}

\addcontentsline{toc}{section}{Abstract}
\begin{thesisabstracten}
Knowledge graphs represent entities and relationships in a graph-structured form and are widely used in information systems. In a property graph store, ad hoc questions such as ``how entities are linked'' are usually answered by Cypher: one must know labels, directed relationship types, pattern match syntax, and how to read tabular result rows. The \emph{usability gap} in this sense is the training and trial cost before a non-developer can write a first correct query. This work targets end users who wish to explore an ingested knowledge graph in a web browser and do not intend to invest in a full Cypher and desktop-client path for occasional queries. As an acceptance criterion, the delivered service should allow Chinese questions, under a documented and testable path, to yield readable tables and a node--link view under the same application-layer safety requirements as in Chapters~3 and~6. Details of user stories and test tables are in those chapters.

This work implements neo4jSlave, a browser--server system. The backend reads a \emph{schema summary} (a short list of current node labels and relationship types) from Neo4j, sends it with the question to a large language model, and receives parameterized \emph{read-only} Cypher; here \emph{read-only} means statements that do not modify graph data or call administrative procedures. The FastAPI layer validates read safety and parameters before execution. Configurable environment variables set endpoints and connection strings. Evaluation is organized in two parts: automated tests check API contracts, read-only validation, and graph conversion, while a manual walkthrough observes natural-language queries in the fixed environment described in Chapter~6. All automated cases passed, and 81.0\% of the 21 Chinese questions were accepted under the documented protocol. This result describes the end-to-end usability of the implemented system on the given graph data, rather than a leaderboard comparison for Text2Cypher models. A public source repository, relative path convention, environment table, and detailed test scope are provided in Chapters~2 and~6.
\end{thesisabstracten}

\section{系统概述}

\subsection{研究背景}

本课题要解决的问题可概括为三点：让不掌握图查询语言的人也能在浏览器中查询领域知识库，把查询结果用关系图和表格展示出来，并在对外提供服务时控制误写、误改数据的风险。

这里的``门槛''不是泛指界面不够友好，而是指用户在手写 Cypher 前必须完成的一组技术动作。以《迷你世界》类属性图为例，用户若想查询``查找石剑''或``木料能合成什么''，通常需要先知道实体对应的节点标签，再选择 \texttt{MATCH} 中的关系方向和跳数，并在 \texttt{WHERE} 中写出合适的属性条件。对非图数据库方向读者而言，这些步骤明显不同于普通网页检索。本研究希望把这组操作收束到浏览器中的中文问句入口，再由系统返回可读的表格和关系图。

知识图谱以有向图的方式表示实体和关系。实体是图中的顶点，关系是图中的边，顶点和边还可以附带属性。借助这种结构，系统可以回答“某个实体与哪些实体相连”“它们沿怎样的路径相连”等问题。知识图谱已在信息检索、推荐、金融风控和生物信息等领域得到应用。

工程实现中，领域数据常被整理成图数据库中的可查询图。属性图（Property Graph）是其中一种常见抽象，它把实体和关系都看成可带键值对属性的对象，并用声明式图查询语言描述模式匹配与路径。另一类路线是资源描述框架（RDF, Resource Description Framework）及其查询语言 SPARQL。RDF 适合开放数据互联与规范化语义网场景，属性图则更贴近面向工程实现的存储和产品迭代。二者没有绝对优劣，具体选择取决于数据形态、已有图实例和开发条件。

Neo4j 是常用的开源属性图数据库之一，其官方查询语言为 Cypher。Cypher 属于形式化查询语言，使用者需要了解标签、关系类型和路径匹配等概念。对许多只希望用日常语言查看关系图的终端用户而言，直接手写 Cypher 的学习成本仍然偏高。近年大语言模型（LLM, Large Language Model）推动了自然语言生成结构化查询的发展，自然语言到 SQL、Cypher 等任务已经具备工程落地基础。基于这一背景，本研究实现一个可说明依赖、可复现运行的 B/S（Browser/Server）服务，服务对象是以属性图组织、以中文描述为主的领域数据。可复现依赖、公开仓库地址与路径约定见第2章。

下文从知识图谱的建模与查询、自然语言到图查询、大语言模型与知识图谱协同三个方面归纳相关工作\cite{ref6,ref7,ref8,ref12,ref13,ref14,ref15}。这些内容分别对应后文的技术选型、系统设计和实现验证。现有文献与产品通常能够回答“图如何建”或“自然语言到 Cypher 的模型指标如何”，但较少把中文问句、受约束 Cypher、表格结果和关系图展示组织成一个可按文档复现的 Web 闭环。本文的工作即围绕这一闭环展开，并在第2--6章依次说明环境、需求、设计、实现和测试。

\subsection{国内外研究现状}

\subsubsection{知识图谱的数据模型、存储与查询基础}

从数据如何被写成“图”看，研究界与产业界已形成了若干可对照的范式。一类是以 RDF 与 SPARQL 为代表的三元组建模：把事实拆成大量三元组，便于在开放万维网上发布、链接和复用，并与本体、推理规则等机制配合；其代价往往是查询与维护需要熟悉较完整的语义网技术栈\cite{ref17}。另一类是属性图：在实体为节点、关系为边的图之上直接挂载属性，查询语言通常围绕模式匹配与路径展开。王鑫等从数据管理角度概括了图数据在存储、查询与知识融合等任务上的一类共性问题（如规模与索引、查询优化、事务等）\cite{ref1}。Robinson 等以图思维对比关系表建模\cite{ref3}；Miller 等从应用侧回顾了以 Neo4j 等为代表的图数据库用法\cite{ref2}。

在实例数据已具备可导入、可复现属性图形态、且应用层需以主流语言与 HTTP 服务对接时，Cypher 与 Neo4j 的文档与驱动支持成熟，Cypher 语义以官方手册为准\cite{ref12}。RDF/SPARQL 在强互联或 RDF 存量下更合适；在已有属性图、且交付以最少组件提供“可运行的 Web 查询”时，选择属性图与 Neo4j 是上述约束下的一种技术选型。但仅有图库仍不解决第1.1 节中的易用问题。易用问题在此特指：即便库内已有数据，只要入口仍是``手写 Cypher''，用户就仍须越过上一段所述的语法与 Schema 理解门槛；故自然语言接口在下一小节单独讨论。

\subsubsection{自然语言到结构化查询}

上一小节说明了“把领域数据存成可查询的图”的主流做法；但终端用户更常见的需求是用自然语言（NL, Natural Language）提问，而不愿学习形式化语言。为把讨论落到可对照的科研脉络上，先看关系型数据库中研究时间更长、文献更集中的文本到 SQL（Text-to-SQL）问题：其目标是把问句转写为可执行的 SQL（Structured Query Language）。Qin 等的综述将主流技术路线归纳到编码器—解码器、中间表示与预训练语言模型等框架下，并指出模式（Schema）感知与可执行性约束（保证生成可执行）是决定效果的关键因素\cite{ref5}。Text-to-SQL 可作为“自然语言到结构化取数”的一条常见参照，便于与后文至 Cypher 的任务对照。

再看图数据库场景。当底层存储是属性图、查询语言是 Cypher 时，与 Text-to-SQL 可类比的任务常被称作将自然语言映射为 Cypher 的 Text2Cypher：Ozsoy 等指出，即便借助大语言模型做零样本生成，在复杂语义与多跳约束上仍易产出不完整或错误语句；在公开高质量语料相对有限的前提下，用清洗、整合多源数据做微调，往往能在 双语互译评估（BLEU, Bilingual Evaluation Understudy）等指标上获得可观收益\cite{ref16}。这提示在工程上不能仅靠更换更强大的基座模型本身，还需要与当前库结构对齐的提示与约束，方能使生成结果稳定可执行。

在问答（QA, Question Answering）等相近任务中，从自然语言到 SPARQL 或到 SQL/其它逻辑形式，也长期存在解析或检索—排序等流水线\cite{ref5}。它们与图库自然语言查询的共同点在于：都必须在自然语言与符号查询之间的词汇鸿沟、组合式泛化、以及执行安全（只读、避免危险子串等）之间做权衡量化以外的工程取舍。本研究未在专用语料上训练新的 Text2Cypher 小模型，而在通用大语言模型与提示词约束的基础上，在生成之后叠加应用层只读校验、参数绑定与超时/行数等规则，以限制误执行风险，与当前常见“通用 LLM + 受控后处理”的落地方式一致\cite{ref4,ref9}。

\subsubsection{大语言模型、提示学习与知识图谱的融合}

承接上一小节，大语言模型除了用于“写查询”，还被广泛用于与知识图谱在知识增强、对齐、推理与可解释性等方向协同。Pan 等的路线图式综述指出，二者既可互相补充（图可为模型提供事实依托，模型可为图提供自然语言入口），也带来幻觉与结果不一致等风险，通常需借助检索、约束或外部验证抑制\cite{ref11}。在图数据库自然语言查询子场景中，常见分工是：大语言模型作问句到 Cypher 的非确定性（软）映射，Neo4j 作确定性的执行与可记录日志；中间层在文献与工程中多为只读、参数绑定、超时、结果量限制等，以控制部署风险\cite{ref4,ref9,ref11}。以数据集与自动评测为主的 Text2Cypher 工作较多\cite{ref16}，与在单一领域、多标签 Schema 下、把同一应用层同时接到 HTTP 接口与可交互关系图展示的论述相对独立。

综上，本文关注的研究缺口是：在既有属性图和通用大语言模型条件下，如何形成一个可在本地按文档启动的中文问句查询闭环。本文不引入新的图查询语言，也不以训练新模型为目标，而是让大语言模型生成参数化、只读 Cypher，再由应用层进行安全与参数校验，最后以 ECharts 关系图和表格输出结果。后文第3章说明需求，第4章说明设计，第5章说明实现，第6章说明测试与运行效果。

\subsection{问题描述与挑战}

贯穿全文的简例是``查找石剑''。不借助本系统时，用户需要打开 Cypher 客户端，参照 Schema 判断道具类标签，手写含 \texttt{Name} 条件的查询语句，再从表格式结果中辨认节点。借助本系统时，用户只需在浏览器中提交同一句中文问句，系统即按第5章所述链路生成查询、执行校验并返回表格与关系图。后文讨论模块和测试时，均以这一类问句说明系统替用户省去的操作步骤。

不掌握 Cypher 的终端用户难直接在手写 Cypher 路径下完成探索；仅依赖现场代写则难以稳定复现与自学。表格式中间结果不突出拓扑；在浏览器中呈现点与边，有利于把握“谁与谁以何种边相连”。上两项对应第1.1 节目标中的易用与可感知。

在引入大语言模型之后，还会叠加新的工程问题：模型可能生成非只读语句、遗漏与 Schema 绑定的参数，或在多标签的库结构上选到不理想的路径。因此，本系统不能只把“大语言模型接到字符串出口”，而必须在前后端中明确划分只读校验、参数与占位符、超时与结果量限制，并协调查询结果的形态，使其同时便于表格展示与关系图绘制。以上矛盾将在后文落实在自然语言模块、数据访问与执行模块、结果转换与可视化模块、以及安全与约束模块中分别展开。

\subsection{课题意义}

从实践意义看，本系统可作为领域知识库上的轻量查询入口。数据维护、内容编辑、教学或演示等场景中的用户，可通过浏览器访问系统，无需在本地安装数据库图形客户端，也无需记忆固定 Cypher 模板。从教学与复现意义看，论文后续章节将按“需求—设计—实现—测试”展开，并给出环境配置、应用程序接口（API, Application Programming Interface）约定与关键算法说明。具备 Python 与 Web 基础的读者，可在替换图数据和 LLM 端点后复现同类流程。

从技术意义看，本研究把自然语言到 Cypher 的受约束生成、应用层只读与参数规则、结果到关系图的转换放在同一条实现链中说明。多标签 Schema、提示词和取数形态之间的关系，将在第4章数据设计和第5章实现中展开。本文所说的``低门槛访问''，具体指用户不必先学会手写完整 Cypher 即可完成基本图查询；其在其他领域图数据上的适用性，取决于目标 Schema 与本文假设是否相近，以及部署环境是否满足同样的只读与接口约束。

\subsection{本文主要工作}

第6章的测试以第\ref{subsec:ch3-functional-req}~节所列功能为对象；本节仅给出四模块的划分及与后章的对应，避免与第3、4章逐条重复“原因—作用—优势”体例。

（1）自然语言查询模块（NL→Cypher）。职责：自 Neo4j 取 Schema 摘要、调用大语言模型、得到参数化只读 Cypher；详细流程与问题见第\ref{subsec:ch3-functional-req}~节、第4.3.1 与第5.1节。

（2）图数据访问与 Cypher 执行模块。职责：在只读、超时、统一参数语义下执行 Cypher，提供 \texttt{/run-cql}、\texttt{/schema} 等；见第\ref{subsec:ch3-functional-req}~节、第4.3.2 与第5.2节。

（3）结果转换与可视化模块。职责：将驱动返回的节点/关系/路径等转为 ECharts 与表结构；见第\ref{subsec:ch3-functional-req}~节、第4.3.4 与第5.3节。

（4）安全与约束模块。职责：在自然语言入口和直接 Cypher 入口前统一处理语句与参数。该模块保证三项规则：应用层已拦截的写类子串不进入数据库；\texttt{\$param} 与请求体中参数名一致时方可执行；单次查询在超时与行数上限内结束。说明见第4.3.3节与第5.4节。

第6章汇总自动化用例、人工问句与模块对应关系。

\section{系统开发环境与技术选型}

\subsection{开发环境、公开仓库与路径约定}
\label{subsec:repo-and-paths}

本系统在 GNU/Linux 环境下开发与调试，例如 Ubuntu 22.04 LTS 或 Debian 12。宿主机需安装可运行的 Neo4j 实例，既可连接本机服务（如 \texttt{bolt://127.0.0.1:7687}），也可连接局域网或云主机上的远程实例（如 \texttt{bolt://10.x.x.x:7687}）。浏览器建议使用近两年的主流版本，如 Chrome、Edge 或 Firefox 的 120 及以上版本，以保证前端图形交互稳定。

后端使用 Python 3.10 及以上版本，并通过虚拟环境管理第三方依赖。典型步骤为使用 \texttt{python -m venv .venv} 创建环境，再通过 \texttt{pip install -r requirements.txt} 安装 \texttt{fastapi}、\texttt{neo4j}、\texttt{httpx}、\texttt{pytest} 等包。前端不依赖 Node 构建链，采用原生 HTML/CSS/JavaScript，便于课程与答辩现场演示。版本控制使用 Git，日常编辑可使用任意文本编辑器或集成开发环境。

\subsection{技术路线与第1章问题的对应关系}

第1章把系统矛盾归纳为三方面：形式化查询门槛、查询结果的可感知性，以及生成语句在开放网络环境下的安全约束。第2章的技术路线即围绕这三点展开，而不是单纯罗列技术名称。大语言模型与提示词用于减轻手写 Cypher 的负担；ECharts 与第5章转换器用于把数据库行集转化为点边图；FastAPI 与只读校验用于在对外接口上拦截写类与危险调用。由此形成如下技术链路：

\begin{itemize}
    \item 大语言模型 + HTTP 客户端：大语言模型负责把中文问句映射为 Cypher；HTTP 客户端通过超文本传输安全协议（HTTPS, Hypertext Transfer Protocol Secure）调用 OpenAI 兼容接口，便于后续更换云厂商或模型端点。
    \item FastAPI：统一以表述性状态转移（REST, Representational State Transfer）风格的 Web API 对外暴露\cite{ref6,ref10}，同时承载自然语言入口和直接 Cypher 入口，并托管静态前端。
    \item Neo4j 官方 Python 驱动：保证与数据库协议一致，支持只读会话与查询超时。
    \item ECharts：将图查询结果映射为前端可渲染的数据结构，降低自研渲染引擎的成本。
\end{itemize}

\subsection{技术栈}

在给出按组件独立比较的总览后，本节按“后端语言与框架—数据库—模型服务—可视化—工具链”的顺序说明各技术项的职责与取舍理由，避免读者只见名词而不见关系。

\subsubsection{Python 3.10+}

Python 语法清晰，在数据与 Web 方向库齐全。本系统后端完全采用 Python，便于与 Neo4j 驱动、HTTP 客户端、pytest 测试框架在同一生态内集成。选择 3.10+ 是为了使用较新的类型标注与标准库特性，并与主流 Linux 发行版默认或软件源中的 Python 版本相匹配。

\subsubsection{FastAPI}

FastAPI 基于 Starlette 与 Pydantic，支持异步路由、自动 OpenAPI 文档与请求体验证。相较 Flask，其异步与类型提示更适合同时处理 LLM 网络请求与数据库 IO 的混合场景；相较 Django，其体量更轻，契合本课题“单一服务 + 静态前端”的部署形态。在本系统中的职责是：注册 \texttt{/nlq}、\texttt{/run-cql}、\texttt{/schema} 等路由，挂载静态资源，并在中间件层配置 CORS（演示环境宽松、生产环境应收紧，见第3章非功能需求）。

\subsubsection{Neo4j 图数据库}

Neo4j 使用 Cypher 与属性图模型，社区资料与示例丰富，且支持多标签节点——这与本文领域数据中“道具 / 方块 / 生物”的继承式建模一致（详见第4章）；其应用与概念讨论可见早期文献\cite{ref2}，产品与协议细节以官方资料为准\cite{ref7}。未首选 JanusGraph、ArangoDB 等的原因主要是：本课题已有 Neo4j 导入管线与实测数据规模，且 Cypher 与 LLM 训练语料中的“图查询”示例重合度较高，有利于提示词设计。

\subsubsection{大语言模型（LLM）}

系统通过 OpenAI 兼容的 HTTPS API 调用大模型，端点和模型名由 \texttt{LLM\_API\_BASE}、\texttt{LLM\_MODEL} 等环境变量配置。大模型提示学习与指令遵循能力的研究脉络见文献\cite{ref4}，商用 GPT-4 系列的技术报告见文献\cite{ref9}。在本系统中，大模型的作用是接收用户问句和 Schema 摘要，并返回包含 Cypher 与参数的 JSON。该方式无需训练专用模型即可迭代，但依赖网络与 API 调用，且生成结果必须经过后文安全模块过滤。

\subsubsection{ECharts}

ECharts 为 Apache 开源项目，关系图（Graph）组件成熟，支持力导向布局与交互。在本系统中负责将后端返回的 \texttt{nodes}/\texttt{links} 绘制为可拖拽、可缩放的图谱；与 D3.js 相比上手更快，与 Cytoscape 相比更贴近 Web 单页应用嵌入需求。

\subsubsection{其他工具}

\begin{itemize}
    \item Git：版本管理与协作。
    \item pytest：后端单元测试与 API 契约测试（LLM 可 Mock）。
    \item httpx：异步/同步 HTTP 客户端，用于调用 LLM API。
\end{itemize}

\subsection{系统架构与数据流}

本系统采用前后端分离的 B/S 架构，如图~\ref{fig:architecture}所示：

\begin{figure}[H]
    \centering
    \includegraphics[width=0.95\textwidth]{img/architecture.png}
    \caption{系统总体架构图}
    \label{fig:architecture}
\end{figure}

各层职责及相互关系可概括为一条主数据流：

\begin{enumerate}[(1)]
    \item 浏览器（前端层）：用户输入自然语言或 Cypher；将请求发往 FastAPI；接收 JSON 后由 ECharts 绘圈、由表格区展示属性。
    \item FastAPI（后端层）：对 \texttt{/nlq} 请求先拉取 Schema、再调 LLM 生成 Cypher；对两类执行请求均先做只读校验与参数检查，再调用 Neo4j。
    \item Neo4j（数据层）：在只读会话中执行 Cypher，返回记录流。
    \item 转换逻辑（后端层内）：将记录转为图与表结构，随响应返回前端。
\end{enumerate}

由此可见，自然语言模块、执行模块、安全模块、可视化模块并非孤立罗列：前两者串成“问句→语句→结果”，安全模块横切在执行之前，转换模块衔接数据库与前端展示。第4章将从设计角度进一步展开模块接口与数据库模型。

\subsection{可行性分析}

\subsubsection{技术可行性分析}

Web 框架、图数据库、经 HTTPS 调用的大语言模型和浏览器端可视化均已有成熟实现。FastAPI 与 Neo4j 官方 Python 驱动可在单机完成“启动服务、连接图库、返回 JSON”的基本链路；大语言模型经兼容接口接入，便于与具体厂商解耦；ECharts 可根据 JSON 绘制关系图。本仓库已提供后端、静态前端、\texttt{pytest} 用例与 \texttt{.env.example}，因此系统具备搭建和运行条件。自然语言问句效果受提示词与 Schema 影响，属于后续效果调优问题，不影响技术可行性判断。

\subsubsection{经济可行性分析}

开发期以人力与算力为主，运行期以图库存储与 LLM 调用量为主。Neo4j 社区版与相关 Python 依赖可零许可费使用，LLM 多按量计费；通过缓存 Schema 与第5章中 \texttt{QUERY\_HARD\_LIMIT}、\texttt{QUERY\_TIMEOUT\_MS} 等可限制单次查询成本。

\subsubsection{操作可行性分析}

B/S 架构下终端用户仅需浏览器，自然语言与直接 Cypher 可切换，与第4章单页、Schema 可见与语句回显设计一致。

\subsection{需求分析}

本系统的功能需求围绕第1.1节的问题展开。自然语言查询用于减少手写 Cypher 的要求；直接 Cypher 执行保留精确调试和教学对照通道；图谱可视化与表格展示共同呈现拓扑和属性；Schema 查询帮助用户和模型了解当前图库中的标签、关系和属性；安全限制则约束所有进入数据库的语句。下文按六项功能说明具体需求，表~\ref{tab:req-map} 进一步给出它们与第1.1节问题要点的对应关系。

\subsubsection{参与者与系统边界}

\subsubsection{功能需求}
\label{subsec:ch3-functional-req}

（1）自然语言查询。该功能是系统的主要入口。用户输入中文问句后，系统调用大语言模型生成参数化 Cypher，并在校验通过后执行。它依赖 Schema 查询提供标签和关系摘要，也依赖安全限制过滤写操作，执行结果再交由图谱和表格展示模块呈现。

（2）直接 Cypher 执行。自然语言入口适合终端用户，但调试、对比和教学演示仍需要可精确控制的语句通道。因此，系统保留 \texttt{/run-cql} 接口，使高级用户可提交只读 Cypher。该接口与自然语言入口共用同一套只读校验、参数校验和 Neo4j 会话，避免形成两套安全逻辑。

（3）图谱可视化。图数据的价值在于实体之间的拓扑关系，单纯表格难以表达“谁与谁相连”。系统需将查询结果中的节点、关系和路径映射为 ECharts Graph 可渲染的节点与边，使用户能够直接观察关系结构。

（4）数据表格展示。关系图适合观察结构，表格更适合阅读数值属性、长文本描述和具体字段。因此，系统应在同一响应中并行返回表格数据，使前端能够在图谱之外展示详细属性。

（5）Schema 查询。自然语言问句和模型生成语句都需要知道当前图库中有哪些标签、关系类型和关键属性。系统因此提供 \texttt{/schema} 等接口，供自然语言链路生成前调用，也供高级用户在前端查看。

（6）安全限制。系统向外暴露执行通道后，必须在访问数据库前拦截写操作、危险调用和参数不完整语句。安全限制横切自然语言查询和直接 Cypher 执行两条路径，并与超时、行数限制共同控制执行风险。

\subsubsection{非功能需求}

（1）安全性。即便数据库账号为只读，应用层仍应拒绝明显写语句，防止配置失误或 LLM 异常输出。

如何达成：正则黑名单 + 可选 EXPLAIN + 参数白名单式校验（见第4章安全模块与第5章实现）。

（2）性能。交互式查询对尾延迟敏感；LLM 已引入秒级延迟，图查询与渲染不应再放大瓶颈。

如何达成：Neo4j 会话超时、结果行硬上限、前端力导布局在合理节点规模内运行。

（3）可扩展性。课题结束后可能更换模型、迁移图实例或替换可视化方案。

如何达成：环境变量配置 URI 与 API；转换器与路由分层，避免将业务规则硬编码、固化于单一源文件中。

（4）易用性。主要受众不掌握 Cypher。易用性在此不归结为界面装饰性指标，而落实为：用户不必离开浏览器即可完成``提问—看生成句—看表与图—改问''的闭环；与（1）并列，共同构成第1.1 节中``自然语言在浏览器中查询并感知结果''的验收子集。

如何达成：单页、少步骤、生成语句可回看；错误信息返回可读提示（如缺参列表），减少``接口返回 500 而不知道为什么''的再次调试成本（该成本在评审视角下与``门槛''同构）。

（5）部署与网络策略。演示环境与生产环境风险不同。

如何达成：开发期宽松 CORS；上线前收紧为白名单并置于反向代理之后（与第1章非功能叙述一致）。

\subsection{性能分析}

（1）LLM API 响应时间。自然语言转 Cypher 通常占端到端延迟的主要部分（常见为 1–3 秒量级，视模型与网络而定）。

应对：不重复调用模型（单次请求携带 Schema）；失败时快速返回错误，避免阻塞线程池。

（2）Neo4j 查询性能。与模式复杂度、索引及数据量相关；本课题数据规模为万级节点、万级有向边，常见路径查询在毫秒至数百毫秒量级。

应对：限制 \texttt{LIMIT}、避免无界全图扫描类语句进入生产（由提示词引导与硬上限共同约束）。

（3）前端渲染性能。ECharts 在万级节点仍较流畅，但力导布局成本随边数上升。

应对：与（2）同一上限逻辑耦合——控制返回节点/边数量即控制渲染成本。

综上，性能目标通过限制规模、设置超时和优化提示词组合达成，而非依赖未经验证的无限扩容假设。

\subsection{本章小结}

本章从可行性、需求与性能三方面论证了系统的建设边界。技术上，Web 框架、图数据库、大语言模型接口和前端可视化可以组成可运行流水线；经济上，系统以开源组件和可配置的 LLM 调用为主；操作上，浏览器单页能够减少第1.1节所述的手写 Cypher 门槛。需求分析明确了六类功能和五项非功能要求，并通过用例图、表~\ref{tab:req-map} 与用户故事说明可验收行为。性能分析则把端到端延迟拆解为 LLM、Neo4j 与前端三个环节，并给出与第5章实现参数对应的控制思路。以上结论为第4章的系统结构与第5章的详细实现提供依据。

\section{系统设计}

\subsection{设计原则与模块关联总览}

第3章说明系统需要满足的功能边界，第4章进一步说明这些功能在结构上如何组织。读者即使不熟悉 Neo4j，也可先把握一条主线：自然语言查询和直接 Cypher 查询在进入 Neo4j 前都必须经过同一套只读、参数和超时策略；查询结果出库后，再由转换层整理为前端可绘制的 JSON。贯穿例子仍是中文问句``查找石剑''，其处理过程与第1.1节的问题描述和第6章测试样例保持一致。

系统设计遵循三条原则：

（1）职责分离——自然语言生成、语句校验、数据库访问、结果转换、前端渲染各有明确边界；

（2）横切安全——凡进入 Neo4j 的语句必经同一安全与参数策略，避免 NL 与直接 Cypher 两条路径不一致；

（3）可观测——生成语句与错误原因可返回给调用方，便于调试与论文复现。

从宏观关联看，五个模块并非并列孤立。自然语言模块产生查询意图，执行模块负责访问 Neo4j，安全模块位于执行之前，转换模块处理出库记录，可视化模块消费转换后的 JSON。图~\ref{fig:modules}给出模块划分，下文各小节均围绕这一数据与控制流顺序展开。

\subsection{系统总体架构}

系统采用分层架构，分为表示层、业务逻辑层、数据访问层和外部服务层。分层的目的在于明确边界：前端只处理交互和展示，业务层组织路由、自然语言编排与转换，数据访问层封装 Neo4j 会话，外部服务层处理 LLM 调用。这样既便于替换模型端点，也便于在测试中对 Neo4j 与 LLM 做模拟。

\begin{enumerate}[(1)]
    \item 表示层：前端 HTML 页面，使用 ECharts 进行图谱可视化，并通过 JavaScript 与后端 API 交互。该层仅依赖 JSON 响应，不感知 Neo4j 协议细节。
    \item 业务逻辑层：FastAPI 应用，包含 API 路由、自然语言编排、只读校验调用和数据转换。该层对下调用 Neo4j 客户端与 httpx LLM 客户端，对上统一返回 HTTP 与 JSON。
    \item 数据访问层：Neo4j 客户端，负责会话、只读模式、数据库名和超时设置。驱动代码集中在该层，可避免路由处理函数直接散落数据库访问细节。
    \item 外部服务层：大语言模型 API。模型服务的可用性、配额和延迟不由本系统完全控制，因此在架构上将其视为外部依赖。
\end{enumerate}

\subsubsection{自然语言查询模块}

自然语言查询模块负责把中文问句转化为可执行的 Cypher，是面向终端用户的核心入口。该模块不能单独工作：它需要先从数据访问层获取 Schema 摘要，再把 Schema 与问句组装为提示词并调用外部 LLM；模型返回的语句必须进入安全校验和执行模块，出库记录再交给转换与可视化模块。主要流程如图~\ref{fig:nlq-flow}所示：

（1）用户输入自然语言查询；

（2）系统获取当前知识图谱的Schema信息（节点标签、关系类型）；

（3）构建提示词，将 Schema 信息和用户查询发送给 LLM；

（4）LLM生成Cypher查询语句和参数；

（5）系统进行安全校验（只读关键字黑名单；可选的EXPLAIN预检，默认关闭，见\texttt{ENABLE\_EXPLAIN\_VALIDATE}）；

（6）执行查询，获取结果；

（7）将结果转换为ECharts图数据或表格数据返回。

为提高生成结果的可用性，提示词采用系统约束与少量示例，重点说明多标签 Schema、路径型 \texttt{RETURN path} 和参数化占位符。这样可以引导模型生成更利于前端绘边的只读查询。自然语言到结构化查询（如 SQL）的深度学习综述可与“NL→Cypher”任务对照阅读\cite{ref5}。模型输出最终仍须经过黑名单与参数校验，生成成功不等于可以直接执行。

    \caption{自然语言查询时序图}

\subsubsection{Cypher执行模块}

Cypher 执行模块服务于调试、教学演示和已知正确语句验证等场景。它允许高级用户绕过 LLM，直接提交只读 Cypher。为避免自然语言入口和直接入口形成两套安全策略，该模块与自然语言模块共用 Neo4j 客户端、只读校验和参数校验逻辑。查询成功后，结果同样进入转换模块，由前端按相同方式展示。其执行过程包括：

（1）解析Cypher语句，进行只读安全检查；

（2）提取查询中的参数占位符；

（3）校验必需参数是否提供；

（4）执行查询并返回结果。

\subsubsection{安全校验模块}

安全校验模块位于自然语言和直接 Cypher 两条入口之后、Neo4j 执行之前。LLM 输出和用户输入都不能默认可信，即使数据库账号配置为只读，应用层仍应拒绝明显写操作和危险调用。该模块与 Neo4j 驱动的超时设置共同形成“语句级 + 会话级”的约束。安全流程如图~\ref{fig:security-flow}所示：

\begin{figure}[H]
    \centering
    \includegraphics[width=0.8\textwidth]{img/security-flow.png}
    \caption{安全校验流程图}
    \label{fig:security-flow}
\end{figure}

安全机制包括以下四项：

（1）黑名单检查：使用正则表达式匹配，禁止CREATE、MERGE、DELETE、SET等写操作关键字；

（2）EXPLAIN预检（可选，默认关闭）：环境变量\texttt{ENABLE\_EXPLAIN\_VALIDATE=true}时，对查询执行\texttt{EXPLAIN}预检；生产或开发环境可按需开启。

（3）参数校验：确保所有参数都被正确提供，防止注入攻击；

（4）超时限制：通过Neo4j驱动为单次查询设置超时（\texttt{QUERY\_TIMEOUT\_MS}），减轻长时间查询对服务的占用。

\subsubsection{可视化模块}

可视化模块解决的是数据库返回对象与浏览器展示结构之间的差异。Neo4j 返回的记录可能包含节点、关系或路径，而 ECharts 需要明确的 \texttt{nodes}/\texttt{links} 结构。系统将转换逻辑集中在后端，前端只消费约定好的 \texttt{graph}/\texttt{table} 字段，从而减少页面中的适配代码，也便于测试转换结果。图数据转换流程如图~\ref{fig:data-convert}所示。

\subsection{数据库设计}

本系统使用 Neo4j 图数据库存储《迷你世界》知识图谱，采用多标签属性图模型。为避免术语歧义，这里先说明本文中的两个领域词：方块指游戏世界中可采集或可放置的地形/材料单元，生物指可交互实体（如动物、怪物或 NPC）。数据库设计充分利用 Neo4j 的多标签特性，将方块和生物作为“道具”这一上位概念的子类，通过复合标签表达继承关系。

数据库设计直接影响自然语言查询和可视化效果。Schema 中的标签与关系类型会进入自然语言模块的提示词，节点上的 \texttt{Name}、\texttt{MineTool} 等属性会影响模型生成 \texttt{WHERE} 条件，也会影响转换模块如何归类节点和生成表格列。因此，数据建模不是孤立的存储设计，而是与查询体验和展示效果联动的整体设计。

\subsubsection{架构演进}

早期曾考虑兼容阿里云 GDB（Graph Database）等不支持同一节点多标签的图引擎。在该约束下，无法直接使用 \texttt{:item:block}、\texttt{:item:monster} 这类复合标签表达“方块/生物是一种道具”的继承语义，只能将方块、生物与通用道具拆成不同节点，再通过业务 ID 用桥接关系（历史上曾建模为 PLACE、SUMMON 等）与道具节点相连。

该单标签桥接方案会拉长查询路径。自然语言转 Cypher 时，模型需要记住“先找道具，再经 PLACE 或 SUMMON 跳到方块/生物”的固定模式，生成错误的概率随之上升。同时，这种建模与用户对“方块和生物也是道具类别”的直观理解不一致，维护成本较高。

当前实现已取消 GDB 兼容路径，数据仅面向 Neo4j 导入与查询。方块与生物直接采用多标签节点建模，不再使用 PLACE、SUMMON 等桥接关系。后文提示词与测试用例均以当前多标签模型为准。

\subsubsection{多标签架构设计}

Neo4j支持为一个节点设置多个标签（Multiple Labels），本系统利用这一特性设计继承关系：

\begin{itemize}
    \item 基础标签 :item - 所有游戏道具的父类，包含通用属性（ID, Name, Type, Disc, GetWay）
    \item 复合标签 :item:block - 方块类，继承:item所有属性，同时具有方块专属属性（MineTool, ToolLevel）
    \item 复合标签 :item:monster - 生物类，继承:item所有属性，同时具有生物专属属性（Life, Attack）
\end{itemize}

这种设计带来三方面优势：
\begin{enumerate}[(1)]
    \item 查询简化：相较旧版单标签桥接方案，无需经 PLACE/SUMMON 等关系跳转即可获取方块/生物完整属性，如 \texttt{MATCH (n:item:block) RETURN n.Name, n.MineTool}。
    \item 语义清晰：标签本身表达了类继承关系，符合领域模型
    \item 性能优化：避免了额外的JOIN操作，查询效率更高
\end{enumerate}

\subsection{自然语言查询模块的设计与实现}

自然语言查询模块负责将中文问句映射为参数化、只读、便于路径可视化的 Cypher。为使模型生成结果与当前图库一致，后端在调用 LLM 前先拉取 Schema 摘要，并将其与用户问句共同放入提示词。模型返回结果被约束为 JSON，主要字段为 \texttt{cql} 与 \texttt{params}，便于后续安全校验和 Neo4j 参数绑定。

对外入口为表~\ref{tab:api}中的 \texttt{POST /nlq}。实现上，FastAPI 路由编排“取 Schema、调用 \texttt{llm\_client.generate\_cypher}、进入安全模块、访问 Neo4j、调用转换器”的流水线。以下给出请求和响应形状，便于与第4章时序图对照。

请求体格式：
\begin{lstlisting}[language=json, caption={自然语言查询请求体}]
{
    "query": "查找石剑",
    "options": {
        "limit": 100,
        "debug_raw": false
    }
}
\end{lstlisting}

响应体格式（字段名 \texttt{cql} 为历史命名，内容为Cypher字符串）：
\begin{lstlisting}[language=json, caption={自然语言查询响应体}]
{
    "cql": "MATCH (n:item) WHERE n.Name CONTAINS $name RETURN n",
    "params": {"name": "石剑"},
    "graph": {
        "nodes": [...],
        "links": [...],
        "categories": [...]
    },
    "table": {...}
}
\end{lstlisting}

提示词与生成策略可概括为四点：限定只读查询，注入当前库 Schema，说明多标签与路径式 \texttt{RETURN path} 的使用方式，并要求输出 JSON。仅返回孤立节点时，前端难以恢复边信息，因此提示词倾向于引导模型返回路径。完整标签、关系类型、多组示例与调参说明见附录\ref{app:prompt-supplement}。正文仅保留一个代表性示例：
\begin{lstlisting}[language=cypher, caption={合成问句与路径式返回（与附录中其余 few-shot 同型）}]
-- 木料能合成什么
MATCH path = (material:item)-[:CONSUMES]-(r:recipe)-[:PRODUCES]-(product:item)
WHERE material.Name CONTAINS '木料'
RETURN path LIMIT 20
\end{lstlisting}

LLM 调用与 JSON 解析的实现摘录如下。自然语言模块将系统提示词与用户问句拼装为 OpenAI 兼容的 \texttt{chat/completions} 请求，并要求响应为 JSON，以便提取 \texttt{cql} 与 \texttt{params}。

\begin{lstlisting}[language=Python, caption={LLM 生成 Cypher（llm\_client.py 摘录）}]
async def generate_cypher(self, nlq, schema_hint, limit):
    payload = {
        "model": self.model,
        "messages": [
            {"role": "system", "content": load_system_prompt()},
            {"role": "user", "content": f"只读 Cypher，LIMIT 默认 {limit}：\n{nlq}"},
        ],
        "temperature": 0.1,
        "response_format": {"type": "json_object"},
    }
    resp = await client.post(f"{self.api_base}/chat/completions", json=payload, headers=headers)
    obj = json.loads(resp.json()["choices"][0]["message"]["content"])
    return obj.get("cql", "").strip(), obj.get("params", {})
\end{lstlisting}

\subsection{图数据访问与 Cypher 执行模块的设计与实现}

图数据访问与 Cypher 执行模块将最终进入数据库的查询收敛到统一的 Neo4j 客户端封装。会话使用只读访问模式，查询带数据库名和超时配置，返回记录字典与列键序，供转换模块继续处理。\texttt{GET /schema} 为自然语言链路和人工浏览提供标签、关系摘要；\texttt{POST /run-cql} 为高级用户提供与自然语言入口相同的执行语义；\texttt{GET /health} 用于部署探活。路由层在通过 \texttt{cql\_validator} 校验后才调用 \texttt{run\_read}。下列代码摘录展示只读会话与超时换算。

\begin{lstlisting}[language=Python, caption={Neo4j 只读查询（neo4j\_client.py 摘录）}]
def run_read(self, cql: str, params: Dict[str, Any] | None = None):
    params = params or {}
    with self._driver.session(
        database=settings.NEO4J_DATABASE,
        default_access_mode="READ",
    ) as session:
        timeout_seconds = settings.QUERY_TIMEOUT_MS / 1000.0
        result = session.run(cql, parameters=params, timeout=timeout_seconds)
        records = [r.data() for r in result]
        return records, list(result.keys())
\end{lstlisting}

def add_rel(rel: graph.Relationship) -> None:

\subsection{安全与约束模块的设计与实现}

安全模块以横切方式作用于 \texttt{/nlq} 与 \texttt{/run-cql}。在调用 Neo4j 前，系统先对语句做只读关键字黑名单匹配，再检查 \texttt{\$param} 占位符与请求体参数是否一致；可选 \texttt{EXPLAIN} 预检由配置项控制，默认关闭以降低图库负载。即使数据库账号配置为只读，应用层仍应拒绝明显写操作与危险 \texttt{CALL dbms.*}，并向调用方返回可读错误。校验逻辑集中在 \texttt{cql\_validator.py}，下列黑名单与第6章表~\ref{tab:readonly-test} 中测试用例对应：

\begin{lstlisting}[language=Python, caption={只读校验黑名单（与 cql\_validator.py 一致）}]
WRITE_BLACKLIST = [
    r"\bCREATE\b",
    r"\bMERGE\b",
    r"\bDELETE\b",
    r"\bDETACH\s+DELETE\b",
    r"\bSET\b",
    r"\bREMOVE\b",
    r"\bDROP\s+(?:INDEX|CONSTRAINT|NODE\s+LABEL|REL\s+TYPE|PROPERTY)\b",
    r"\bLOAD\s+CSV\b",
    r"\bCALL\s+DBMS\.",
]
\end{lstlisting}

除上述关键字拦截外，系统未对 \texttt{CALL} 自定义过程（如 APOC 等插件过程）做白名单约束。部署假设为：图库以只读账号或只读副本访问、服务置于内网或经反向代理与鉴权后暴露，从而将可执行 Cypher 的攻击面与误用面限制在可信主体内；在不可信网络下若须开放，应另加网关层策略或扩展校验。

参数校验（实现说明）。使用正则表达式提取 Cypher 中的参数占位符 \texttt{\$param}，与请求体提供的参数对比，确保必需参数齐全（\texttt{/run-cql} 与 \texttt{/nlq} 共用逻辑，实现见源码树 \texttt{backend/app/} 下路由，路径以第\ref{subsec:repo-and-paths}~节为准）。

\subsection{本章小结}

本章按四个核心模块给出了详细设计与关键实现：自然语言模块给出策略、请求/响应形状与实现摘录，提示词分块与多组 Few-shot 见附录\ref{app:prompt-supplement}；图数据访问与执行模块统一了只读会话、超时和端点契约；结果转换与可视化模块明确了记录到 \texttt{nodes}/\texttt{links} 的映射过程；安全与约束模块以前置校验方式拦截高风险语句并检查参数完整性。上述实现细节与第4章设计一一对应，并为第6章按模块组织测试提供直接依据。

从统计结果看，道具查找、合成查询、掉落查询和生物查询表现较稳定，这说明多标签 Schema 与路径型提示对常见探索问句具有一定适配性。失败主要集中在方块细分问法、分组查询、复杂查询和边缘问句中。其原因通常不是接口不可用，而是问句中的自然语言表达与图库属性键之间存在差异。例如，用户以中文分组名发问时，模型可能直接匹配 \texttt{:group} 的数值型 \texttt{ID}；又如，多实体对比或依赖 \texttt{StationLabel} 的问句，容易生成过严或不合法的条件。上述问题说明，后续优化应重点加强实体映射、属性说明和复杂问句拆解。

\begin{thesisconclusion}

本研究设计并实现了自然语言知识图谱查询与可视化系统（neo4jSlave），并在《迷你世界》属性图数据上完成应用性验证。系统面向不准备系统学习 Cypher 的终端用户，提供中文问句入口、受约束的图数据库查询执行、表格与关系图展示等功能。

验证结果由两类证据组成。自动化测试覆盖 API 契约、只读校验、参数检查与图数据转换，38 个 \texttt{pytest} 用例均通过。自然语言效果在第6章固定的模型、图数据和人工判定协议下统计，21 条中文问句中约 81.0\% 可得到可接受结果。上述结果说明系统在既定数据与配置下形成了可运行、可检查的 HTTP 全链路；其自然语言比例不用于与公开 Text2Cypher 基准或未采用同一问句集的模型作排名比较。

\subsection*{主要工作}

（1）系统架构。系统采用前后端分离的 B/S 结构。FastAPI 提供 REST 风格 API，静态前端由 HTML/CSS/JavaScript 实现，关系图由 ECharts 渲染。

（2）自然语言到 Cypher。系统在提示词中注入 Schema 摘要，并要求大语言模型输出 JSON 形式的只读、参数化 Cypher，使自然语言问句能够进入后续校验与执行流程。

（3）安全与执行。系统在执行前统一进行关键字黑名单检查、可选 \texttt{EXPLAIN} 预检、参数占位符校验、超时和行数限制，使自然语言入口与直接 Cypher 入口遵循同一执行规则。

（4）数据与领域模型。系统采用 Neo4j 多标签属性图组织领域数据，节点、关系与数据规模见第4章。该数据用于应用性验证，重点在于说明自然语言查询、受控执行与可视化展示的完整流程。

\subsection*{可核对要点（与第6章对应，非推广性“创新点”）}

\begin{enumerate}[(1)]
    \item 可部署的 HTTP 全链路：自然语言与直接 Cypher 经同一安全与执行层，行为由第6章 API 与只读表可对照。
    \item Schema 约束的 NL2Cypher 与后处理：与仅报告模型指标、未给应用层实现细节的 Text2Cypher 工作相比，本研究将只读、参数、超时等纳入同一可运行服务，便于复现与审计（第5--6章）。
    \item 在固定协议下的人工走查：表~\ref{tab:nlq-category} 的 21 条、81.0\% 为协议内统计；中文 \texttt{Name CONTAINS} 等属数据与提示选择，可迁移性取决于目标图 Schema 和部署约束。
    \item 结果到 ECharts 的转换：\texttt{nodes}/\texttt{links} 形态与第6章转换器用例及界面截图可对照。
\end{enumerate}

\subsection*{不足与展望}

当前系统仍有以下不足：
\begin{enumerate}[(1)]
    \item 复杂查询、分组查询和多实体对比仍有失败情形，说明当前提示与 Schema 描述对复杂语义的约束能力有限。
    \item 同义词和近义表达处理不足，用户使用不同名称指代同一概念时，系统仍可能无法稳定匹配。
    \item 权限控制仍停留在应用层只读校验和部署假设层面，尚未形成完整的用户身份认证与访问控制机制。
\end{enumerate}

未来工作方向：
\begin{enumerate}[(1)]
    \item 引入检索增强生成（RAG）或实体链接方法，提高问句中实体、属性和关系类型的匹配准确性。
    \item 构建领域同义词与别名词表，使语义等价表述能够映射到同一图数据库对象。
    \item 完善身份认证、访问控制和审计日志，使系统更适合在非演示环境中部署。
    \item 研究多源图数据适配方法，使同一自然语言查询流程能够迁移到结构相近的其他领域知识图谱。
\end{enumerate}

\end{thesisconclusion}
