\begin{abstract}
    知识图谱以图结构表示实体与关系，在知识管理、智能问答与推荐等任务中应用较广。在属性图数据库中，若要查明实体之间如何经由边相连，通常须书写 Cypher：使用者需了解节点标签、关系方向、模式匹配与属性条件，并在客户端阅读表格式结果。上述步骤构成面向终端用户的使用门槛。本研究面向希望在浏览器中浏览领域知识、但不打算为偶尔几次查询而系统学习 Cypher 与专用客户端的读者。

    本文在浏览器/服务器结构下实现系统 neo4jSlave：用户以中文提问；后端从 Neo4j 读取当前库的图结构摘要（节点标签、关系类型等的简短清单），与问句一并送入大语言模型，得到参数化、只读 Cypher；经应用层安全与参数校验后执行。只读 Cypher 指不修改图数据、不调用管理类过程的查询语句。查询结果以表格与基于 ECharts（一种在浏览器中绘制图表的常用开源库）的点边关系图呈现；服务端地址与密钥等通过环境变量配置（即在操作系统或部署环境中设置的命名参数，不写入论文正文）。

    验证包括自动化测试与人工走查：前者检验接口约定（即请求应携带哪些字段、在不同情形下返回何种状态码与数据结构）、只读校验与结果转换；后者在固定环境与判定协议下考察 21 条中文问句的端到端效果。自动化用例全部通过；人工走查约 81.0\% 的问句得到可接受结果。该比例用于说明系统在既定领域数据上的可用性，不作为公开 Text2Cypher 榜单或多模型对比结论。源代码、路径约定与环境变量说明随正文给出；复现与测试细则亦在正文中交代。
\end{abstract}

\addcontentsline{toc}{section}{Abstract}
\begin{thesisabstracten}
Knowledge graphs represent entities and relationships in a graph-structured form and are widely used in information systems. In a property graph store, questions about how entities are linked are usually answered by Cypher, which presupposes familiarity with labels, relationship types, pattern syntax, and tabular results. That preparation cost is the main usability gap for occasional, non-expert users. This work targets readers who want to browse an ingested graph in a web browser without learning Cypher and a desktop client first.

We implement neo4jSlave, a browser--server system. The backend reads a \emph{schema summary} (current node labels and relationship types) from Neo4j, sends it with the Chinese question to a large language model, and receives parameterized \emph{read-only} Cypher; \emph{read-only} means statements that do not modify graph data or call administrative procedures. The FastAPI layer validates safety and parameters before execution. Endpoints and secrets are set via environment variables. Results are shown as tables and an ECharts node--link view.

Evaluation combines automated tests (API contracts, read-only checks, graph conversion) and a manual walkthrough of 21 Chinese questions under a fixed protocol and runtime. All automated tests passed; 81.0\% of the manual questions were judged acceptable. This figure describes end-to-end usability on the chosen dataset, not a public Text2Cypher benchmark ranking. The repository URL, path layout, environment variables, and test protocol are documented in the thesis body.
\end{thesisabstracten}

\section{系统概述}

\subsection{研究背景}

本章围绕一个核心问题展开：在不要求终端用户系统学习 Cypher 的前提下，如何让其通过浏览器完成知识图谱查询，并获得可理解、可核对且可复现的结果。本文将该问题拆分为三个层面：查询入口的使用门槛、查询结果的可感知性，以及对外服务时的执行安全。

本文所称“使用门槛”并非界面易用程度，而是用户在手动编写 Cypher 前必须完成的技术准备。以《迷你世界》属性图为例，若用户希望查询``查找石剑''或``木料能合成什么''，通常需先掌握节点标签、关系方向、模式匹配语法与属性条件写法，再理解返回的表格式记录。对于非图数据库方向读者，这一路径的学习成本较高。本文尝试将该成本前移到系统侧：用户仅提交中文问句，系统返回表格与关系图。

知识图谱以有向图描述实体及其关系，适合回答“实体如何连接”“路径如何构成”等问题。工程落地时，常见存储路线包括属性图与 RDF。属性图（Property Graph）将节点和关系统一为可附带键值属性的对象，便于面向业务迭代；RDF（Resource Description Framework）及其查询语言 SPARQL 更适用于开放数据互联与规范化语义网场景。二者并无绝对优劣，选择取决于数据形态、既有基础和交付目标。

在本文场景中，数据以属性图组织，数据库采用 Neo4j，查询语言采用 Cypher。近年大语言模型（Large Language Model, LLM）推动了自然语言到结构化查询任务的发展，使“中文问句到 Cypher”具备工程实现条件。本文据此实现一个 B/S（Browser/Server，浏览器/服务器）服务：用户侧通过浏览器发起请求并查看结果，业务逻辑与数据访问集中在服务端程序中完成。在保证可复现部署与执行约束的前提下，把中文问句、受约束 Cypher、表格结果与关系图展示整合为同一条可运行链路。

为支撑上述目标，下文先综述知识图谱建模与查询、自然语言到结构化查询、大语言模型与知识图谱协同三类研究\cite{ref6,ref7,ref8,ref12,ref13,ref14,ref15}，再给出本文系统的需求、设计、实现与测试。

\subsection{国内外研究现状}

\subsubsection{知识图谱的数据模型、存储与查询基础}

知识图谱的数据组织与查询主要有两类代表性路线。第一类是 RDF 与 SPARQL：该路线将事实表示为三元组，便于开放发布、互联和复用，并可结合本体与推理机制\cite{ref17}。第二类是属性图：该路线以节点和关系表示实体结构，并在节点或关系上直接附加属性，查询语言通常围绕模式匹配与路径展开。已有研究从数据管理、图建模与应用实践等角度讨论了图数据在存储、索引、查询优化和事务控制上的关键问题\cite{ref1,ref2,ref3}。

若项目已有可导入的属性图数据，且目标是以较少组件交付可运行的 Web 查询服务，则 Neo4j 与 Cypher 在文档、驱动与工程生态上具备较高成熟度\cite{ref12}。但仅完成“数据入库 + 语句可执行”并不足以解决终端用户可用性问题：当入口仍是手动编写 Cypher 时，用户仍需先掌握语法与当前库的 Schema（Schema 在此指当前图数据库中节点类型、关系类型及关键属性等结构信息，而非泛指任何“概要”）。该缺口是下一小节讨论自然语言接口的直接动因。

\subsubsection{自然语言到结构化查询}

终端用户更常见的需求是直接使用自然语言（Natural Language, NL）提问，而非学习形式化查询语法。相关研究中，文本到 SQL（Text-to-SQL）问题发展较早，可作为对照：其目标是将问句转写为可执行 SQL。综述研究显示，生成语句时需知晓数据库当前有哪些表与字段（在文献中常概括为 Schema 感知，即在生成前掌握结构层面的先验信息），并保证语句可被数据库执行（可执行约束），二者是影响生成质量的关键因素\cite{ref5}。这一结论对自然语言到 Cypher 任务同样具有参考价值。

在属性图场景下，相关任务通常被称为 Text2Cypher。已有研究指出，在不针对本任务专门训练模型、仅靠通用模型直接生成语句（常称为零样本生成）时，复杂语义、多跳约束和细粒度条件组合仍易导致错误；在高质量标注语料有限时，通过清洗数据并用标注样本进一步训练模型（有监督微调）往往能提高评测指标\cite{ref16}。这意味着工程系统不能仅依赖“更大模型”本身，还需要结合当前 Schema 的提示约束与执行后处理。

更广义地看，无论是自然语言到 SQL、SPARQL 还是 Cypher，核心难点都在于自然语言与符号查询之间的映射不确定性，以及执行安全约束。本文不训练专用 Text2Cypher 模型，而采用通用大语言模型配合受控后处理：生成后继续施加只读校验、将问句中的取值与查询参数对应绑定（参数绑定）、查询超时与单次返回结果条数上限等约束，以降低误执行或一次性拉取过量数据的风险\cite{ref4,ref9}。

\subsubsection{大语言模型、提示学习与知识图谱的融合}

大语言模型与知识图谱的协同研究已覆盖利用图结构补充模型知识、统一符号与向量表示、辅助推理与增强可解释性等方向\cite{ref11}。在自然语言查询子场景中，二者分工较清晰：大语言模型负责从问句生成候选查询，图数据库负责确定性执行并返回结构化结果；应用层负责执行前后的安全约束和结果整形。该结构能够兼顾灵活性与可控性，但也面临模型生成与事实不符的内容（常称为幻觉）、同一问句多次生成不一致语句、以及访问权限被滥用等风险，需要通过检索外部资料、规则约束与外部验证机制加以抑制\cite{ref4,ref9,ref11}。

据此，本文聚焦的研究缺口是：在既有属性图与通用大语言模型条件下，如何形成一个可按文档复现的中文问句查询闭环。本文不提出新查询语言，也不训练新模型，而是采用“受约束生成 + 应用层校验 + 图表双通道展示”的实现路径。

\subsection{问题描述与挑战}

贯穿全文的示例问句为``查找石剑''。在传统路径下，用户需打开 Cypher 客户端、依据当前库的 Schema（节点类型、关系类型等结构信息）判断标签与关系、手动写出查询条件，再从表格结果中辨认目标节点；在本文路径下，用户仅需在浏览器提交中文问句，由系统生成查询并返回表格与关系图。

该对比对应两类核心挑战。第一类是可用性挑战：不掌握 Cypher 的用户难以稳定完成探索，依赖现场代写也不利于复现。第二类是结果感知挑战：仅有表格记录时，路径结构不直观，难以快速判断实体之间的连接关系。

引入大语言模型后还会新增执行风险：模型可能生成非只读语句、遗漏必要参数，或在多标签 Schema 下选择不合适的路径。为此，系统设计必须同时包含生成约束与执行约束，即在应用层统一处理只读校验、参数一致性检查、超时与结果上限控制，并将查询结果同步转换为图与表两种视图。

\subsection{课题意义}

从实践角度看，本文系统为领域知识库提供了轻量查询入口。用户可直接通过浏览器完成查询与结果浏览，无需安装专用图数据库客户端，也无需记忆固定 Cypher 模板。该模式适用于数据维护、教学演示和业务沟通等场景，能够降低跨角色协作成本。

从方法角度看，本文将自然语言到 Cypher 的受约束生成、应用层执行安全与图表结果转换组织为一条完整实现链，并给出可复现的配置与测试口径。本文所称``低门槛访问''仅指“无需先掌握完整 Cypher 语法即可完成基础查询”，不等同于“无条件适配任意领域图数据”。系统可迁移性取决于目标 Schema 与部署约束是否与本文前提一致。

\subsection{本文主要工作}

本文的实现围绕同一条主数据流展开：用户提交中文问句或手动编写 Cypher，经 Web API（通过 HTTP 协议对外暴露的程序调用接口）进入后端；自然语言查询模块读取 Schema 摘要并调用大语言模型生成参数化只读 Cypher；安全与约束模块在自然语言入口与 \texttt{/run-cql} 入口前统一执行只读校验、参数一致性检查与执行上限控制；图数据访问与执行模块在只读会话中访问 Neo4j；结果转换模块将记录映射为前端所需的图结构与表格字段。

系统测试以第\ref{subsec:ch3-functional-req}~节归纳的功能需求为依据，分别验证自然语言链路、执行链路、安全策略和结果转换的一致性。通过自动化测试与人工走查两类证据，本文评估该实现链在既定领域数据和固定运行配置下的可用性。

\section{系统开发环境与技术选型}

本章说明系统运行所需环境、关键技术选型及架构数据流，目标是给出可复现实验与可解释实现的工程基础。

\subsection{开发环境、公开仓库与路径约定}
\label{subsec:repo-and-paths}

本系统在 GNU/Linux 环境下开发与调试，例如 Ubuntu 22.04 LTS 或 Debian 12。宿主机需安装可运行的 Neo4j 实例，既可连接本机服务（如 \texttt{bolt://127.0.0.1:7687}），也可连接局域网或云主机上的远程实例（如 \texttt{bolt://10.x.x.x:7687}）。其中 \texttt{bolt://} 表示 Neo4j 常用的二进制通信协议 Bolt：应用程序通过该协议与数据库建立会话并发送查询，而非在本机直接打开数据文件。浏览器建议使用近两年的主流版本，如 Chrome、Edge 或 Firefox 的 120 及以上版本，以保证前端图形交互稳定。

后端使用 Python 3.10 及以上版本，并通过虚拟环境管理第三方依赖。典型步骤为使用 \texttt{python -m venv .venv} 创建环境，再通过 \texttt{pip install -r requirements.txt} 安装 \texttt{fastapi}、\texttt{neo4j}、\texttt{httpx}、\texttt{pytest} 等包。前端不依赖 Node 构建链，采用原生 HTML/CSS/JavaScript，便于课程与答辩现场演示。版本控制使用 Git，日常编辑可使用任意文本编辑器或集成开发环境。

本节按“后端语言与框架—数据库—模型服务—可视化—工具链”的顺序说明各技术项职责及取舍依据。

\subsubsection{Python 3.10+}

Python 语法清晰，在数据与 Web 方向库齐全。本系统后端完全采用 Python，便于与 Neo4j 驱动、HTTP 客户端、pytest 测试框架在同一生态内集成。选择 3.10+ 是为了使用较新的类型标注与标准库特性，并与主流 Linux 发行版默认或软件源中的 Python 版本相匹配。

\subsubsection{FastAPI}

FastAPI 基于 Starlette 与 Pydantic，支持异步路由、自动 OpenAPI 文档与请求体验证。相较 Flask，其异步与类型提示更适合同时处理 LLM 网络请求与数据库 IO 的混合场景；相较 Django，其体量更轻，契合本课题“单一服务 + 静态前端”的部署形态。在本系统中注册 \texttt{/nlq}、\texttt{/run-cql}、\texttt{/schema} 等路由，挂载静态资源，并在中间件层配置跨域访问策略（CORS）：用于规定浏览器页面来源与服务接口所在域名不一致时，是否允许网页脚本调用后端接口；演示环境可宽松，生产环境宜收紧为明确允许列表（与下文非功能需求一致）。

\subsubsection{Neo4j 图数据库}

Neo4j 使用 Cypher 与属性图模型，社区资料与示例丰富，且支持多标签节点——这与本文领域数据中“道具 / 方块 / 生物”的继承式建模一致（数据库设计节）；其应用与概念讨论可见早期文献\cite{ref2}，产品与协议细节以官方资料为准\cite{ref7}。未首选 JanusGraph、ArangoDB 等的原因主要是：本课题已有 Neo4j 导入管线与实测数据规模，且 Cypher 与 LLM 训练语料中的“图查询”示例重合度较高，有利于提示词设计。

\subsubsection{大语言模型（LLM）}
\label{subsubsection:llm-stack}

系统通过 OpenAI 兼容的 HTTPS API 调用大模型，端点和模型名由 \texttt{LLM\_API\_BASE}、\texttt{LLM\_MODEL} 等环境变量配置。大模型提示学习与指令遵循能力的研究脉络见文献\cite{ref4}，商用 GPT-4 系列的技术报告见文献\cite{ref9}。在本系统中，大模型的作用是接收用户问句和 Schema 摘要，并返回包含 Cypher 与参数的 JSON。该方式无需训练专用模型即可迭代，但依赖网络与 API 调用，且生成结果须经安全校验通过后方可执行。

\subsubsection{ECharts}

ECharts 为 Apache 开源项目，关系图（Graph）组件成熟，支持力导向布局与交互。在本系统中负责将后端返回的 \texttt{nodes}/\texttt{links} 绘制为可拖拽、可缩放的图谱；与 D3.js 相比上手更快，与 Cytoscape 相比更贴近 Web 单页应用嵌入需求。

\subsubsection{其他工具}

版本管理使用 Git。后端测试使用 \texttt{pytest}，覆盖单元测试与接口约定测试：即用脚本模拟浏览器向后端发请求，检查返回是否符合约定（大语言模型可在测试中替换为固定返回结果的假对象，称为 Mock，以避免每次测试都调用真实云端模型）。HTTP 调用使用 \texttt{httpx} 作为异步/同步客户端连接 LLM API。

\subsection{可行性分析}

\subsubsection{技术可行性分析}

Web 框架、图数据库、经 HTTPS 调用的大语言模型与浏览器端可视化均有成熟实现，形成了可直接复用的技术基础。

在后端侧，FastAPI 与 Neo4j 官方 Python 驱动可在单机完成“服务启动—图库连接—JSON 返回”的基本链路。大语言模型通过兼容接口接入，可降低对单一厂商的耦合。前端使用 ECharts，可直接消费后端返回的结构化图数据并完成关系图绘制。

仓库已提供后端代码、静态前端、\texttt{pytest} 用例与 \texttt{.env.example}，系统具备搭建与运行条件。上述自动化测试用于核对接口路径、请求参数与返回 JSON 是否与实现一致。自然语言问句质量受提示词与 Schema 影响，属于效果调优范畴，不构成技术可行性的否定条件。

\subsubsection{经济可行性分析}

开发期以人力与算力为主，运行期以图库存储与 LLM 调用量为主。Neo4j 社区版与相关 Python 依赖可零许可费使用，LLM 多按量计费。为控制单次请求代价，系统在实现中设置单次查询返回行数上限（\texttt{QUERY\_HARD\_LIMIT}）与单次查询最长等待时间（\texttt{QUERY\_TIMEOUT\_MS}），并与缓存 Schema、提示词约束等措施共同限制无谓的数据库扫描与模型重复调用。

\subsubsection{操作可行性分析}

系统采用浏览器—服务器（Browser/Server, B/S）架构：终端用户仅需通用浏览器即可使用，无需安装专用桌面客户端。自然语言入口与直接 Cypher 可在同一页面切换，并与 Schema 可见、生成语句可回显的设计一致，便于核对系统理解是否与用户意图相符。

\subsubsection{参与者与系统边界}

从用例视角看，系统直接服务两类主体。第一类是终端用户：通过浏览器输入自然语言并查看图谱与表格，通常不具备 Cypher 基础。第二类是高级用户或调试者：直接提交只读 Cypher，对照 Schema 调试生成结果或开展教学演示。两类主体均通过同一 Web 前端访问后端 API。

\subsubsection{非功能需求}

非功能需求描述系统在正确性、响应时间、可维护方式与部署方式上应达到的品质标准。下文分别说明安全性、性能、可扩展性、易用性与部署网络策略；每项均给出为何需要、对读者意味着什么，以及本文采取的做法（具体实现见系统设计章与安全模块小节）。

安全性方面，系统需要形成“数据库侧防护 + 应用侧防护”的双层机制。即便 Neo4j 账号被配置为只读，仍可能出现两类风险：一是运维误将可写账号接入服务；二是大语言模型或用户提交的语句中含有修改数据、删除数据或调用高危管理命令的片段。若不在应用层拦截，这些语句一旦到达数据库，轻则造成运行异常，重则带来数据破坏隐患。因此，应用层在语句发往数据库之前，应对明显危险形式进行识别与拒绝。

本文采用的安全手段包括三类，其含义如下。第一类是关键字黑名单匹配：将常见的数据修改类语句形式（如创建、合并、删除节点或关系等）作为禁止模式，在字符串层面予以拦截，使危险语句在进入数据库前即被拒绝。第二类是可选的执行计划预检：在真正执行查询前，先请求数据库仅生成执行计划而不改动数据，用于在部署策略允许时进一步发现异常访问形态；该步骤可通过配置关闭，以免增加数据库负载。第三类是参数一致性检查：当语句中使用形如“\$名称”的占位符时，请求中必须提供同名参数并与之对应，否则不予执行，从而减少参数遗漏或被恶意拼接的风险。

性能方面，交互式查询对响应时间较为敏感。本系统在自然语言路径上已经包含一次远程大语言模型调用，该步骤往往在秒量级完成；在此前提下，图数据库查询与浏览器绘图不应再显著拉长等待时间。为此，系统对单次查询设置超时时间与返回行数上限，避免单次请求占用过久或一次性拉回过多记录；同时限制前端绘图的节点与边规模，使关系图布局计算保持在可控范围内。

可扩展性方面，本文假定模型服务商、图数据库部署位置或前端展示方案可能在课题结束后更换。若把服务器地址、密钥或业务规则写死在单一源码文件中，替换组件时将涉及大面积改动且容易出错。因此，本文将可变的连接信息与密钥放入环境变量，由部署人员在机器上配置而不写入论文正文；同时在程序结构上区分“接收请求的路由层”“访问数据库的客户端层”“把查询结果转成图表数据的转换层”，使替换某一组件时尽量局部化影响范围。

易用性方面，目标读者往往不以编写 Cypher 为日常工作。系统应尽量让用户在同一浏览器页面内完成完整流程：输入问题、阅读系统生成的查询语句（便于核对含义）、查看表格与关系图、在不满意时修改提问再次提交。当请求不符合约定时，接口宜返回能够理解的错误说明（例如指出缺少哪些参数），而不是仅返回笼统的服务器错误码，从而减少用户在界面与日志之间反复猜测的时间。

部署与网络策略方面，浏览器访问 Web 服务时会涉及“跨域资源共享”（Cross-Origin Resource Sharing, CORS）策略：当网页所在域名与服务接口所在域名不一致时，浏览器是否允许页面调用接口，由服务端返回的 CORS 策略决定。开发阶段为便于前后端同时调试，可暂时放宽跨域限制；面向对外部署时，宜将允许访问的来源收紧为明确列表，并配合反向代理（将外部请求统一接入内网服务）与访问控制，降低接口被无关站点误用或滥用的风险。

\subsection{性能分析}

端到端延迟中，自然语言路径通常以大语言模型 API 调用为主（常见约 1--3 秒，受模型与网络影响）。因此策略上应避免重复调用模型，单次请求携带必要 Schema 信息；一旦失败应尽快返回错误，避免单次请求长时间占用后端处理资源而使其他并发访问受阻。

Neo4j 侧性能主要受查询模式、索引与数据规模影响。本文数据规模为万级节点与万级有向边，常见路径查询多在毫秒至数百毫秒量级。系统在查询语句中使用 \texttt{LIMIT} 限制返回行数上限（类似于“只取前若干条结果”），并结合提示词中对查询形状的约束与实现中的硬上限，避免出现遍历全图、返回巨量记录而导致数据库与前端同时过载的情形。

前端渲染性能受节点和边数量影响。ECharts 在万级节点量级通常可接受，但力导布局成本会随边数上升。为保持前后端一致性，系统将前端渲染上限与数据库结果规模上限联动，控制返回节点与边数量。

综合而言，性能目标依赖规模控制、超时机制与提示约束协同实现，而非依赖未经验证的无限扩容假设。

\section{系统设计}

\subsection{设计原则与模块关联总览}

本章说明上述功能在结构与数据模型上如何组织。不熟悉 Neo4j 的读者可先把握一条主线：自然语言查询与直接 Cypher 查询在进入 Neo4j 前须经过同一套只读、参数与超时策略；查询结果出库后由转换层整理为前端可绘制的 JSON。贯穿例子仍为中文问句``查找石剑''，与第1.1节的问题表述及测试章中的样例一致。

系统设计遵循职责分离（自然语言生成、语句校验、数据库访问、结果转换与前端展示边界清晰）、横切安全（凡进入 Neo4j 的语句适用同一安全与参数策略，避免两条入口不一致）与可观测性（生成语句与错误原因可返回调用方，便于调试与按文档复现）。

从宏观关联看，五个模块并非并列孤立。自然语言模块产生查询意图，执行模块负责访问 Neo4j，安全模块位于执行之前，转换模块处理出库记录，可视化模块消费转换后的 JSON。图~\ref{fig:modules}给出模块划分，下文各小节均围绕这一数据与控制流顺序展开。

\subsection{系统总体架构}

系统采用分层架构，分为表示层、业务逻辑层、数据访问层和外部服务层，见图~\ref{fig:architecture}。表示层为前端 HTML 页面，经 JavaScript 调用后端 API，用 ECharts 绘制图谱，仅消费 JSON，不感知 Bolt 协议。业务逻辑层为 FastAPI 应用，包含路由、自然语言编排、只读校验与数据转换，对下调用 Neo4j 客户端与 LLM HTTP 客户端，对上统一返回 HTTP 与 JSON。数据访问层封装 Neo4j 会话、只读模式、库名与超时，驱动代码集中便于测试替换。外部服务层为大语言模型 API，其可用性与配额不完全由本系统决定，故在结构上视为外部依赖。

\subsection{数据库设计}

本系统使用 Neo4j 图数据库存储《迷你世界》知识图谱，采用多标签属性图模型。为避免术语歧义，这里先说明本文中的两个领域词：方块指游戏世界中可采集或可放置的地形/材料单元，生物指可交互实体（如动物、怪物或 NPC）。数据库设计充分利用 Neo4j 的多标签特性，将方块和生物作为“道具”这一上位概念的子类，通过复合标签表达继承关系。

数据库设计直接影响自然语言查询和可视化效果。Schema 中的标签与关系类型会进入自然语言模块的提示词，节点上的 \texttt{Name}、\texttt{MineTool} 等属性会影响模型生成 \texttt{WHERE} 条件，也会影响转换模块如何归类节点和生成表格列。因此，数据建模不是孤立的存储设计，而是与查询体验和展示效果联动的整体设计。

\subsubsection{架构演进}

早期曾考虑兼容阿里云 GDB（Graph Database）等不支持同一节点多标签的图引擎。在该约束下，无法直接使用 \texttt{:item:block}、\texttt{:item:monster} 这类复合标签表达“方块/生物是一种道具”的继承语义，只能将方块、生物与通用道具拆成不同节点，再通过业务 ID 用桥接关系（历史上曾建模为 PLACE、SUMMON 等）与道具节点相连。

该单标签桥接方案会拉长查询路径。自然语言转 Cypher 时，模型需要记住“先找道具，再经 PLACE 或 SUMMON 跳到方块/生物”的固定模式，生成错误的概率随之上升。同时，这种建模与用户对“方块和生物也是道具类别”的直观理解不一致，维护成本较高。

当前实现已取消 GDB 兼容路径，数据仅面向 Neo4j 导入与查询。方块与生物直接采用多标签节点建模，不再使用 PLACE、SUMMON 等桥接关系。提示词设计与测试用例均以当前多标签模型为准。

\subsubsection{多标签架构设计}

Neo4j 允许同一节点携带多个标签（Multiple Labels）。本文以 \texttt{:item} 作为道具基类标签（通用属性含 ID、Name、Type、Disc、GetWay）；方块类节点同时使用 \texttt{:item:block}，在基类属性之外增加 MineTool、ToolLevel 等；生物类节点同时使用 \texttt{:item:monster}，增加 Life、Attack 等。具体列示见表~\ref{tab:node-types}。相较早期单标签桥接方案，多标签使 ``方块/生物属于道具'' 直接体现在标签上，查询时可写 \texttt{MATCH (n:item:block) RETURN n.Name, n.MineTool} 等语句而无需经 PLACE、SUMMON 等关系绕行，语义与领域模型一致，并减少不必要的跳转与联结开销。

\subsection{自然语言查询模块的设计与实现}

自然语言查询模块负责将中文问句映射为参数化、只读、便于路径可视化的 Cypher。为使模型生成结果与当前图库一致，后端在调用 LLM 前先拉取 Schema 摘要，并将其与用户问句共同放入提示词。模型返回结果被约束为 JSON，主要字段为 \texttt{cql} 与 \texttt{params}，便于后续安全校验和 Neo4j 参数绑定。

对外入口为表~\ref{tab:api}中的 \texttt{POST /nlq}。实现上，FastAPI 路由编排“取 Schema、调用 \texttt{llm\_client.generate\_cypher}、进入安全模块、访问 Neo4j、调用转换器”的流水线。以下给出请求和响应形状，与时序图~\ref{fig:nlq-flow}一致。

请求体格式：
\begin{lstlisting}[language=json, caption={自然语言查询请求体}]
{
    "query": "查找石剑",
    "options": {
        "limit": 100,
        "debug_raw": false
    }
}
\end{lstlisting}

响应体格式（字段名 \texttt{cql} 为历史命名，内容为Cypher字符串）：
\begin{lstlisting}[language=json, caption={自然语言查询响应体}]
{
    "cql": "MATCH (n:item) WHERE n.Name CONTAINS $name RETURN n",
    "params": {"name": "石剑"},
    "graph": {
        "nodes": [...],
        "links": [...],
        "categories": [...]
    },
    "table": {...}
}
\end{lstlisting}

提示词与生成策略可概括为四点：限定只读查询，注入当前库 Schema，说明多标签与路径式 \texttt{RETURN path} 的使用方式，并要求输出 JSON。仅返回孤立节点时，前端难以恢复边信息，因此提示词倾向于引导模型返回路径。完整标签、关系类型、多组示例与调参说明见附录\ref{app:prompt-supplement}。正文仅保留一个代表性示例：
\begin{lstlisting}[language=cypher, caption={合成问句与路径式返回（与附录中其余 few-shot 同型）}]
-- 木料能合成什么
MATCH path = (material:item)-[:CONSUMES]-(r:recipe)-[:PRODUCES]-(product:item)
WHERE material.Name CONTAINS '木料'
RETURN path LIMIT 20
\end{lstlisting}

LLM 调用与 JSON 解析的实现摘录如下。自然语言模块将系统提示词与用户问句拼装为 OpenAI 兼容的 \texttt{chat/completions} 请求，并要求响应为 JSON，以便提取 \texttt{cql} 与 \texttt{params}。

\begin{lstlisting}[language=Python, caption={LLM 生成 Cypher（llm\_client.py 摘录）}]
async def generate_cypher(self, nlq, schema_hint, limit):
    payload = {
        "model": self.model,
        "messages": [
            {"role": "system", "content": load_system_prompt()},
            {"role": "user", "content": f"只读 Cypher，LIMIT 默认 {limit}：\n{nlq}"},
        ],
        "temperature": 0.1,
        "response_format": {"type": "json_object"},
    }
    resp = await client.post(f"{self.api_base}/chat/completions", json=payload, headers=headers)
    obj = json.loads(resp.json()["choices"][0]["message"]["content"])
    return obj.get("cql", "").strip(), obj.get("params", {})
\end{lstlisting}

\subsection{图数据访问与 Cypher 执行模块的设计与实现}

图数据访问与 Cypher 执行模块将最终进入数据库的查询收敛到统一的 Neo4j 客户端封装。会话使用只读访问模式，查询带数据库名和超时配置，返回记录字典与列键序，供转换模块继续处理。\texttt{GET /schema} 为自然语言链路和人工浏览提供标签、关系摘要；\texttt{POST /run-cql} 为高级用户提供与自然语言入口相同的执行语义；\texttt{GET /health} 用于部署探活。路由层在通过 \texttt{cql\_validator} 校验后才调用 \texttt{run\_read}。下列代码摘录展示只读会话与超时换算。

\begin{lstlisting}[language=Python, caption={Neo4j 只读查询（neo4j\_client.py 摘录）}]
def run_read(self, cql: str, params: Dict[str, Any] | None = None):
    params = params or {}
    with self._driver.session(
        database=settings.NEO4J_DATABASE,
        default_access_mode="READ",
    ) as session:
        timeout_seconds = settings.QUERY_TIMEOUT_MS / 1000.0
        result = session.run(cql, parameters=params, timeout=timeout_seconds)
        records = [r.data() for r in result]
        return records, list(result.keys())
\end{lstlisting}

仓库内 \texttt{pytest} 对 API 与校验器做契约级验证。终端测试用例全部通过时的输出示例见图~\ref{fig:screenshot-pytest}；测试范围与断言见系统测试章。

\subsection{结果转换与可视化模块的设计与实现}

Neo4j 驱动返回的记录可能同时包含节点、关系和路径对象，而 ECharts Graph 需要统一的 \texttt{nodes}/\texttt{links} 结构。因此，后端在 JSON 响应中并行返回 \texttt{graph} 与 \texttt{table}，使前端既能展示拓扑，也能展示属性。转换过程主要包括遍历记录、识别节点和关系、生成唯一节点 ID、提取关系端点、去重节点以及处理孤立节点。核心逻辑位于 \texttt{echarts\_converter.py}；下列摘录展示关系边入库时如何保证端点节点先行入表。

\begin{lstlisting}[language=Python, caption={图数据转换（echarts\_converter.py 摘录）}]
def add_rel(rel: graph.Relationship) -> None:
    src, tgt = _node_id(rel.start_node), _node_id(rel.end_node)
    add_node(rel.start_node)
    add_node(rel.end_node)
    links.append({
        "source": src, "target": tgt,
        "category": rel.type, "label": rel.type,
        "value": dict(rel),
    })
\end{lstlisting}

\subsection{安全与约束模块的设计与实现}

安全模块以横切方式作用于 \texttt{/nlq} 与 \texttt{/run-cql}。在调用 Neo4j 前，系统先对语句做只读关键字黑名单匹配，再检查 \texttt{\$param} 占位符与请求体参数是否一致；可选 \texttt{EXPLAIN} 预检由配置项控制，默认关闭以降低图库负载。即使数据库账号配置为只读，应用层仍应拒绝明显写操作与危险 \texttt{CALL dbms.*}，并向调用方返回可读错误。校验逻辑集中在 \texttt{cql\_validator.py}，下列黑名单与表~\ref{tab:readonly-test} 中测试用例对应：

\begin{lstlisting}[language=Python, caption={只读校验黑名单（与 cql\_validator.py 一致）}]
WRITE_BLACKLIST = [
    r"\bCREATE\b",
    r"\bMERGE\b",
    r"\bDELETE\b",
    r"\bDETACH\s+DELETE\b",
    r"\bSET\b",
    r"\bREMOVE\b",
    r"\bDROP\s+(?:INDEX|CONSTRAINT|NODE\s+LABEL|REL\s+TYPE|PROPERTY)\b",
    r"\bLOAD\s+CSV\b",
    r"\bCALL\s+DBMS\.",
]
\end{lstlisting}

除上述关键字拦截外，系统未对 \texttt{CALL} 自定义过程（如 APOC 等插件过程）做白名单约束。部署假设为：图库以只读账号或只读副本访问、服务置于内网或经反向代理与鉴权后暴露，从而将可执行 Cypher 的攻击面与误用面限制在可信主体内；在不可信网络下若须开放，应另加网关层策略或扩展校验。

参数校验（实现说明）。使用正则表达式提取 Cypher 中的参数占位符 \texttt{\$param}，与请求体提供的参数对比，确保必需参数齐全（\texttt{/run-cql} 与 \texttt{/nlq} 共用逻辑，实现见源码树 \texttt{backend/app/} 下路由，路径以第\ref{subsec:repo-and-paths}~节为准）。

本章按四个核心模块给出了详细设计与关键实现：自然语言模块给出策略、请求/响应形状与实现摘录，提示词分块与多组 Few-shot 见附录\ref{app:prompt-supplement}；图数据访问与执行模块统一了只读会话、超时和端点契约；结果转换与可视化模块明确了记录到 \texttt{nodes}/\texttt{links} 的映射过程；安全与约束模块以前置校验方式拦截高风险语句并检查参数完整性。上述实现与系统设计相对应，测试章按同一模块划分组织用例。

\subsection{测试结果汇总}

经测试，系统 38 个 \texttt{pytest} 用例全部通过，API、参数校验、只读黑名单和结果转换行为符合预期。自然语言端到端走查在固定环境下取得 81.0\% 的可接受比例，分项结果见表~\ref{tab:nlq-category}。

\begin{thesisconclusion}

本研究设计并实现了自然语言知识图谱查询与可视化系统（neo4jSlave），并在《迷你世界》属性图数据上完成应用性验证。系统面向不准备系统学习 Cypher 的终端用户，提供中文问句入口、受约束的图数据库查询执行、表格与关系图展示等功能。

验证结果由两类证据组成。自动化测试覆盖 API 契约、只读校验、参数检查与图数据转换，38 个 \texttt{pytest} 用例均通过。自然语言效果在表~\ref{tab:eval-runtime} 所载固定模型、图数据与人工判定协议下统计，21 条中文问句中约 81.0\% 可得到可接受结果。上述结果说明系统在既定数据与配置下形成了可运行、可检查的 HTTP 全链路；该自然语言比例不用于与公开 Text2Cypher 基准或未采用同一问句集的模型作排名比较。

\subsection*{主要工作}

（1）系统架构。系统采用前后端分离的 B/S 结构。FastAPI 提供 REST 风格 API，静态前端由 HTML/CSS/JavaScript 实现，关系图由 ECharts 渲染。

（2）自然语言到 Cypher。系统在提示词中注入 Schema 摘要，并要求大语言模型输出 JSON 形式的只读、参数化 Cypher，使自然语言问句能够进入后续校验与执行流程。

（3）安全与执行。系统在执行前统一进行关键字黑名单检查、可选 \texttt{EXPLAIN} 预检、参数占位符校验、超时和行数限制，使自然语言入口与直接 Cypher 入口遵循同一执行规则。

（4）数据与领域模型。系统采用 Neo4j 多标签属性图组织领域数据，节点、关系与数据规模见数据库设计相关小节。该数据用于应用性验证，重点在于说明自然语言查询、受控执行与可视化展示的完整流程。

\subsection*{可核对要点（非推广性“创新点”表述）}

\begin{enumerate}[(1)]
    \item 可部署的 HTTP 全链路：自然语言与直接 Cypher 经同一安全与执行层，行为可由表~\ref{tab:api}、表~\ref{tab:readonly-test} 等核对。
    \item Schema 约束的自然语言到 Cypher 与后处理：与仅报告模型指标、未给出应用层实现细节的 Text2Cypher 工作相比，本研究将只读、参数、超时等纳入同一可运行服务，便于复现与审计。
    \item 在固定协议下的人工走查：表~\ref{tab:nlq-category} 的 21 条与 81.0\% 为协议内统计；中文 \texttt{Name CONTAINS} 等约束来自数据与提示选择，可迁移性取决于目标图 Schema 与部署条件。
    \item 结果到 ECharts 的转换：\texttt{nodes}/\texttt{links} 形态与转换器用例及界面截图一致。
\end{enumerate}

\subsection*{不足与展望}

当前系统仍有以下不足：
\begin{enumerate}[(1)]
    \item 复杂查询、分组查询和多实体对比仍有失败情形，说明当前提示与 Schema 描述对复杂语义的约束能力有限。
    \item 同义词和近义表达处理不足，用户使用不同名称指代同一概念时，系统仍可能无法稳定匹配。
    \item 权限控制仍停留在应用层只读校验和部署假设层面，尚未形成完整的用户身份认证与访问控制机制。
\end{enumerate}

未来工作方向：
\begin{enumerate}[(1)]
    \item 引入检索增强生成（RAG）或实体链接方法，提高问句中实体、属性和关系类型的匹配准确性。
    \item 构建领域同义词与别名词表，使语义等价表述能够映射到同一图数据库对象。
    \item 完善身份认证、访问控制和审计日志，使系统更适合在非演示环境中部署。
    \item 研究多源图数据适配方法，使同一自然语言查询流程能够迁移到结构相近的其他领域知识图谱。
\end{enumerate}

\end{thesisconclusion}
