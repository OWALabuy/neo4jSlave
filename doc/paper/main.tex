\begin{abstract}
    知识图谱作为一种语义网络，能够以图结构形式描述实体及其之间的关系，在知识管理、智能问答、推荐系统等领域有着广泛的应用。然而，传统知识图谱查询需要用户掌握Cypher等查询语言，对非技术用户存在一定门槛。

    本文设计并实现了一个基于自然语言的知识图谱查询与可视化系统（neo4jSlave），该系统采用B/S架构，通过FastAPI构建后端服务，结合大语言模型实现自然语言到Cypher查询语句的转换，并利用ECharts进行查询结果的可视化展示。系统具有只读安全限制、参数化查询、以及通过环境变量切换Neo4j与LLM端点等特性，降低了知识图谱的使用门槛。

    实验结果表明，系统在pytest自动化测试（38项）中全部通过；在人工设计的21条自然语言用例评测中，整体通过率约为57.1\%，其中方块、召唤类查询表现较好，复杂多跳查询仍有明显提升空间。系统能够将多数常见问句转为可执行Cypher并以图谱形式展示，具有一定的实用价值。
\end{abstract}

\addcontentsline{toc}{section}{Abstract}
\begin{thesisabstracten}
Knowledge graphs, as semantic networks, can describe entities and their relationships in graph structure, and have wide applications in knowledge management, intelligent question answering, and recommendation systems. However, traditional knowledge graph queries require users to master query languages such as Cypher, which presents certain barriers for non-technical users.

This paper designs and implements a natural language-based knowledge graph query and visualization system (neo4jSlave). The system adopts B/S architecture, builds backend services with FastAPI, uses large language models to convert natural language to Cypher queries, and utilizes ECharts for visualization of query results. The system features read-only security restrictions, parameterized queries, and configurable Neo4j/LLM endpoints via environment variables, lowering the barrier to using knowledge graphs.

Automated pytest tests (38 cases) all pass. On a manually curated set of 21 natural-language questions, the end-to-end success rate is about 57.1\%, with stronger performance on block- and summon-related queries and weaker performance on complex multi-hop questions. The system demonstrates practical value for common exploration tasks over a domain knowledge graph.
\end{thesisabstracten}

\section{系统概述}
\subsection{概述}

知识图谱（Knowledge Graph）是一种用于描述实体、概念及其之间关系的语义网络，采用图结构对知识进行建模和存储。近年来，随着大数据和人工智能技术的发展，知识图谱在智能搜索、推荐系统、金融风控、生物医学等领域得到了广泛应用。Neo4j作为目前最流行的图数据库之一，采用属性图模型存储数据，使用Cypher查询语言进行数据操作，为知识图谱的存储和查询提供了高效的支持。

然而，Cypher查询语言具有一定的学习门槛，非技术背景的用户难以直接编写查询语句。为了降低知识图谱的使用门槛，让普通用户也能方便地查询和探索知识图谱中的数据，本文设计并实现了一个基于自然语言的知识图谱查询与可视化系统（neo4jSlave）。该系统允许用户以自然语言的形式输入查询需求，系统自动将其转换为Cypher查询语句并执行，最终以直观的图谱形式展示查询结果。

\subsection{课题意义}

现有的知识图谱查询方式主要存在以下问题：

（1）\textbf{学习门槛高}：用户需要掌握Cypher、SPARQL等图查询语言的语法，对非技术用户不够友好；

（2）\textbf{查询效率低}：复杂的图查询语句编写耗时，容易出错；

（3）\textbf{结果不直观}：传统的表格形式难以直观展示图结构数据中的关联关系。

基于上述问题，本系统利用大语言模型（LLM）的自然语言理解能力，实现自然语言到Cypher查询的自动转换，结合ECharts图可视化库，为用户提供直观、便捷的知识图谱查询体验。该系统在教育、企业知识管理、数据分析等场景具有重要的应用价值。

\subsection{主要内容}

本文从系统需求分析、架构设计、详细实现、系统测试等方面，全面阐述了基于自然语言的知识图谱查询与可视化系统的设计与实现过程。主要工作包括：

（1）设计并实现自然语言到Cypher查询的转换模块，集成大语言模型API；

（2）设计并实现Neo4j图数据库的只读查询接口，确保数据安全；

（3）设计并实现基于ECharts的知识图谱可视化展示模块；

（4）设计并实现系统的安全校验机制，防止恶意查询；

（5）对系统进行全面的功能测试和性能评估。

\section{系统开发环境与技术选型}
\subsection{开发环境}

本系统在GNU/Linux环境下开发，采用Python作为后端开发语言，使用HTML/CSS/JavaScript实现前端界面。

\subsection{技术栈}

\subsubsection{Python 3.10+}
Python是一种解释型、高级、通用的编程语言，具有语法简洁、生态丰富的特点。本系统后端采用Python 3.10以上版本开发。

\subsubsection{FastAPI}
FastAPI是一个现代、高性能的Python Web框架，基于Starlette和Pydantic构建，支持异步编程，具有自动API文档生成、数据验证等功能，非常适合构建RESTful API服务。

\subsubsection{Neo4j 图数据库}
Neo4j是目前最流行的图数据库之一，采用属性图模型，使用Cypher查询语言，支持ACID事务，为知识图谱的存储和查询提供了高效、可靠的解决方案。

\subsubsection{大语言模型（LLM）}
系统通过调用OpenAI兼容的大语言模型API（如GPT-4、Kimi、豆包等），实现自然语言到Cypher查询的转换。利用LLM的语义理解与上下文建模能力，\textbf{辅助}将用户问句映射为可执行的只读查询（最终仍须经黑名单等校验）。

\subsubsection{ECharts}
ECharts是百度开源的数据可视化库，提供了丰富的图表类型，支持大规模数据渲染，本系统使用其关系图（Graph）组件展示知识图谱。

\section{系统分析}
\subsection{可行性分析}

\subsubsection{技术可行性分析}

本系统所采用的技术均为成熟、稳定的技术方案。FastAPI作为现代Python Web框架，性能优异且生态完善；Neo4j作为主流图数据库，提供了稳定的Python驱动；大语言模型API服务（如OpenAI、Kimi等）已广泛应用于生产环境；ECharts是百度开源的可视化库，在数据可视化领域有着广泛应用。因此，本系统在技术层面完全可行。

\subsubsection{经济可行性分析}

本系统开发所使用的工具和框架均为开源免费软件，无需购买商业授权。大语言模型API调用虽然会产生一定费用，但可以通过合理设计降低调用频率，且市面上存在多种免费或低成本的API选项。总体而言，系统开发和运行成本较低，经济可行。

\subsubsection{操作可行性分析}

本系统采用Web界面，用户无需安装额外软件，通过浏览器即可访问。自然语言查询方式符合用户习惯，无需学习复杂的查询语法，操作简便，用户接受度高。

\subsection{需求分析}

\subsubsection{功能需求}

（1）\textbf{自然语言查询}：用户输入自然语言描述，系统自动转换为Cypher查询并执行；

（2）\textbf{直接Cypher执行}：支持用户在「Cypher」模式下直接输入查询语句执行（API路径仍沿用\texttt{/run-cql}命名）；

（3）\textbf{图谱可视化}：将查询结果以关系图形式展示，支持节点拖拽、缩放等交互；

（4）\textbf{数据表格展示}：对于属性查询，以表格形式展示结果；

（5）\textbf{Schema查询}：展示知识图谱的节点标签和关系类型；

（6）\textbf{安全限制}：禁止写入操作，确保数据安全。

\subsubsection{非功能需求}

（1）\textbf{安全性}：系统只读，禁止CREATE、DELETE、SET等写操作；

（2）\textbf{性能}：查询响应时间控制在合理范围内；

（3）\textbf{可扩展性}：支持通过配置更换大语言模型后端与Neo4j连接，便于迁移至其他图实例；

（4）\textbf{易用性}：界面简洁，操作直观。

（5）\textbf{部署与网络策略}：当前实现为便于课程与本地演示，后端对浏览器侧采用较宽松的跨域（CORS）策略；若对外提供服务，应改为白名单域名并配合HTTPS、反向代理等常规Web加固措施。

\subsection{性能分析}

系统性能主要受以下因素影响：

（1）\textbf{LLM API响应时间}：自然语言转Cypher的耗时主要取决于大语言模型的推理速度，通常在1-3秒；

（2）\textbf{Neo4j查询性能}：取决于查询复杂度和数据规模，简单查询可在毫秒级完成；

（3）\textbf{前端渲染性能}：ECharts可支持万级节点流畅渲染。

通过设置查询结果上限、超时控制等机制，可保证系统的整体响应性能。

\subsection{数据库设计}

本系统使用Neo4j图数据库存储《迷你世界》游戏知识图谱，采用多标签属性图模型。数据库设计充分利用Neo4j的多标签特性，将方块和生物作为道具的子类，通过复合标签实现继承关系。

\subsubsection{多标签架构设计}

Neo4j支持为一个节点设置多个标签（Multiple Labels），本系统利用这一特性设计继承关系：

\begin{itemize}
    \item \textbf{基础标签 :item} - 所有游戏道具的父类，包含通用属性（ID, Name, Type, Disc, GetWay）
    \item \textbf{复合标签 :item:block} - 方块类，继承:item所有属性，同时具有方块专属属性（MineTool, ToolLevel）
    \item \textbf{复合标签 :item:monster} - 生物类，继承:item所有属性，同时具有生物专属属性（Life, Attack）
\end{itemize}

这种设计的优势：
\begin{enumerate}[(1)]
    \item \textbf{查询简化}：无需通过关系跳转即可获取完整属性，如`MATCH (n:item:block) RETURN n.Name, n.MineTool`
    \item \textbf{语义清晰}：标签本身表达了类继承关系，符合领域模型
    \item \textbf{性能优化}：避免了额外的JOIN操作，查询效率更高
\end{enumerate}

\subsection{核心算法实现}

\subsubsection{自然语言转Cypher算法}

系统通过精心设计的Prompt引导大语言模型生成\textbf{可执行且满足只读约束}的Cypher查询。Prompt采用结构化设计，充分考虑了Neo4j多标签架构的特点。

\textbf{（1）角色定义}

首先明确告知LLM扮演Neo4j Cypher专家的角色，并限定其只生成只读查询，禁止任何写操作。

\textbf{（2）Schema描述}

详细描述知识图谱的多标签结构：
\begin{itemize}
    \item \textbf{基础标签}：:item（所有道具的父类，包含通用属性ID, Name, Type, Disc, GetWay）
    \item \textbf{复合标签}：:item:block（方块，继承:item并添加MineTool, ToolLevel属性）、:item:monster（生物，继承:item并添加Life, Attack属性）
    \item \textbf{其他标签}：:group（分组）、:recipe（配方）、:smelt（熔炼）、:device（设备）
    \item \textbf{关系类型}：IN\_GROUP、CONSUMES、PRODUCES、DROPS、TOOLMINEDROPS等12种关系
\end{itemize}

\textbf{（3）多标签查询语法}

Prompt中特别强调多标签查询语法：
\begin{itemize}
    \item 查询方块：\texttt{MATCH (n:item:block) WHERE n.Name CONTAINS '关键词'}
    \item 查询生物：\texttt{MATCH (n:item:monster) WHERE n.Name CONTAINS '关键词'}
    \item 查询纯道具：\texttt{MATCH (n:item) WHERE NOT n:block AND NOT n:monster}
\end{itemize}

\textbf{（4）路径查询设计（核心）}

引导LLM生成返回\textbf{Path对象}的查询，使用\textbf{无方向关系}最大化匹配结果。

❌ 错误示例（返回孤立节点）：
\begin{lstlisting}[language=cypher]
MATCH (r:recipe)-[:PRODUCES]->(p:item)
MATCH (r)-[:CONSUMES]->(c:item)
RETURN r, p, c  -- 只返回节点，前端无法显示关系
\end{lstlisting}

❌ 不推荐（有方向，可能遗漏）：
\begin{lstlisting}[language=cypher]
MATCH path = (material:item)<-[:CONSUMES]-(r:recipe)-[:PRODUCES]->(product:item)
RETURN path  -- 有方向限制，可能漏匹配
\end{lstlisting}

✅ 正确示例（无方向路径查询）：
\begin{lstlisting}[language=cypher]
MATCH path = (material:item)-[:CONSUMES]-(r:recipe)-[:PRODUCES]-(product:item)
WHERE material.Name CONTAINS '木料'
RETURN path  -- 无方向，最大化匹配，返回完整路径
\end{lstlisting}

路径查询的优势：
\begin{itemize}
    \item 返回的Path对象包含完整的节点和关系信息
    \item 使用无方向关系（\texttt{-[]-}）最大化匹配所有可能结果
    \item 前端ECharts可以直接解析Path中的关系进行渲染
    \item 用户可以看到完整的知识网络，而非孤立的点
\end{itemize}

\textbf{（5）Few-shot示例（无方向路径查询）}

提供7个典型查询示例，展示无方向路径查询语法：

\begin{lstlisting}[language=cypher, caption={方块挖掘路径查询（无方向）}]
-- 深积岩用什么工具挖
MATCH path = (n:item:block)-[:TOOLMINEDROPS]-(tool:item)
WHERE n.Name CONTAINS '深积岩'
RETURN path LIMIT 10
\end{lstlisting}

\begin{lstlisting}[language=cypher, caption={生物掉落路径查询（无方向）}]
-- 击败野人会掉落什么
MATCH path = (n:item:monster)-[:DROPS]-(drop:item)
WHERE n.Name CONTAINS '野人'
RETURN path LIMIT 20
\end{lstlisting}

\begin{lstlisting}[language=cypher, caption={合成配方路径查询（无方向）}]
-- 木料能合成什么
MATCH path = (material:item)-[:CONSUMES]-(r:recipe)-[:PRODUCES]-(product:item)
WHERE material.Name CONTAINS '木料'
RETURN path LIMIT 20
\end{lstlisting}

\begin{lstlisting}[language=cypher, caption={配方原料路径查询（无方向）}]
-- 木剑的合成配方
MATCH path = (product:item)-[:PRODUCES]-(r:recipe)-[:CONSUMES]-(material:item)
WHERE product.Name CONTAINS '剑'
RETURN path LIMIT 20
\end{lstlisting}

\textbf{（6）输出格式要求}

要求LLM返回JSON格式，包含cql（查询语句）和params（参数）两个字段。

\subsubsection{Prompt优化策略}

\textbf{（1）路径查询优先}

强制使用 \texttt{MATCH path = ()-[]-() RETURN path} 语法，确保返回的查询结果可以被前端正确渲染为知识图谱。使用\textbf{无方向关系}（\texttt{-[]-}）可以最大化匹配所有可能的结果，避免有方向查询导致的遗漏。

测试表明：
\begin{itemize}
    \item 路径查询相比孤立节点查询，可视化效果提升显著
    \item 无方向关系查询相比有方向查询，匹配结果更完整
    \item 复合路径查询可以展示完整的知识网络（如原料-配方-产物链）
\end{itemize}

\textbf{（2）多标签架构优势}

利用Neo4j多标签特性，方块和生物节点同时具有:item标签和专属标签（:block或:monster）。这使得：
\begin{itemize}
    \item 可以直接通过:item:block查询方块，同时获取Name和MineTool属性
    \item 无需通过PLACE/SUMMON关系跳转，简化查询逻辑
    \item 查询语义更清晰，符合领域模型（方块是一种道具）
\end{itemize}

\textbf{（3）无方向关系匹配}

在探索性查询中，使用无方向关系语法：
\begin{itemize}
    \item 无方向：\texttt{(a)-[:TYPE]-(b)} 可以匹配任意方向的关系
    \item 有方向：\texttt{(a)-[:TYPE]->(b)} 只能匹配从a到b的关系
    \item 在迷你世界知识图谱中，无方向查询可以捕获更多关联
\end{itemize}

\textbf{（4）中文模糊匹配}

由于迷你世界道具名称包含融合道具（如"黄铜斧+石剑"），Prompt中强调使用\texttt{Name CONTAINS}进行模糊匹配。

\textbf{（5）参数化查询}

强制使用参数占位符（\texttt{\$param}）代替直接值，降低Cypher注入风险。

\subsubsection{图数据转换算法}

将Neo4j返回的记录转换为ECharts Graph格式：

（1）遍历查询结果中的每个记录；

（2）识别节点（Node）和关系（Relationship）对象；

（3）为每个节点生成唯一ID，提取标签作为类别，属性作为节点数据；

（4）为每条关系提取起始节点、目标节点和关系类型；

（5）进行节点去重，处理孤儿节点。

前端采用简洁的单页面设计，包含以下区域：

（1）\textbf{查询输入区}：文本输入框，支持自然语言与直接Cypher两种模式；

（2）\textbf{图谱展示区}：使用ECharts Graph展示查询结果，支持拖拽、缩放、点击查看详情；

（3）\textbf{表格展示区}：以表格形式展示节点属性；

\subsubsection{只读限制实现}

使用正则表达式黑名单匹配禁止的操作：

\begin{lstlisting}[language=Python, caption={只读校验黑名单（与 cql\_validator.py 一致）}]
WRITE_BLACKLIST = [
    r"\bCREATE\b",
    r"\bMERGE\b",
    r"\bDELETE\b",
    r"\bDETACH\s+DELETE\b",
    r"\bSET\b",
    r"\bREMOVE\b",
    r"\bDROP\s+(?:INDEX|CONSTRAINT|NODE\s+LABEL|REL\s+TYPE|PROPERTY)\b",

除上述关键字拦截外，系统\textbf{未}对 \texttt{CALL} 自定义过程（如APOC插件）做白名单约束；在不可信网络环境下部署时，应优先使用Neo4j\textbf{只读账号}或只读副本，并将服务置于内网或网关之后。

本文设计并实现了一个基于自然语言的知识图谱查询与可视化系统（neo4jSlave），针对《迷你世界》游戏知识图谱进行了实际应用验证。

\subsection*{主要工作}

\textbf{（1）系统架构设计}

系统采用前后端分离的B/S架构，后端使用FastAPI框架提供RESTful API服务，前端使用纯HTML/CSS/JavaScript结合ECharts实现知识图谱可视化。这种架构具有轻量级、易部署、跨平台的特点。

\textbf{（2）自然语言查询模块}

设计并实现了基于大语言模型的自然语言到Cypher查询转换模块。通过精心设计的Prompt模板，包含角色定义、Schema描述、Few-shot示例和输出格式要求，引导LLM生成可执行的只读Cypher。人工评测（21条）总体通过率约57.1\%，其中「道具查找」类约75\%；复杂多关系问句仍为短板。

\textbf{（3）安全校验机制}

实现了多层安全保护机制：
\begin{itemize}
    \item 正则表达式黑名单：禁止CREATE、MERGE、DELETE等写操作关键字
    \item 可选的EXPLAIN预检（默认关闭）：需要时在部署环境开启后，可借助执行计划做额外检查
    \item 参数校验：确保所有必需参数都已提供，防止注入攻击
\end{itemize}

\textbf{（4）知识图谱数据}

成功导入《迷你世界》游戏知识图谱数据：道具侧可拆分为纯物品、方块、生物等互斥类别（见表~\ref{tab:data-scale}，节点总计7,766），并包含配方、分组、熔炼、设备等实体；有向关系合计12,240条（统计口径见该节说明）。图谱涵盖合成、掉落、挖掘等多类游戏机制。

\subsection*{创新点}

\begin{enumerate}[(1)]
    \item \textbf{游戏领域应用}：将自然语言查询技术应用于游戏知识图谱领域，降低了玩家查询游戏资料的门槛
    \item \textbf{中文优化}：针对中文游戏数据特点，优化Prompt设计，支持中文模糊匹配
    \item \textbf{轻量级架构}：不依赖重量级前端框架，使用原生技术栈实现，部署简单
    \item \textbf{模块化设计}：各功能模块解耦，便于更换LLM后端或适配其他知识图谱
\end{enumerate}

\subsection*{不足与展望}

当前系统存在以下不足：
\begin{enumerate}[(1)]
    \item 复杂查询（涉及多关系联合）在人工评测中通过率偏低（表~\ref{tab:nlq-category}中「复杂查询」类为0\%；整体NLQ平均约57.1\%，低于理想水平）
    \item 缺乏同义词处理能力，用户对同一概念的不同表述无法正确识别
    \item 未实现用户认证和查询历史记录功能
\end{enumerate}

未来工作方向：
\begin{enumerate}[(1)]
    \item 引入检索增强生成（RAG）技术，结合向量数据库提升查询准确性
    \item 构建游戏领域同义词词典，支持语义等价表述
    \item 添加用户反馈机制，根据用户点击行为优化查询结果排序
    \item 支持多数据源融合，实现跨知识图谱的联合查询
\end{enumerate}
