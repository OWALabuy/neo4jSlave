## 一、扩写时的通用过滤清单（先读这条）

无论扩写哪一节，下列内容**绝不应当出现在论文正文里**。如果 agent 输出含其中任何一项，立即让它删掉重写。

### 元注释（关于"论文写作本身"的注释）

- ❌ "不写入论文正文"、"在此不做展开"、"具体见仓库"
- ❌ "本节将先讨论 X，再讨论 Y"——本科论文不需要每节都写导言
- ❌ "出于篇幅限制"
- ❌ "本表为撰写时的提交版本"

### 过度防御性免责

- ❌ "本结论不构成与公开榜单的横向对比"
- ❌ "本研究不提出新查询语言、不训练新模型"
- ❌ "本文所称'低门槛'仅指……，不等同于……"

本科毕业设计**不需要主动声明自己没做什么**。说做了什么、做到了什么程度即可。

### 重复定义

- ❌ "只读 Cypher 指不修改图数据、不调用管理类过程的查询语句"——这个定义全文出现一次（在第 4 章首次提出时），其他地方直接用术语。

### 学术腔伪严密

- ❌ 反复用 "通常"、"较"、"在某种程度上"、"可能"——少量可，泛滥即心虚
- ❌ "凡进入 Neo4j 的语句"、"宜将允许列表收紧"、"亦"——文言色彩
- ❌ 生造的"……性"名词："可观测性"、"可感知性"、"可外推性"

### 把工程笔记当论文写

- ❌ 列出环境变量名称大全（应放附录）
- ❌ 列出仓库相对路径约定（应放附录或 README，不是论文正文）
- ❌ "代码与数据以当时 git rev-parse --short HEAD 在 ... 的提交为参照"

### 凑字数式的过渡

- ❌ "上述结论为后续系统设计与详细实现提供约束条件和验收依据"——空话
- ❌ "下文先综述 X、Y、Z，再给出本文系统的需求、设计、实现与测试"——这是目录的工作

### LLM 自带的幻觉痕迹

- ❌ "约 81.0%"——0 是无意义的精度位，写 81% 即可
- ❌ "万级节点量级"——你的数据是 7,766 节点，直接说"约 7,800 个节点"
- ❌ 所有不带具体数字、只说"较高"、"较低"、"显著"的量化表述

#### 1.2.1 知识图谱的存储与查询（约 300 字）

**写什么**：
- 知识图谱主要有两条工程路线：RDF + SPARQL，与属性图 + Cypher
- RDF 路线适合开放数据互联与本体推理 [\cite{ref17}]
- 属性图路线在节点与边上直接附加属性，便于业务建模与迭代 [\cite{ref1,ref2,ref3}]
- 在已有可导入的属性图数据时，Neo4j 在文档与生态上较为成熟 [\cite{ref12}]
- **但是**：仅做到"数据入库 + 语句可执行"并不能解决终端用户使用门槛——这是本研究关注的具体问题

**不要写**：

#### 1.2.2 自然语言到结构化查询（约 350 字）

**写什么**：
- Text-to-SQL 是该方向研究较早的任务，已有大量综述工作 [\cite{ref5}]
- 综述结论：影响生成质量的两类关键因素是"知道当前数据库结构"与"生成的语句必须可被执行"
- 这一结论同样适用于本课题关注的 Text2Cypher
- 已有研究指出，纯零样本生成下复杂语义、多跳约束、细粒度条件组合仍易出错；标注样本充足时通过有监督微调可提升指标 [\cite{ref16}]
- 工程系统因此不能只依赖"更大的模型"，还需要结合 Schema 提示约束与执行后处理

**不要写**：
- 不要给"零样本生成"加括号定义"常称为零样本生成"——这是术语堆砌
- 不要写"无论是 SQL、SPARQL 还是 Cypher，核心难点都在于……"这类大而无当的结论

#### 1.2.3 大语言模型与知识图谱的协同（约 350 字）

**写什么**：
- 大语言模型与知识图谱的协同研究方向较多：知识增强、统一表示、可解释推理等 [\cite{ref11}]
- 在自然语言查询子场景中，分工较明确：模型负责把问句翻译成候选查询，图数据库负责确定性执行
- 该结构兼顾灵活性与可控性
- 同时存在风险：模型生成与事实不符（幻觉）、同一问句多次生成不一致、误执行权限被滥用——需要通过提示约束、外部验证机制加以抑制 [\cite{ref4,ref9,ref11}]
- 本研究采用"受约束生成 + 应用层校验 + 图表双视图展示"的工程路径，不训练专用模型，在已有属性图与通用大语言模型条件下完成中文问句查询闭环

**不要写**：
- 不要写"研究缺口"——本科毕业设计不是论文综述

- 不要在表格后再加一段文字解释表格——让表格自己说话
- 不要解释什么是"OpenAI 兼容"——这是行业常识，本科答辩老师能懂

### 2.3 系统总体架构与数据流

**这一节回答什么**：从浏览器到数据库这条链路上，请求是怎么走的？

**字数**：约 600 字。

#### 段 1（约 200 字）：分层架构

**写什么**：
- 系统采用分层架构：浏览器中的页面（表示层）、FastAPI 应用（业务逻辑层）、Neo4j 驱动（数据访问层），以及外部的大语言模型 HTTPS 服务
- 配图：原文 `img/architecture.png`，照搬

#### 段 2（约 200 字）：主数据流（自然语言路径）

**写什么**：
- 浏览器接收用户中文问句，发送 POST 请求至后端
- 后端先获取当前图数据库的 Schema 摘要，连同问句一并送入大语言模型
- 大语言模型返回结构化输出（包含查询语句与参数）
- 后端在执行前对生成的语句做安全校验，通过后访问 Neo4j 取得查询结果
- 后端将查询结果整理为前端可绘制的节点—边结构与表格字段，返回给浏览器
- 浏览器使用 ECharts 绘图、用表格展示属性

#### 段 3（约 200 字）：主数据流（直接 Cypher 路径）

**写什么**：
- 高级用户可绕过自然语言层，直接提交 Cypher 语句
- 该路径与自然语言路径**共用同一套**安全校验、执行模块、结果转换逻辑
- 这样保证两条入口具有一致的执行语义

**不要写**：
- 不要写"自然语言链路与执行链路在执行前共享同一安全策略，在执行后共享同一结果转换策略，形成'问句（或语句）→执行→图表结果'的统一闭环"——这一长句信息密度低、修辞过重，简化为"两条入口共用同一套安全校验与结果转换"即可

## 五、第 3 章 系统分析（结构改造）

**新结构**（精简为两节）：

- 3.1 可行性分析（精简为单节，不再切分技术/经济/操作）
- 3.2 需求分析（合并参与者、用例、功能需求为一节；非功能需求另列一小节）

字数预算：全章约 1800-2200 字。

### 3.1 可行性分析

**这一节回答什么**：技术层面、经济层面、运行层面分别是否可行？

**字数**：约 500 字。**单节即可，不要再开三个 subsubsection**。

#### 段 1（约 200 字）：技术可行性

**写什么**：
- Web 框架、图数据库、大语言模型 HTTPS 服务、浏览器端可视化均已有成熟实现
- 后端可在单机完成"服务启动—图库连接—JSON 返回"的基本链路
- 前端使用 ECharts 直接消费后端返回的结构化图数据
- 整体不存在工程上无法跨越的技术门槛

#### 段 2（约 150 字）：经济可行性

**写什么**：
- 开发期成本主要为人力
- 运行期成本主要为大语言模型调用费用
- Neo4j 社区版与 Python 依赖均可零许可费使用
- 系统通过查询超时与行数上限控制单次模型调用与数据库扫描成本

**不要写**：
- 具体环境变量名（QUERY_HARD_LIMIT、QUERY_TIMEOUT_MS）——放第 4-5 章

#### 段 3（约 150 字）：运行可行性

**写什么**：
- 终端用户仅需通用浏览器即可使用，无需安装专用客户端
- 自然语言入口与直接 Cypher 入口在同一页面切换，便于教学演示与对比
- 系统可在校园网或内网环境中部署，也可经反向代理对外提供服务

### 3.2 需求分析

**这一节回答什么**：系统应当满足哪些功能与非功能要求？

**字数**：约 1500 字。

**结构建议**：合并精简为三个子节：

- 3.2.1 系统边界与用例（包含原文的"参与者"+"用例分析"）
- 3.2.2 功能需求
- 3.2.3 非功能需求

#### 3.2.1 系统边界与用例（约 300 字）

**写什么**：
- 系统直接服务两类主体：终端用户（不一定具备 Cypher 基础，通过浏览器输入中文问句）和高级用户（可直接提交 Cypher 用于调试与对照）
- 系统外部依赖：大语言模型服务、Neo4j 图数据库
- 用例图（保留原文 `img/use-case-diagram.jpg`）

**不要写**：
- 不要写一长串"用户故事"表格——本科论文不必要的敏捷开发风味
- 不要写"边界之外的模型端点与图库连接由部署环境通过环境变量和连接串配置"——这是元描述

#### 3.2.2 功能需求（约 600 字）

**写什么**：用 5-6 个段落，每段对应一类功能。**不要再写"功能与第 1.1 节问题要点的对应"那张表（原 `tab:req-map`），那张表的存在本身就是凑字数。**

每段写法：

- **自然语言查询**（约 130 字）：用户输入中文问句，系统调用大语言模型生成查询语句、执行并返回图谱与表格结果。这是面向终端用户的主入口。
- **直接 Cypher 查询**（约 100 字）：高级用户可直接提交 Cypher 语句，与自然语言入口共用同一套安全校验与执行逻辑，主要用于调试与教学对照。
- **图谱可视化**（约 100 字）：把查询结果中的节点、关系、路径映射为可在浏览器中拖拽缩放的关系图。
- **数据表格**（约 80 字）：与图谱并行展示，用于呈现长文本属性、数值字段等。
- **Schema 查询**（约 80 字）：提供当前图数据库的标签、关系类型、关键属性等结构信息，既供模型生成时使用，也供前端展示给用户参考。
- **安全约束**（约 110 字）：横切作用于两条执行路径，在访问数据库前过滤写操作、参数不完整语句、危险调用等。

#### 3.2.3 非功能需求（约 600 字）

**结构建议**：精简为四方面，每方面 1 段，每段约 150 字：

- **安全性**：双层防护（数据库账号只读 + 应用层语句校验）；具体手段三类：写操作关键字黑名单、参数占位符一致性、查询超时与行数上限。
- **性能**：交互式查询对响应时间敏感；自然语言路径以模型 API 调用为主（秒级），数据库查询应保持在毫秒至数百毫秒级；前端绘图节点与边规模受控。
- **可扩展性**：连接信息与密钥通过环境变量注入，便于更换模型服务商或数据库部署位置；后端按"路由层 / 客户端层 / 转换层"分离，使局部替换不影响整体。
- **易用性**：用户在同一页面完成"提问—查看生成语句—查看结果—修改提问"的循环；接口在请求异常时返回可读错误说明而非笼统的服务器错误码。

**不要写**：
- 不要写 CORS、反向代理、跨域配置——这些是部署细节，应该在第 4 章或第 5 章实现部分一笔带过即可
- 不要给 CORS 加全称 "Cross-Origin Resource Sharing"——本科答辩老师不需要这个解释，用得着的人本来就懂
- 不要写"宜将允许列表收紧"等文言色彩
- 不要把每点写成"X 方面，系统需要……即便……仍可能出现两类风险：一是……；二是……。因此……"——这个套路出现在每一段就太机械

## 六、给学生的扩写指引

### 流程

每个段落按以下步骤扩写：

1. **打开新会话**（不能用之前抽取要点的旧会话）
2. **粘贴 STYLE.md** 全文给 agent
3. **粘贴本骨架中的某一段** 的"写什么"列表给 agent
4. **告诉 agent**：

```
请按 STYLE.md 的风格要求，把以下要点扩写成一段约 N 字的中文学术散文。
要求：
1. 严格遵守 STYLE.md 中的反面模式与禁用词清单。
2. 不要引入要点之外的事实。
3. 段落必须有清晰主题句。
4. 输出后附自检表（字数、最长句字数、括号处数、是否触发禁用词）。

【字数要求】约 N 字
【段落主题】（一句话）
【要点】
- ...
- ...
```

5. **看 agent 给的自检表**，不达标就让它重写
6. **如果 agent 输出含本骨架"不要写"清单中的内容**，让它删掉

### 核心纪律

- **每段独立扩写**——不要让 agent 一次写完整章。一段一段来，每段都要看自检表。
- **不要给 agent 看原论文**——这是整个流程的命门。
- **触发"不要写"清单中的内容立即删**——如果你看到 agent 输出"不写入论文正文"、"不构成横向对比"这种话，立刻让它删，并提醒它读 STYLE.md 第三节。

### 时间预算

按本骨架扩写，每段约需 5-10 分钟（含与 agent 来回沟通）：

- 摘要 4 段：约 30-40 分钟
- 第 1 章 6 段：约 50-60 分钟
- 第 2 章 9 段：约 70-90 分钟
- 第 3 章 8 段：约 60-80 分钟

合计约 4-5 小时，分两天做完较为合理。

## 七、最后一条提醒

骨架不是禁锢。如果你扩写时发现某些事实没列在"写什么"清单里、但又确实是这一段必须有的——就加上。但加之前请先问自己：

> 这件事是否属于"过滤清单"中的元注释、过度免责、重复定义、工程笔记？

如果答案是"是"，删掉。如果答案是"否"，加上去。
