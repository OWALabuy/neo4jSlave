### 可落地整改清单（据上条导师意见拆解）

#### 1. 需求说透（「看不出到底要啥」）

- [ ] **1.1 一句总述**：在摘要、第1章引言末、需求分析节开头，用非设问、非排比的一句话写清：系统要解决**谁、在什么场景、相对现状缺什么、验收上什么叫做好**；可单列「问题陈述/研究目标」小段落。
- [ ] **1.2 目录可检验**：对照目录每节标题自问：读者**只看目录**能否说清「本章在回答需求的哪一步」；对不上则改名或加 2～3 句承上启下，说明本节与总需求关系。
- [ ] **1.3 需求与后文挂勾**：功能/非功能/约束用表或条列，每条能**回指**总需求；避免只列功能点却不写解决哪条缺口。

#### 2. 把「门槛/障碍」显式写出

- [ ] **2.1 先定义后口号**：凡写「降低门槛」「易用」等，**紧接一句**写清：门槛=**什么具体动作、掌握什么、花多少时间**；禁只写结论、不写对象。
- [ ] **2.2 技术词首现**：对评审可能陌生的词（如 Text2Cypher、只读 Cypher、Schema 摘要等），**首次**用「通俗名 + 一句用途」再进入缩写/接口名。
- [ ] **2.3 少让人猜前提**：关键前提不堆在脚注、附录或「见后文」；同段内**先因后果**再进实现细节。

#### 3. 解释写到底（避免「像解释了其实没解」）

- [ ] **3.1 每节三问**：本节**主张什么 → 为什么成立 → 不认可会卡在哪**；卡壳点本节点名，不推到下节。
- [ ] **3.2 因果满一层**：重要结论前至少**一条**「因为…所以…」或「与…相比，本方案…」的完整短链；禁只有「采用 X 以提升 Y」无中间环节。
- [ ] **3.3 实例锚点**：架构/方案中穿插**一个贯穿全文的用户任务或问句**（可虚构但一致），同一句任务对照**不用本系统时难在哪 / 用后变哪步**，把抽象钉到例子上。

#### 4. 过渡与读者友好（降技术代沟）

- [ ] **4.1 段首承前**：章首、大节首各加 **2～4 句**：上章/节留何问题、本章解决几件事、读毕应带走什么。
- [ ] **4.2 小标题下先引导**：每个小标题下**第一句**不直接甩术语或列表；先用半段说明本节在整体中的位置与读法。
- [ ] **4.3 读者画像**：默认读者**不**常写 Cypher/不调图库；多写评审可见的**行为与结果**，少写对开发者「显然」的默会知识。

#### 5. 结论与原因：原因也须一次能懂

- [ ] **5.1 防套娃解释**：每写「为改善 X」即自问：X 下一层是否又需新名词；若是，上句先通俗定义 X 或改写成可观察事实。
- [ ] **5.2 评价先给框**：谈准确率、可复现、可部署时，**先**一句写清**与什么比、在何约束下**再报数。

#### 6. 风格、查重与「不能特立独行」

- [ ] **6.1 以厚补薄**：在需求、动机、每章小结、讨论与局限等处允许加长，优先**因果说全**，不为简练而省句。
- [ ] **6.2 去个人文风、近学位论文体**：**主张—依据—范围**的平实说明；少警句、少需二次解码的修辞。
- [ ] **6.3 特色让位于扎实**：**特色**不体现在文风口癖，而体现在**可复述的问题—方法—结果链**。

#### 7. 建议执行顺序

1. 先改**摘要、第1章、需求分析**（对应 1.x、2.x，定调全文）。
2. 再按章过 **3.x、4.x**（每节自检/补过渡）。
3. 全文扫**「门槛/降低/易用/提升」**等表述，做 **2.1 专项**。
4. 终稿前通读 **5.x、6.x**，收结论与过简段。
